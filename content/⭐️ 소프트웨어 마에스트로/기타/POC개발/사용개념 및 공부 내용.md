## 1. Spring 비동기 + 이벤트 (가장 큰 덩어리)

코드: `AsyncConfig.java:11`, `IngestPipelineEventHandler.java:16`, `IngestUploadCompleted.java:5`

쓰인 개념:

- `@EnableAsync` + `@Async("ingestExecutor")` — 메서드를 다른 스레드에서 실행
- `ThreadPoolTaskExecutor` — corePoolSize / maxPoolSize / queueCapacity 의 의미
- `ApplicationEvent` + `@EventListener` (record 로 이벤트 정의하는 모던 방식)
- `@TransactionalEventListener(phase = AFTER_COMMIT)` — **트랜잭션이 커밋된 뒤에만** 핸들러 실행
- **왜 AFTER_COMMIT + @Async 를 같이 쓰는가** ← 이게 핵심. 업로드 트랜잭션이 커밋되기 전에 비동기 스레드가 DB를 읽으면 데이터가 없는 race condition 발생.

학습 자료:

- Baeldung: [Spring @Async](https://www.baeldung.com/spring-async)
- Baeldung: [Spring Events (@TransactionalEventListener 포함)](https://www.baeldung.com/spring-events)
- 우아한형제들 기술블로그: ["Spring 트랜잭션 이벤트 처리"](https://techblog.woowahan.com/7835/) — AFTER_COMMIT 패턴 한국어 설명, 거의 같은 그림
- Reflectoring: ["Asynchronous Calls with Spring's @Async"](https://reflectoring.io/spring-boot-async/) — 스레드 풀 튜닝

## 2. Spring 트랜잭션 고급

코드: `IngestPipelineService.java:33-51`, `FileIngestion.java:33`

쓰인 개념:

- `TransactionTemplate` — **프로그래매틱 트랜잭션** (`@Transactional` 의 함수형 버전). 파이프라인 중간중간 짧게 끊어 쓰려고 채택.
- `Propagation.REQUIRES_NEW` — per-file 독립 트랜잭션. 파일 하나 롤백돼도 다른 파일 / run 자체는 살아남음.
- `PlatformTransactionManager`
- **Self-invocation 함정** — `@Async`/`@Transactional` 은 프록시 기반이라 같은 빈 안에서 `this.foo()` 로 호출하면 안 먹음. 그래서 `IngestPipelineEventHandler` 가 `IngestPipelineService` 를 **다른 빈으로 분리**해서 호출함.

학습 자료:

- Baeldung: [Transaction Propagation](https://www.baeldung.com/spring-transactional-propagation-isolation) (REQUIRES_NEW 가 왜 별도 트랜잭션인지)
- Spring Docs: [Programmatic Transaction Management](https://docs.spring.io/spring-framework/reference/data-access/transaction/programmatic.html)
- 토비의 봄 TV — "스프링 트랜잭션 추상화" (유튜브) — 한국어로 PlatformTransactionManager 본질 설명
- 향로 블로그: ["Spring AOP Self-Invocation 문제"](https://jojoldu.tistory.com/635)

## 4. LangChain4j

코드: `IngestClassifier.java`, `WikiPageMerger.java`, `Classification.java`

쓰인 개념:

- **AI Services 패턴** — `interface` + `@UserMessage` 만 정의하면 `AiServices.builder()` 가 동적 프록시로 구현체 생성. (Spring Data Repository 와 같은 그림)
- `@UserMessage("""...""")` — 프롬프트 템플릿 (Mustache 문법, `{{var}}`)
- `@V("var")` — 메서드 인자 → 템플릿 변수 바인딩
- **Structured Output** — 반환 타입을 `record Classification(...)` 으로 선언하면 LangChain4j 가 LLM 응답을 알아서 JSON → record 로 파싱
- `ChatModel` 추상화 — Upstage / Anthropic 같은 Provider 를 빈 교체로 스왑

학습 자료:

- 공식 문서: [LangChain4j Tutorials](https://docs.langchain4j.dev/tutorials)
    - 특히 [AI Services](https://docs.langchain4j.dev/tutorials/ai-services) — 우리가 쓴 패턴 정확히 이거
    - [Structured Outputs](https://docs.langchain4j.dev/tutorials/structured-outputs)
- GitHub: [langchain4j/langchain4j-examples](https://github.com/langchain4j/langchain4j-examples) — Spring Boot 통합 예제 다수
- 유튜브: ["LangChain4j Crash Course"](https://www.youtube.com/results?search_query=langchain4j+spring+boot) (영어, 짧은 영상 위주)
- 한국어로는 인프런/유데미 강의는 아직 거의 없음. 공식 문서가 짧고 잘 쓰여 있어 그대로 읽는 게 빠름.

> 보너스: LangChain4j 의 AI Services 는 **Java Dynamic Proxy** + **Reflection** 으로 인터페이스 구현체를 런타임에 만드는 거라, Spring Data JPA Repository 가 어떻게 동작하는지 배운 적 없다면 같이 보면 머릿속이 정리됨.

## 7. Strategy 패턴 + Spring DI

코드: `ParserRegistry.java`

- 생성자 인자에 `List<ContentParser>` 를 받으면 Spring 이 **모든 구현체 빈을 자동으로 List 로 주입**. 이걸로 확장자 → 파서 매핑 테이블을 만듦.
- 새 파서(`HwpParser`) 추가 시 `@Component` 만 붙이면 자동 등록되는 구조.

학습 자료:

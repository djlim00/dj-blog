# 2. 컨텍스트 스위칭

- PCB를 기반으로 프로세스 상태를 저장 -> 다시 복원
	- 스레드도 동일
- <mark class="hltr-yellow">프로세스가 종료, 인터럽트 발생 시에 컨텍스트 스위칭</mark>

### 컨텍스트 스위칭 비용

1. **유휴시간(idle time)** : 컨텍스트 스위칭 할 때 마다 프로세스 당 유휴시간이 발생
2. **캐시미스** : 프로세스가 가지고 있는 <mark style="background: #FFF3A3A6;">메모리 주소가 그대로 있으면 잘못된 주소 변환이 생기기 때문에</mark>, <mark style="background: #FFF3A3A6;">캐시클리어 과정</mark>이 무조건 일어나야 하므로, 캐시미스가 발생
3. **오버헤드** :  CPU중단단 작업의 상태(레지스터, 프로그램 카운터, 메모리 매핑) 등을 저장하고, 새 작업의 상태를 로드해야함. 이때 일정한 오버헤드가 발생하게 되고 이 오버헤드가 자주 발생하면 <mark style="background: #FFF3A3A6;">시스템 성능이 저하됨. CPU 유효가동 시간이 영향을 미침</mark>

#### 스레드는 스택 영역만 고유하고, 나머지는 공유 -> 컨텍스트 스위칭 비용/시간 적음

# 3. 프로세스 상태

#### 1. 생성 상태
- fork(), exex() : 프로세스 생성
	- 부모 프로세스의 주소 공간을 복사

#### 2. 대기 상태
- CREATE된 후에 메모리를 할당 받았지만, 준비큐에 들어가 있는 상태
	- CPU소유권이 없음

#### 3. 대기 중단 상태
- 준비큐가 꽉참 -> 메모리가 부족

#### 4. 실행 상태
- CPU 소유권 있음 -> 명령어 수행중

#### 5. 중단 상태

blocked 상태는 <mark style="background: #FFF3A3A6;">어떤 이벤트가 발생한 이후 기다리며 프로세스가 차단된 상태</mark>.
ex) 예를 들어 프린트 인쇄 버튼을 눌렀을 때 실행되고 있던 프로세스가 잠깐 멈춘 듯 할 떄가 있음
	이는 프린트 인쇄에 대한 IO요청으로 인해 인터럽트가 발생되어 현재 실행하고 있던 프로세스가 blocked 상태로 잠시 변경된 것.


#### 6. 일시 중단 상태
blocked suspended는 대기 중단과 유사 -> 중단된 상태에서 프로세스가 실행되려 했지만 <mark style="background: #FFF3A3A6;">메모리 부족으로 일시 중단</mark>


#### 7. 종료 상태
terminated(exit) 상태는 프로세스 실행이 완료되어 해당 프로세스에 대한 <mark style="background: #FFF3A3A6;">자원을 반납하며, PCB가 삭제</mark>되는 상태. 
자연스럽게 종료되는 것도 있지만, 부모 프로세스가 자식 프로세스를 강제적으로 종료시켜 <u>비자발적 종료(abort)</u>도 있음
	자식 프로세스에 할당된 자원의 한계치를 넘거나
	부모 프로세스가 종료 되거나
	사용자가 process.kill 명령어로 종료
	시킬 때 abort 발생

# 5. 공유자원, race condition, critical section

- 공유자원 : 프로세스, 스레드가 함께 접근하는 자원/변수
- 경쟁상태 : 공우자원을 하나 이상의 프로세스(스레드)가 접근하는 상황
- <mark class="hltr-yellow">critical section : 두개가 동시에 접근하면 결과가 달라지는 코드영역</mark>


#### 1. Race Condition 관리의 중요성

1. **데이터 정합성**
2. **데이터 무결성**

#### 해결법1. 뮤텍스
- 공유자원을 lock()으로 잠금 -> 한번에 하나만 critical section에 진입 가능함

#### 해결법2. 세마포어
- 뮤텍스의 일반화 -> 정수 S값을 주고 쓴다.
- 여러개의 프로세스가 접근 가능
	- wait()을 하면 : S-=1 -> 음수가 되면 더 접근 불가
		- 대기열에 넣음
	- signal()을 하면 : S+=1 -> 자리 하나 만들어주기.

네, 음수가 될 수 있습니다. 바로 그 점이 이 교재(큰돌 강의)에서 설명하는 세마포어 구현 방식의 핵심입니다.

다시 wait() 코드를 보면:

```
wait(Semaphore s) {
    s.value = s.value - 1;   // 먼저 무조건 감소
    if (s.value < 0) {       // 감소시킨 결과가 음수면
        q.push(p);           // 대기열에 넣고
        block();             // 차단
    }
}
```

S를 **조건 검사 없이 무조건 먼저 1 감소**시킵니다. 그래서 자원이 0개인 상태(S = 0)에서 wait()가 호출되면 S는 -1이 되고, 그 음수값 자체가 "지금 1개가 기다리는 중"이라는 정보를 담게 됩니다.

예를 들어 S = 2(자원 2개)에서 출발하면:

- wait() → S = 1 (진입, 1자리 남음)
- wait() → S = 0 (진입, 자리 없음)
- wait() → S = -1 (자원 없음 → 차단, 1개 대기)
- wait() → S = -2 (차단, 2개 대기)

---

다만 한 가지 짚어야 할 점이 있습니다. **이건 세마포어의 유일한 구현 방식이 아닙니다.**

세마포어를 구현하는 방식은 크게 두 가지입니다.

1. **음수를 허용하는 방식** (이 교재가 설명하는 방식): S가 음수가 되고, 절댓값이 대기 수를 의미
2. **음수를 허용하지 않는 방식**: S는 항상 0 이상으로 유지. wait()에서 S가 0이면 감소시키지 않고 바로 대기시킴. 이 경우 대기 중인 프로세스 수는 별도로 세야 함

운영체제 교과서(예: 공룡책이라 불리는 Silberschatz)나 실제 OS 구현은 보통 2번 방식을 씁니다. S가 0 미만으로 내려가지 않게 막죠. 그래서 "세마포어 값은 음수가 될 수 없다"고 배우는 경우도 많습니다.

공룡책(Silberschatz의 Operating System Concepts)의 세마포어 구현은 **S 값을 음수로 두지 않는** 방식입니다.

핵심 차이는 wait()에서 **감소시키기 전에 먼저 검사**한다는 점입니다.

```
wait(S) {
    while (S <= 0)
        ; // 자원 없으면 기다림 (아무것도 안 함)
    S--;  // 자원이 있을 때만 감소
}

signal(S) {
    S++;
}
```

즉 S가 0 이하면 감소 자체를 하지 않고 대기하므로, **S는 항상 0 이상**으로 유지됩니다. 교재 방식처럼 "먼저 빼고 음수면 대기"가 아니라 "양수일 때만 빼는" 순서입니다.

이 단순 버전은 while 루프가 CPU를 계속 점유하는 **바쁜 대기(busy waiting, 자원을 기다리며 루프를 도는 것)** 문제가 있습니다. 공룡책은 이를 개선한 버전도 제시하는데, 여기서는 대기 프로세스를 큐에 넣고 재웠다가(block) 깨우는(wakeup) 방식을 씁니다. 이 개선 버전에서는 S가 음수가 될 수 있고, **절댓값이 대기 프로세스 수**를 의미합니다. 지금 보시는 교재 코드가 바로 이 개선 버전과 같은 구조입니다.

정리하면:

- **단순 버전**: S ≥ 0 유지, busy waiting 발생
- **개선 버전(블록·웨이크업)**: S 음수 가능, 절댓값 = 대기 수 — 교재 코드와 동일

# 6. 데드락

- 서로가 서로의 자원을 원하면서 프로세스가 이러지도 저러지도 못하고 중단된 상태

## Deadlock 원인

#### 1. 상호 배제(mutual exclusion)
: 주어진 시간 내에 하나의 프로세스만 자원을 독점할 수 있음. 즉 다른 프로세스는 접근이 불가능함.
#### 2. 점유 대기
: 특정 프로세스가 점유한 자원을 다른 프로세스가 요청하며 대기하는 상태.
#### 3. 비선점
: 다른 프로세스의 자원을 강제적으로 가져올 수 없음.
#### 4. 환형 대기 
: 프로세스 A는 프로세스 B의 자원을 요구하고, 프로세스 B는 프로세스 A의 자원을 요구하는 등, 서로가 서로의 자원을 요구하는 상황.

#### 😡 **4가지 상황이 모두 충족되야지 Deadlock -> 하나만 해소되도 일어나지 않음!**

### Deadlock 해결방법
1. 4가지 조건 중 하나만 풀면 됨
2. 사이클 찾기
3. 그냥 두고
	- 나중에 해결 -> 방지 비용이 너무 큼

# 7. CPU 스케줄링 알고리즘

### 1. 비선점형 (CPU가 강제로 프로세스를 중단하지 않음)

#### 1. FCFS
- 먼저 온걸 먼저 함
- <mark class="hltr-yellow">convoy effect 발생(병목 -> 앞에가 너무 길면..)</mark>

#### 2. SJF
- 가장 짧은 프로세스를 가장 먼저 실행
- <mark class="hltr-yellow">근데 얼마나 걸릴지 알 수는 없음 -> 과거의 기록으로</mark>

#### 3. 우선순위
- 우선순위를 줘서 -> 오래 기다릴수록(에이징) Prio가 올라감


### 2. 선점형(CPU가 강제로 프로세스를 중단시킴)
#### 1. RR(라운드 로빈)
- 동일한 시간동안 프로세스가 동일한 시간을 점유하도록 함

#### 2. SRF (Shortest Remaining time First)
- SJF는 짧은 것 부터 실행 -> <mark class="hltr-yellow">Ready Queue에 만약 더 빠른 애가 들어오면?</mark>
	- 더 빠른 애를 실행시켜 버림!

#### 3. 다단계 큐
- 우선순위를에 따라서 Ready Queue를 다르게 만들고(큐마다 우선순위가 있는거임)
	- Queue마다 다른 알고리즘 적용
- 큐 간에 프로세스 이동 안됨 -> 스케쥴링 부담 Low, 유연성 Low
- <mark class="hltr-yellow">근데 우선순위가 낮은 큐는 Starvation 발생 가능</mark>

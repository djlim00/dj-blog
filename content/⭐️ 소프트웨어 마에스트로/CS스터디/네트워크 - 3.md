# 2. TSL 핸드셰이크

- SSL -> TLS

TLS(Transport Layer Security) 핸드셰이크는 **본격적인 데이터를 주고받기 전에, 서로를 확인하고 암호화에 쓸 열쇠를 안전하게 합의하는 과정**입니다. 

먼저 큰 그림부터 잡고 가겠습니다.

**핸드셰이크가 풀어야 하는 두 가지 문제**

1. **인증**: 내가 접속한 서버가 진짜 그 서버가 맞나? (가짜 사이트 아닌가)
2. **열쇠 합의**: 암호화에 쓸 비밀 열쇠를 어떻게 안전하게 공유하나?

특히 2번이 어렵습니다. 도청자가 중간에서 다 보고 있는 네트워크에서, 어떻게 둘만 아는 비밀 열쇠를 만드냐는 거죠. 이걸 푸는 게 핸드셰이크의 핵심입니다.

여기서는 현재 많이 쓰이는 방식을 기준으로, 단계가 명확한 흐름으로 설명하겠습니다.

**전체 과정**

**1단계 — Client Hello (클라이언트 → 서버)**

브라우저가 먼저 인사를 보냅니다. 이때 같이 보내는 것:

- 내가 지원하는 TLS 버전들
- 내가 쓸 수 있는 암호화 방식 목록 (cipher suite, 사이퍼 스위트라고 부름 — "어떤 암호 알고리즘 조합을 쓸지"의 후보들)
- 클라이언트가 만든 무작위 값 (random)
- 열쇠 교환용 재료 (키 셰어 — 뒤에서 설명)

"안녕, 나는 이런 방식들 쓸 수 있어, 여기 내 재료야"라고 보내는 겁니다.

**2단계 — Server Hello (서버 → 클라이언트)**

서버가 답합니다:

- 받은 목록 중 **고른 암호화 방식** ("그럼 우리 이 방식으로 하자")
- 서버가 만든 무작위 값 (random)
- 서버의 열쇠 교환용 재료 (키 셰어)
- **인증서(Certificate)**

**3단계 — 인증서 검증 (클라이언트가 처리)**

여기서 1번 문제(인증)가 해결됩니다. 서버가 보낸 인증서는 "나는 진짜 example.com이 맞다"는 신분증입니다. 이 신분증은 신뢰할 수 있는 기관(CA, Certificate Authority — 인증서 발급 기관)이 서명해서 발급한 것입니다.

브라우저는 이 인증서가:

- 믿을 만한 CA가 서명한 게 맞는지
- 접속하려는 도메인과 일치하는지
- 유효기간이 지나지 않았는지

를 확인합니다. 통과하면 "이 서버는 진짜구나" 하고 신뢰합니다. 가짜 사이트는 이 검증을 통과 못 합니다(정상 CA의 서명을 위조할 수 없으니까요).

**4단계 — 공통 비밀 열쇠 생성 (양쪽이 각자 계산)**

여기가 핵심이자 가장 똑똑한 부분입니다. 2번 문제(열쇠 합의)를 푸는 단계입니다.

1·2단계에서 서로 교환한 **키 셰어(열쇠 재료)**를 가지고, 양쪽이 **각자 자기 컴퓨터에서** 동일한 비밀 열쇠를 계산해냅니다.

핵심은 이겁니다. **비밀 열쇠 자체는 네트워크로 보내지지 않습니다.** 양쪽이 재료만 교환하고, 그 재료를 조합해서 똑같은 열쇠를 각자 만들어냅니다. 도청자는 오가는 재료는 볼 수 있지만, 그 재료로는 최종 열쇠를 계산할 수 없습니다.

이 방식을 디피-헬만(Diffie-Hellman) 키 교환이라고 부릅니다. 원리를 비유로 설명하면:

두 사람이 각자 비밀 색을 하나씩 정합니다(아무한테도 안 알려줌). 그리고 공개된 노란색에 자기 비밀 색을 섞어서 상대에게 보냅니다. 받은 혼합색에 또 자기 비밀 색을 섞으면, 두 사람 모두 **똑같은 최종 색**이 나옵니다. 도청자는 오가는 혼합색은 봤지만, 각자의 비밀 색을 모르니 최종 색을 못 만듭니다. 페인트는 한번 섞으면 원래 색으로 분리하기 어렵다는 점을 이용한 겁니다.

실제로는 페인트 대신 "거꾸로 계산하기 매우 어려운 수학 연산"을 씁니다. 그래서 재료가 다 노출돼도 최종 열쇠는 안전합니다.

**5단계 — Finished (양쪽 확인)**

양쪽이 "나 이제 이 열쇠로 암호화할게"라는 신호를 주고받습니다. 이때 지금까지 주고받은 내용을 방금 만든 열쇠로 검증해서, 중간에 변조가 없었는지도 확인합니다. 양쪽 다 통과하면 핸드셰이크 완료입니다.

**6단계 — 암호화 통신 시작**

이제부터 실제 데이터(HTTP 요청·응답)를 4단계에서 만든 비밀 열쇠로 암호화해서 주고받습니다. 도청자가 봐도 암호문이라 내용을 알 수 없습니다.

**전체 흐름 요약**

```
1. Client Hello   → (지원 방식 + 무작위값 + 키 재료)
2. Server Hello   ← (선택한 방식 + 무작위값 + 키 재료 + 인증서)
3. 인증서 검증     : 서버가 진짜인지 확인 (인증 해결)
4. 비밀 열쇠 생성  : 양쪽이 재료로 같은 열쇠를 각자 계산 (열쇠 합의 해결)
5. Finished        : 변조 없었는지 확인, 준비 완료 신호
6. 암호화 통신     : 실제 데이터 주고받기
```

**핵심 두 가지만 기억하시면**

1. **인증**은 인증서로 해결합니다. CA가 서명한 신분증을 검증해서 가짜 서버를 걸러냅니다.
2. **열쇠 합의**는 디피-헬만으로 해결합니다. 비밀 열쇠를 직접 보내는 게 아니라, **재료만 교환하고 양쪽이 각자 같은 열쇠를 계산**합니다. 그래서 도청당해도 안전합니다.

### <mark class="hltr-yellow">쿠키 Secure 코딩</mark>

먼저 배경을 잡아야 합니다. 앞에서 다룬 흐름과 연결하면, 로그인은 보통 이렇게 동작합니다.

- 로그인 성공하면 서버가 그 사용자를 위한 **세션**을 만들고, 세션을 식별하는 번호인 **세션 ID**를 발급합니다.
- 이 세션 ID를 **쿠키**에 담아 브라우저에 저장합니다. (쿠키 = 브라우저가 보관하는 작은 데이터 조각)
- 이후 요청마다 브라우저가 이 쿠키를 자동으로 같이 보내서, 서버가 "아 이 사람 아까 로그인한 그 사람이네" 하고 알아봅니다.

문제는, 이 세션 ID가 탈취되면 공격자가 그걸로 **로그인한 척 위장**할 수 있다는 겁니다. 비밀번호를 몰라도 세션 ID만 있으면 그 사람 행세를 하는 거죠. 그래서 슬라이드의 수칙들은 전부 "세션 ID를 어떻게 보호하느냐"에 관한 것입니다.

**수칙 1 — 세션 ID로 개인정보를 추측할 수 없게 하라**

세션 ID를 만들 때, 사용자의 개인정보(이메일, 회원번호, 이름 등)를 기반으로 만들면 안 된다는 뜻입니다.

예를 들어 세션 ID가 이런 식이면 위험합니다.

```
나쁜 예: user_djlim00_20260619  (아이디 + 날짜가 그대로 보임)
```

공격자가 이 패턴을 눈치채면, 다른 사람의 세션 ID를 **추측해서 만들어낼** 수 있습니다. "djlim00이 있으니 admin도 있겠네" 하는 식으로요.

그래서 세션 ID는 아무 의미 없는 **완전한 무작위 값**으로 만들어야 합니다.

```
좋은 예: a3f8e1c92b7d4f06...  (의미 없는 랜덤)
```

이러면 추측이 불가능합니다. Spring을 쓰시니 덧붙이면, 스프링 시큐리티나 서블릿 컨테이너가 세션 ID(JSESSIONID)를 만들 때 이미 안전한 난수로 생성해줍니다. 그러니 직접 세션 ID를 만들지 말고 프레임워크가 주는 걸 쓰라는 게 실무 교훈입니다.

**수칙 2 — HttpOnly와 Secure 옵션을 걸어라**

쿠키를 발급할 때 붙이는 두 가지 안전장치입니다.

**HttpOnly** — "자바스크립트로는 이 쿠키를 못 읽게" 막는 옵션입니다.

이게 왜 중요하냐면, **XSS**라는 공격 때문입니다. XSS(Cross-Site Scripting)는 공격자가 웹페이지에 악성 자바스크립트를 몰래 심는 공격입니다. 만약 그 스크립트가 실행되면, 보통 이런 코드로 쿠키를 훔쳐갑니다.

javascript

```javascript
document.cookie  // 쿠키 내용을 읽어서 공격자 서버로 전송
```

그런데 쿠키에 HttpOnly를 걸어두면, 자바스크립트의 `document.cookie`로 그 쿠키가 **아예 안 보입니다.** 악성 스크립트가 심어져도 세션 ID를 못 훔치는 거죠. 쿠키는 브라우저가 HTTP 요청 보낼 때만 자동으로 쓰고, 스크립트는 건드릴 수 없게 됩니다.

**Secure** — "HTTPS 연결에서만 이 쿠키를 보내라"는 옵션입니다.

앞에서 TLS를 다뤘죠. HTTPS는 암호화된 연결이고, 그냥 HTTP는 암호화 안 된 평문 연결입니다. Secure 옵션이 없으면, 브라우저가 실수로 `http://`(암호화 안 됨)로 요청할 때도 쿠키를 같이 보냅니다. 그러면 그 평문 구간에서 도청자가 세션 ID를 그대로 볼 수 있습니다.

Secure를 걸면 **암호화된 HTTPS 연결에서만** 쿠키가 전송됩니다. 평문 구간으로는 절대 안 나가니, 도청으로 세션 ID가 새는 걸 막습니다.

두 옵션을 정리하면:

|옵션|막는 위협|원리|
|---|---|---|
|HttpOnly|XSS(악성 스크립트의 쿠키 탈취)|자바스크립트가 쿠키를 못 읽게|
|Secure|도청(평문 구간 노출)|HTTPS에서만 쿠키 전송|

**수칙 3 — 세션 타임아웃을 걸어라 (슬라이드가 강조하는 부분)**

세션을 **일정 시간 후 자동 만료**시키라는 뜻입니다. 형광펜이 쳐진 핵심 항목이죠.

왜 필요하냐면, 세션이 영원히 유지되면 두 가지 문제가 생깁니다.

1. 세션 ID가 한번 탈취되면, 공격자가 **무한정** 그걸로 위장할 수 있습니다. 만료가 있으면 일정 시간 뒤 그 ID가 무용지물이 되니 피해 시간이 제한됩니다.
2. 공용 PC(카페, 회사)에서 로그아웃을 깜빡하고 자리를 뜨면, 다음 사람이 그 세션을 그대로 씁니다. 타임아웃이 있으면 일정 시간 후 자동으로 끊깁니다.

그래서 세션에 유효기간을 둬서, 그 시간이 지나면 다시 로그인하게 만드는 겁니다.

**아래 예시 (업비트 화면)가 말하는 것**

슬라이드 아래쪽은 이 타임아웃을 **실제 금융 사이트가 어떻게 적용하는지** 보여줍니다.

- 화면 우측 상단에 **"23시 59분"** 카운트다운이 보입니다. 이게 "로그인 유지 시간이 이만큼 남았다"는 표시입니다. 시간이 다 되면 자동 로그아웃됩니다.
- "로그인 상태 유지" 활성화 여부에 따라 시간이 달라집니다. <cite index="...">활성화하면 24시간, 비활성화하면 3시간</cite>으로 설정된다는 거죠.

핵심 메시지는 마지막 줄입니다. **금융 사이트처럼 민감한 곳은 타임아웃을 짧게 설정해야 한다**는 겁니다.

이유는 명확합니다. 거래소는 세션이 탈취되면 곧장 자산 탈취로 이어집니다. 그러니 일반 사이트보다 훨씬 빡빡하게, 짧은 시간만 로그인을 유지하고 금방 다시 인증을 요구하는 게 안전합니다. 편의성(오래 로그인 유지)과 보안(자주 로그아웃) 사이에서, 금융은 보안 쪽으로 기울여야 한다는 뜻입니다.

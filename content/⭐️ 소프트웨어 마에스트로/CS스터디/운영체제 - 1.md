# 1. CPU

1. **산술논리장치(ALU)**
	- 사칙연산, 논리연산
2. **제어장치**
	- 프로세스 조작
	- 데이터 처리 순서 결정
3. **레지스터**
	- CPU내의 매우 빠른 임시기억장치

**1단계 인출(Fetch)** — 메모리에서 다음 명령어를 가져옵니다. 이때 레지스터 두 개가 핵심입니다. **PC(Program Counter)**는 "다음에 실행할 명령어가 메모리 어디에 있는지" 주소를 들고 있습니다. 이 주소를 **MAR(Memory Address Register)**에 넣어 메모리에 보내면, 해당 위치의 명령어가 **MDR(Memory Data Register)**로 올라옵니다.

**2단계 해석(Decode)** — 가져온 명령어를 제어장치가 해석합니다. "이게 덧셈인지, 메모리에서 값을 읽으라는 건지" 알아내서, 어느 회로를 켤지 정합니다. 이게 아까 정리하신 제어장치의 "데이터 처리 순서 결정"입니다.

**3단계 실행(Execute)** — 해석한 대로 ALU가 계산하거나, 메모리에서 값을 읽거나 씁니다. 사칙연산·논리연산이 여기서 일어납니다.

**4단계 저장(Store)** — 결과를 레지스터나 메모리에 씁니다.

이 4단계가 끝나면 PC가 다음 명령어 주소로 넘어가고, 다시 1단계로 돌아갑니다. CPU는 전원이 켜진 동안 이 사이클을 초당 수십억 번 반복합니다. 회로도에서 화살표가 빙글빙글 도는 그림이 바로 이겁니다.

# 5. Virtual 메모리, Swapping, Page fault, 스레싱

- 실제 메모리 보다 더 크게 사용할 수 있도록 하는 기법
- Physical 주소를 virtual 주소로 변환해서 사용

![[Pasted image 20250317182636.png]]

#### 1. 페이지 테이블
- 가상 메모리를 실제 메모리에 매핑 테이블
- TLB(캐싱)를 통해서 바로 있으면 바로 매핑 해줌
	- 안그러면 테이블을 풀스캔 해야함

#### 2. 메모리 할당 과정

- 프로세스가 종료가 되면, 가상 메모리 주소와 실제 메모리 주소는 아무 연관이 없음!
- 가변적이다!

#### 3. Page fault와 Swapping

- 가상 메모리 주소가 실제로는 없음 -> page fault
- 현재 RAM에 없으니까 참조를 할 수가 없음
	- 이때 메모리에서 지금 사용안하는 부분을 HDD(SDD)로 옮기고 마치 RAM 처럼 사용하는 방법을 Swapping
	- 즉 항상 처리는 RAM에서만 일어난다!
		- **이때 옮겨지는 값은, 변수나 중간 결과와 같이 수정됐지만 아직 저장이 되지 않은..어디서 다시 읽어올 수 없는 값들임 -> 이걸 구분하는 비트가 dirty bit!**

#### 4. Page fault 과정

1. 유효한 가상 주고 접근 -> 없음 -> 트랩! -> 운영체제에 알림
2. <u>디스크에서 빈 프레임</u> 찾음
3. 페이지 교체 알고리즘 기반으로 Swapping
4. <u>페이지 테이블 갱신</u>  -> 다시 명령어 시작

#### 5. 스레싱
- <mark class="hltr-yellow">page fault가 너무 자주 일어나!!!</mark>
- swapping이 잦아지면 성능 에바임
- 근데 page fault가 일어나서 swapping이 발생하면 CPU는 계속 IO트랩이 발생하니까 이용률이 낮아짐
	- **디스크 관련 IO연산은 RAM에 비해서 굉장히 느림 -> OS가 기다리면서 할게 없어짐**
- 그럼 이용률을 높이기 위해서 더 많은 프로세스를 올림 -> 악순환 발생

##### 해결방안
1. SSD를 사용한다.
2. 작업세트 & PFF

#### 작업세트
- 프로세스의 과거 사용이력을 기반으로 많이 사용하는 페이지 집합을 만들어 한꺼번에 미리 메모리에 로드하는 것

#### PFF
- **페이지 폴트 빈도를 보고 프로세스가 RAM에서 차지하는 프레임을 조절 -> threshold를 만든다**
- 상한선 -> 해당 프로세스가 RAM에서 차지하는 프레임 늘리기
- 하한선 -> 해당 프로세스가 RAM에서 차지하는 프레임 줄이기(다른 애들 자리 만들어주기)

# 6. 페이지 히트 & 페이지 미스

- **페이지 히트**
	- 프로세스가 요청한 페이지가 이미 메모리에 있음 -> 바로 접근 가능

- **페이지 미스**
	- 프로세스가 요청한 페이지가 메모리에 없음

#### 1. Page miss != Page fault

- **접근 권한 오류**
	- 접근 권한이 없으면 page miss 남(그럼 당연히 메모리에 로드하려는 시도를 안함, 페이지 폴트면 찾아서 로드함)
- **성능최적화를 위해 미리 로드 할 때의 페이지 미스**
	- 성능을 위해서 사용하기 전에 미리 요청을 함

>실제 데이터 사용전에 다른 데이터로 덮어쓰게 되면 페이지 미스는 발생하지만, 사용되지 않아 페이지 폴트로 발전되지는 않음

이 문장에서 말하려던 건 이겁니다.
**"미리 올려둔 페이지가 쓰이기도 전에 쫓겨났지만, 실제로 요청되지 않았으므로 디스크에서 다시 가져와야 하는 진짜 페이지 폴트로는 이어지지 않는다."**

즉 프리패칭이 빗나가도 **헛수고(미리 올린 게 낭비됨)는 있지만, 진짜 비용(디스크 재접근)은 안 든다**는 게 핵심입니다. 예측이 빗나가도 손해가 크지 않다는, 프리패칭의 안전한 면을 설명하는 메모인 거죠.

# 7. 페이지 교체 알고리즘
![[Pasted image 20260622105156.png]]

### 1. LFD (Longest Forward Distance) 오프라인 알고리즘

- 더 이상 참조 안됨
- 가장 늦게 참조되는 페이지

### 2. FIFO

- 가장 먼저 들어온 페이지, 먼저 교체

### 3. LRU(가장 사용하지 않은)
- 가장 사용하지 않은 페이지를 바꾼다.

### 4. NUR(Not Used Recently) == Clock
![[Pasted image 20250319172253.png]]
- 한바뀌 도는 동안 사용 안되면 0이 됨.
	- 0을 찾으면 페이지 교체

### 5. LFU(Least Frequenctly Used)
- 가장 참조횟수가 적은 페이지 교체

# 7. 프로세스와 스레드 차이

- 프로세스는 CPU에서 작업을 처리하는 단위입니다.
- 스레드는 프로세스 내부의 작업의 흐름

### 차이1. 메모리
1. <mark style="background: #FFF3A3A6;">프로세스</mark> : **코드, 데이터, 스택, 힙 메모리 영역**을 기반으로 작업
2. <mark style="background: #FFF3A3A6;">쓰레드</mark> : 프로세스 내의 스택 메모리를 제외한 다른 메모리 영역을 프로세스 내 다른 스레드들과 공유 -> **메모리 이점**
- 쓰레드는 힙, 데이터, 코드 메모리영역을 공유한다 -> 멀티 쓰레드의 상황에서 힙, 데이터, 코드를 아껴쓸 수 있음

### 차이2. 통신
1. 프로세스 : 프로세스간에는 격리되어 있기 때문에 서로 <mark style="background: #FFF3A3A6;">통신을 위해서는 IPC를 사용</mark>
2. 쓰레드 : 공유된 메모리 영역을 사용하기 때문에 <mark style="background: #FFF3A3A6;">그냥 통신 가능(빠름)</mark>

### 차이3. 격리성
1. 프로세스 : 프로세스간에는 격리되어 있기 때문에 하나가 문제가 생겨도 서로 영향 X
2. 쓰레드 : 격리되어 있지 않기 때문에 한 쓰레드의 문제가 다른 쓰레드에도 영향 O

### 차이4. 속도
- 프로세스는 생성과 종료에 더 많은 시간이 듦, 쓰레드는 상대적으로 빠름

# 1. REST API

- REST규칙을 지키는 API
- 총 8가지의 규칙이 있음.

#### 1. Uniform-Interface
- API에서 자원들은 각각의 독립적인 인터페이스를 가짐
	- URL 자원식별
	- 표현을 통한 자원조작(HTTP메서드)
	- Self-descriptive 메세지
		- HTTP 헤더(자체적인 정보가 있어야함)
	- HATEOAS 구조
		- 하이퍼링크에 따라 다른 페이지를 보여줘야함

#### 2. 무상태성
- HTTP
- 세션 로그인 -> REST API 서버로 만들거면, 서버에 세션관련 로직이 없어야함
	- 즉 세션 관련 처리는 브라우저에서 해야함.

#### 3. 캐싱 가능
- HTTP는 캐싱이 됨.
	- GET

#### 4. 클라이언트-서버 구조
- 서버는 화면 API 제공, 비즈니스 로직 처리
- 클라이언트는 HTTTP 로직만 처리

#### 5. Layered 시스템
- 계층구조 아키텍처
- 보통 WEB기반 서비스들은 만족함

#### REST API의 URL 규칙
1. URL에 동사X
2. 파일 확장자 표시X
3. 오직 명사
4. 계층적으로
5. 언더바( _ ) 안쓰고 바( - )  써야함
6. 응답코드 잘 써야함

# 2. 브라우저 랜더링 과정

- 브라우저 엔진, 랜더링 엔진
- 네트워크 통신부
- JS 해석
- UI 백엔드
- 자료 저장소(Local Storage, Cookie, Session Storage)

#### DOM 트리와 CSSDOM 구축
- html페이지는 여러 요소를 가짐
- 하나의 요소들이 HTML파서에서 -> 하나씩 Node가 되어서 DOM트리 구성

#### CSSDOM 구축
- DOM과 동시에 CSSDOM도 구축

#### 렌더트리와 렌더레이어 생성
- <mark class="hltr-yellow">DOM+CSSDOM = 랜더객체</mark>
- 렌더객체 트리 -> 렌더 트리

#### 렌더 레이러를 대상으로 레이아웃 설정
- 좌표는 부모노드를 기준으로 상대적으로 설정

#### 렌더 레이어를 대상으로 칠하기
#### 레이어 합치기 및 표시
- 각각 레이어에서 비트맵 생성
- GPU에서 텍스트로 업로드
- 텍스트들이 합쳐져서 하나의 이미로 렌더링 및 출력

>**Q. 자주 묻는 면접질문 : 렌더트리와 DOM트리는 1:1 대응일까요?**
	아닙니다. DOM트리 > 렌더객체 > 렌더트리가 되는 과정에서 <mark style="background: #FFF3A3A6;">display:none으로 사라지는
	렌더 객체(노드)</mark>들이 있을 수 있기 때문에 <font color="#d83931">1 : 1 대응이 아닙니다.</font>

# 3. www.naver.com의 과정
주소창에 어떤 주소를 쳤을 때 생기는 과정을 알아보자!

![[Pasted image 20250224130759.png]]

#### 1. 리다이렉트
: 리다이렉트가 있으면 리다이렉트, 없으면 해당 요청에 대한 과정 진행


#### 2. 캐싱
- 해당 요청이 캐싱이 가능하진 여부 파악
- 이미 캐싱된 요청이면 캐싱된 값을 반환(빠름) / 없는 새로운 요청이면 다음 단계(느림)

- 캐싱 : 요청된 값의 결과값을 저장하고, 그 값을 다시 요청하면 다시 제공
	- **브라우저 캐시**
	- **공유 캐시**

#### 1. 브라우저 캐시
: 쿠키, 로컬 스토리지 등을 포함한 캐시 aka 개인캐시(private cache)
- 브라우저 자체가 데이터를 가지고 있음


#### 2. 공유캐시
: 클라이언트와 서버 사이에 있으며 사용자간에 공유할 수 있는 응답을 저장

ex) 프록시 서버가 캐싱 -> "리버스 프록시를 둬서 내부 서버로 포워드 한다"
<span style="background:rgba(205, 244, 105, 0.55)">Node.js로 서버 구축, 앞단에 프록시 서버로 nginx를 둬서 캐싱 서버로 사용</span>
![[Pasted image 20250224132419.png]]

ex) AWS Clouddfront 또는 Cloudflare와 같은 콘텐츠 전송 네트워크(CDN)을 둘 수도 있음
![[Pasted image 20250224132508.png]]
- 이렇게 캐시가 없다면 요청이 계속되지만, 브라우저 캐시를 통해 요청을 줄일 수 있음

#### # 3. DNS
: 캐싱이 끝나고 요청을 보내야 된다면, 실제 서버로 요청!
: 브라우저가 요청한 FQDN(Fully Qualified Domain Name)인 www.naver.com 등의 이름을
: <mark style="background: #FFF3A3A6;">DNS을 통해 실제 IP 주소를 확인</mark>

> **FQDN** : 호스트와 도메인이 합쳐진 완전한 도메임 이름 www(호스트), naver.com(도메인)

![[Pasted image 20250224132727.png]]
- <span style="background:rgba(205, 244, 105, 0.55)">역순이다</span>
![[Pasted image 20250224132827.png]]

#### DNS 캐싱
: 미리 저장해 놓은 도메인 이름
: 로컬 PC에 자동적으로 저장
: **브라우저 캐싱 / OS 캐싱**
![[Pasted image 20250224132900.png]]

#### 4. IP 라우팅
: <mark style="background: #FFF3A3A6;">IP를 기반으로 IP 라우팅 -> ARP 과정을 거쳐서 실제 서버를 찾음
</mark>

#### # 5. TCP 연결 구축
: <mark style="background: #FFF3A3A6;">TCP 3way-handshake와 SSL 연결 등의 연결을 설정</mark>
: <font color="#d83931">요청을 보내고 응답을 받음</font>
: HTTP/3 부터는 TCP연결 안하고 QUIC으로 연결

#### 6. 콘텐츠 다운로드
: 브라우저는 사용자가 요청한 <u>데이터를 서버로부터 다운</u>받음

#### 7. 브라우저 렌더링
: 6에서 받은 데이터를 바탕으로 브라우저 엔진이
: <u>브라우저렌더링</u>을 함

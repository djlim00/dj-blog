### 라우팅 테이블
- 라우터들의 IP 주소를 저장한 DB
	- 연결하는 방법도 있음.
#### 라우팅 테이블 구성
![[Pasted image 20250213171151.png]]
1. <mark style="background: #FFF3A3A6;">Network Destination</mark> : 목적지 네트워크의 대역
2. <mark style="background: #FFF3A3A6;">Netmask</mark> : 대상 주소를 설명할 때 쓰이는 값
3. <mark style="background: #FFF3A3A6;">Gateway</mark> : 이 장치와 연결된 홉, 즉 패킷이 전달되는 다음 IP주소(외부 네트워크와 연결된 장치). 
	- 목적지가 로컬 네트워크라면 "연결됨(connected)"라고 표기되며, 다른 네트워크면 해당 네트워크의 게이트웨이
4. <mark style="background: #FFF3A3A6;">Interface</mark> : 게이트웨이로 가기 위해 거치는 장치
5. <mark style="background: #FFF3A3A6;">Metric</mark> : 우선순위, 패킷 전송을 위해 최적의 경로가 선택되도록 참고되는 값. 동일한 라우팅 테이블 요소가 2개 있을 때, 이 값이 낮은 요소가 선택. 일반적으로 hop count가 들어가며, 지연시간, 처리량이 들어갈 수도 있음

Network Destination은 최종 목적지가 아니라 **목적지 네트워크 대역**입니다.

- 위 예시에서 101.25.67.0이 네트워크 주소이고, Netmask 255.255.255.0과 짝을 이뤄서 "101.25.67.0 ~ 101.25.67.255 범위로 가는 패킷"이라는 뜻입니다. 개별 호스트(예: 101.25.67.50)가 아니라 그 네트워크 전체를 가리킵니다.

- 255.255.255.0 → 앞 3자리(101.25.67)가 네트워크, 마지막 1자리(0~255)가 호스트

- 다음으로 패킷을 넘겨줄 장치의 IP(다음 홉).
	- 핵심은 이겁니다. 101.25.67.0 네트워크는 보통 내 컴퓨터에 직접 붙어있지 않습니다. 그래서 그쪽으로 가려면 중간에서 패킷을 대신 전달해 줄 라우터가 필요한데, 그 라우터의 IP가 10.0.0.2라는 의미입니다. 내 패킷을 일단 10.0.0.2한테 던지면, 그 라우터가 나머지 경로를 알아서 처리합니다.

# 5. 클래스풀(Classful IP Addressing)

- **네트워크를 구별하기 위해서 사용한다**
	- 네트워크 주소가 동일하다 -> 로컬 네트워크이다!
	- 호스트주소 -> 네트워크 내의 각각의 주소!

![[Pasted image 20260618225754.png]]

**클래스별 구분**

|클래스|1옥텟 범위|네트워크 부분|호스트 부분|
|---|---|---|---|
|A|1 ~ 126|1옥텟|2,3,4옥텟|
|B|128 ~ 191|1,2옥텟|3,4옥텟|
|C|192 ~ 223|1,2,3옥텟|4옥텟|

앞부분은 정확합니다. 맞습니다, 클래스를 구분하려고 앞 비트를 고정한 겁니다. 첫 비트만 읽어도 "아 이건 0으로 시작하니 A구나" 하고 즉시 클래스를 판별하려는 거죠. (참고로 A는 `0`, B는 `10`, C는 `110`입니다. 회원님이 쓰신 `01`은 `0`의 오타로 보입니다.)

그런데 "개손해"라는 결론은 다시 봐야 합니다. **손해가 거의 없습니다.**

이유는 이겁니다. 그 고정 비트들은 **호스트 부분을 깎아먹는 게 아니라, 네트워크 부분의 일부일 뿐**입니다.

B클래스로 보겠습니다.

```
10nnnnnn.nnnnnnnn.HHHHHHHH.HHHHHHHH
└┬┘└──────┬─────┘
 고정    네트워크로 쓰는 부분
 2비트
```

앞 2비트(`10`)는 고정이라 네트워크 번호를 매기는 데 못 씁니다. 맞습니다. 하지만 그 뒤 14비트(6+8)는 네트워크 번호로 자유롭게 씁니다. 그리고 **호스트 부분(뒤 16비트, H)은 전혀 손대지 않았습니다.**

즉 손해 보는 건 "만들 수 있는 네트워크 개수"지, "한 네트워크에 담을 수 있는 호스트 수"가 아닙니다.

**손해 규모를 따져보면**

B클래스 네트워크 개수: 2의 14제곱 = 16,384개

만약 고정 비트가 없었다면 2의 16제곱 = 65,536개를 쓸 수 있었겠죠. 그러니 네트워크 개수가 4분의 1로 준 건 맞습니다.

하지만 이게 왜 큰 손해가 아니냐면, 호스트 쪽은 그대로이기 때문입니다.

- 한 네트워크당 호스트 수: 2의 16제곱 = 약 65,000대 (고정 비트 영향 0)
- 16,384개 네트워크 × 약 65,000대 = 여전히 10억 개가 넘는 주소

호스트 칸을 한 비트도 안 깎았기 때문에, 실제 IP 주소 총량에서 잃는 건 미미합니다.

**더 중요한 관점**

고정 비트가 없으면 클래스를 **구분할 방법 자체가 없습니다.** 첫 옥텟이 100인 주소를 받았을 때, 이게 A인지 B인지 알 수가 없으면 네트워크 부분이 1옥텟인지 2옥텟인지도 모릅니다. 그러면 라우팅 자체가 불가능합니다.

그러니 이건 "비트를 손해 보는 대가로 구분 능력을 산" 거래입니다. 적은 비용으로 꼭 필요한 기능을 얻은 거라, 손해라기보다 합리적인 설계였습니다.

**다만, 회원님 직감이 틀린 건 아닙니다**

"이거 좀 낭비 같은데"라는 감각 자체는 정확합니다. 클래스풀의 진짜 낭비는 고정 비트가 아니라 **다른 데** 있었습니다.

앞에서 말씀드린 것처럼, 호스트 300대가 필요한 회사가 C클래스(254대)로는 부족해서 B클래스(6만 대)를 받으면 5만 9천 개가 통째로 노는 문제 말입니다. 고정 비트 몇 개와는 비교도 안 되는 거대한 낭비죠.

그래서 클래스 방식을 버리고, Netmask로 경계를 자유롭게 긋는 CIDR로 넘어간 겁니다. 회원님의 "손해 아니냐"는 문제의식이 바로 CIDR이 등장한 이유와 정확히 같은 방향입니다. 다만 진짜 범인은 고정 비트가 아니라 클래스 단위로만 나눠줘야 했던 뻣뻣함이었던 거죠.

## 입력 내용: 아 그러면 이걸 십진수로 표현하면 클래스A는 011111111이 최대 네트워크 주소의 크기니까.

아 그러면 이걸 십진수로 표현하면 클래스A는 011111111이 최대 네트워크 주소의 크기니까. 0~2^7까지인건가?

## Claude 응답: 거의 맞는데, 두 가지를 손봐야 합니다.

거의 맞는데, 두 가지를 손봐야 합니다.

**첫째, 비트 자릿수**

A클래스 네트워크 부분은 1옥텟 = 8비트입니다. 그 8비트 중 맨 앞 1비트가 `0`으로 고정이죠. 그러니 자유롭게 쓰는 건 뒤 7비트입니다.

회원님이 쓰신 `011111111`은 9자리라 옥텟(8자리)을 넘었습니다. 8자리로 쓰면 `01111111`이 맞고, 이게 십진수로 127입니다.

**둘째, 범위 계산**

7비트로 표현 가능한 값은 0부터 2의 7제곱 빼기 1까지, 즉 **0 ~ 127**입니다.

`2^7`은 128인데, 0부터 세기 시작하니 최댓값은 128이 아니라 127입니다. 0, 1, 2 ... 127 해서 총 128개(2의 7제곱 개)인 거죠. 개수가 128개고, 가장 큰 값이 127입니다.

그러니 회원님 표현을 고치면 "0 ~ (2의 7제곱 빼기 1)" = "0 ~ 127"입니다.

**확인**

1옥텟 비트로 다시 보면:

- 최소: `00000000` = 0
- 최대: `01111111` = 127

그래서 A클래스 1옥텟 범위가 0~127이라고 배우신 겁니다. 맨 앞 비트를 0으로 고정한 결과가 정확히 이 범위입니다.

**다만 실제 쓰는 범위는 1~126**

이론상 1옥텟 값은 0~127이지만, 실제 호스트에 할당하는 A클래스는 **1~126**입니다. 양 끝 두 개가 특수 용도로 빠지기 때문입니다.

- **0** (`00000000`): "이 네트워크 자신"을 가리키는 특수 주소라 일반 할당 안 함
- **127** (`01111111`): 루프백(loopback) 주소로 예약됨. 127.0.0.1이 바로 "내 컴퓨터 자신"을 가리키는 그 주소입니다. 코딩하실 때 localhost로 쓰는 그것 맞습니다.

그래서 이론값(0~127)과 실사용값(1~126)이 다릅니다. 시험이나 문서에서 "A클래스는 1~126"이라고 나오는 이유가 이 두 개를 뺐기 때문입니다.


## 입력 내용: 아하.

아하. 좀 실수를 했구만. 그럼 클래스B는 10이 고정이니까 10xxxxxx.xxxxxxxx만큼이 네트워크 주소이고 그 이하의 2개의 옥텟이 호스트가 되는거네? 그러면 최소 10000000.00000000~10111111.11111111까지인거네?

오후 11:08


## Claude 응답: 네, 이번엔 전부 정확합니다.

네, 이번엔 전부 정확합니다. 실수 없이 잘 잡으셨습니다.

정리해드리면:

**네트워크 부분: `10xxxxxx.xxxxxxxx`**  
앞 2비트(`10`)는 고정, 뒤 14비트(6+8)가 자유롭게 쓰는 네트워크 번호. 맞습니다.

**호스트 부분: 뒤 2개 옥텟 (16비트)**  
3,4옥텟이 호스트. 맞습니다.

**네트워크 부분의 범위**

- 최소: `10000000.00000000` = 128.0
- 최대: `10111111.11111111` = 191.255

그래서 B클래스 첫 옥텟이 **128~191**이 되는 거고, 회원님이 쓰신 범위와 정확히 일치합니다.

**한 가지만 용어 정밀하게**

"최소 ~ 까지"라고 쓰신 그 범위는 정확히는 **네트워크 부분이 가질 수 있는 값의 범위**입니다. 즉 "B클래스에 몇 개의 네트워크를 만들 수 있나"를 보여주는 거죠.

- 128.0 부터 191.255 까지의 네트워크 번호들
- 개수로는 2의 14제곱 = 16,384개

호스트 부분(3,4옥텟)은 여기서 별개로, 각 네트워크마다 0.0 ~ 255.255 범위를 따로 가집니다.

헷갈리기 쉬운 부분이라 한 번 더 나누면:

- **네트워크가 결정하는 범위**: 128.0.x.x ~ 191.255.x.x (어느 네트워크냐)
- **호스트가 결정하는 범위**: x.x.0.0 ~ x.x.255.255 (그 안의 어느 장치냐)

앞 2옥텟으로 "어느 동네"인지 정하고, 뒤 2옥텟으로 "그 동네의 몇 호"인지 정하는 구조입니다.

C클래스도 같은 방식으로 풀어보시면 (`110` 고정 → 192~223) 바로 감 잡으실 겁니다.

### <span style="background:#b1ffff">서브넷 마스크</span>
- **네트워크 부분만 모두 1**
- **호스트주소 부분은 모두 0**

ex) 192.10.52.12(클래스C겠지요?)
- 그럼 여기에 서브넷 마스크 255.255.255.0을 AND연산자를 해버리면
- 192.10.52.0이 나오겠지? -> **아 이 부분만 네트워크 주소구나!!**

어 그러면 기존에는 
255.0.0.0(A)
255.255.0.0(B)
255.255.255.0(C)
근데 굳이? 그냥 11111111.11111111.11111111.0000010 뭐 이런식으로도 할 수 있는거 아니야? 훨씬 가변적이네!

- 나누기 2씩 더 나눌 수 있게 된다!

>"내 맥북이 와이파이로 요청을 쏘면, 패킷에는 최종 목적지 IP와 포트가 실리고, 당장 건네줄 상대인 공유기의 MAC을 목적지로 해서 공유기에 전달된다. 공유기는 목적지 IP를 보고 라우팅 판단을 한 뒤, 출발지 IP를 자기 공인 IP로 바꿔서(NAT) 통신사 쪽으로 내보낸다."

- 192.168.50.112/24 -> 3개의 옥텟이 서브넷 마스크(255.255.255.0) -> 클래스C
- 192.168.50.112/25 -> 3옥텟+1(25)개가 서브넷 마스크(255.255.255.126) -> 클래스C보다 절반이 호스트

Q. 10개의 사무실이 있으며 사무실당 12개의 장치를 연결해야 한다면?
- 우선 12개의 장치 -> 호스트주소가 12개 이상 -> 16개(4비트)
- 11111111.11111111.11111111.1111hhhh -> **255.255.255.240**

네, 회원님이 잡으신 데까지는 정확합니다. 거기서부터 사무실 10개를 어떻게 처리하는지 이어가겠습니다.

**회원님이 맞게 잡은 부분**

장치 12개 → 호스트 4비트 → 마지막 옥텟이 `xxxxhhhh` (앞 4비트 네트워크, 뒤 4비트 호스트). 정확합니다.

```
11111111.11111111.11111111.xxxxhhhh
                            └┬─┘└┬─┘
                          서브넷  호스트
                          구분용  4비트
                          4비트
```

**여기서 핵심: 사무실 10개는 "남는 앞 4비트"로 나눕니다**

마지막 옥텟의 앞 4비트(`xxxx`)가 바로 사무실(서브넷)을 구분하는 자리입니다. 이 4비트로 만들 수 있는 조합은 2의 4제곱 = **16개**. 사무실이 10개니까 16개 중 10개를 쓰면 충분합니다. (6개는 남습니다.)

즉 한 옥텟을 16칸씩 끊는 게 아니라, **호스트 16개짜리 덩어리를 16개 만들 수 있고, 그중 10개를 사무실에 배정**하는 구조입니다.

**서브넷마스크**

`xxxxhhhh`에서 앞 4비트까지가 네트워크니, 1의 개수가 24 + 4 = 28개. 따라서 **/28**.

마지막 옥텟 `11110000`을 십진수로 바꾸면 240. 그래서 서브넷마스크는:

**255.255.255.240** (= /28)

**네트워크가 나뉘는 방식**

마지막 옥텟의 앞 4비트를 0000부터 하나씩 올리면 각 사무실 대역이 나옵니다. 호스트 부분(뒤 4비트)이 0이 되는 값들이 네트워크 시작 주소고, 16씩 증가합니다.

|사무실|마지막 옥텟 앞4비트|네트워크 주소|장치 붙일 범위|브로드캐스트|
|---|---|---|---|---|
|1|0000|.0|.1 ~ .14|.15|
|2|0001|.16|.17 ~ .30|.31|
|3|0010|.32|.33 ~ .46|.47|
|4|0011|.48|.49 ~ .62|.63|
|5|0100|.64|.65 ~ .78|.79|
|6|0101|.80|.81 ~ .94|.95|
|7|0110|.96|.97 ~ .110|.111|
|8|0111|.112|.113 ~ .126|.127|
|9|1000|.128|.129 ~ .142|.143|
|10|1001|.144|.145 ~ .158|.159|

(11~16번째 대역인 .160, .176 ... .240은 안 쓰고 남겨둠)

**각 줄 읽는 법**

사무실 1번을 예로 들면:

- 네트워크 주소 .0 → 대역의 시작, 장치에 못 붙임
- .1 ~ .14 → 실제 장치 14대까지 (12대 필요하니 충분)
- 브로드캐스트 .15 → 그 대역 전체에 한 번에 알리는 용도, 장치에 못 붙임

이래서 16개 중 양 끝 2개(.0, .15)를 빼고 14개가 쓸 수 있는 주소인 겁니다. 앞 대화에서 "16 - 2 = 14"가 이거였습니다.

**전체 그림 한 줄 정리**

- 마지막 옥텟 8비트를 → 앞 4비트(사무실 구분) + 뒤 4비트(장치 구분)로 쪼갬
- 앞 4비트로 16개 서브넷 생성 가능 → 10개 사무실에 배정
- 뒤 4비트로 각 서브넷마다 14대 수용 가능 → 12대 요구 충족
- 서브넷마스크: 255.255.255.240 (/28)

회원님이 `xxxxhhhh`까지 정확히 쪼개셨으니, 그 `xxxx` 4비트가 사무실 16개를 구분하는 자리이고 거기에 10개를 채운다, 이 한 가지만 추가로 잡으시면 됩니다.

### 요청을 줄이기 위한 기술들

1. **이미지 스프라이트**
	- 한번에 수많은 이미지를 가져와서 쪼개서 보여주기
	- ![[Pasted image 20260619091232.png]]

2. **코드 압축**
	- 코드를 압축해서 서빙
3. **이미지 Base64 인코딩**
	- 이미지 -> 64진법 문자열로 인코딩 -> 이미지 서버에 HTTP 요청을 하지 않아도 됨
	- 대신 파일 크기가 37% 커짐

**상황: 웹페이지에 고양이 이미지를 띄우고 싶다**
평소 방식은 이렇습니다.
```html
<img src="cat.png">
```
브라우저가 이 HTML을 읽다가 `cat.png`를 보면, "아 이미지 파일이 따로 있구나" 하고 **그 파일을 받으러 서버에 요청을 한 번 더** 보냅니다.
```
1. 브라우저: HTML 주세요 → 서버: 여기 (요청 1)
2. 브라우저: HTML 보니 cat.png가 필요하네, cat.png 주세요 → 서버: 여기 (요청 2)
```
이미지가 10개면 요청이 1 + 10 = 11번 일어납니다. 요청이 많을수록 페이지가 느려지죠. 이걸 줄이자는 게 핵심입니다.
**Base64 방식: 이미지를 글자로 바꿔서 HTML 안에 직접 박는다**
이미지 파일(고양이 사진)을 Base64라는 방식으로 변환하면, 이런 긴 글자 문자열이 나옵니다.
```
iVBORw0KGgoAAAANSUhEUgAAAAUA... (실제론 엄청 김)
```

이 글자 덩어리가 **이미지 그 자체**입니다. 사진을 글자로 표현한 거죠. 이걸 `src`에 파일 이름 대신 직접 넣습니다.

```html
<img src="data:image/png;base64,iVBORw0KGgoAAAANSUhEUgAAAAUA...">
```

`cat.png`라는 파일 이름이 들어갈 자리에, 파일을 가리키는 대신 **이미지 데이터 전체를 글자로 적어넣은** 겁니다.
**여기서 핵심 — 왜 요청이 줄어드나**
이제 브라우저가 이 HTML을 읽으면:
```
1. 브라우저: HTML 주세요 → 서버: 여기 (요청 1)
2. 브라우저: HTML 안에 이미지가 글자로 이미 다 들어있네? 받으러 갈 필요 없다. 그냥 그림.
```

요청 2번이 사라졌습니다. 이미지가 HTML 안에 글자로 포함돼서 같이 왔으니, 따로 받으러 갈 필요가 없는 겁니다.

`data:image/png;base64,` 이 앞부분은 브라우저한테 "지금부터 나오는 글자는 파일 경로가 아니라, png 이미지를 base64로 바꾼 데이터야"라고 알려주는 표시입니다. 그래서 브라우저가 그 글자를 이미지로 해석해서 그립니다.

**비유**

평소 방식은 편지(HTML)에 "사진은 우체국에 맡겨놨으니 따로 찾아가세요"라고 적는 것과 같습니다. 받는 사람이 우체국에 한 번 더 가야 하죠(추가 요청).

Base64 방식은 편지 봉투 안에 사진을 같이 넣어 보내는 겁니다. 편지를 열면 사진이 이미 들어있으니, 우체국에 갈 필요가 없습니다.

다만 사진을 글자로 풀어쓰다 보니 편지가 두꺼워집니다(크기 33~37% 증가). 그래서 작은 아이콘 정도에만 쓰고, 큰 사진엔 안 씁니다.

**핵심 아이디어: 8비트 묶음을 6비트 묶음으로 다시 자른다**

이름의 64가 단서입니다. 2의 6제곱 = 64. 즉 Base64는 데이터를 **6비트씩** 끊어서, 그 6비트로 표현 가능한 64가지를 각각 하나의 글자에 대응시키는 방식입니다.

원본 바이너리는 8비트(1바이트) 단위인데, 이걸 6비트 단위로 다시 자르는 게 전부입니다.

**왜 6비트인가**

안전한 글자가 64개 있기 때문입니다. 알파벳 대문자 26개 + 소문자 26개 + 숫자 10개 + 기호 2개(+, /) = 정확히 64개. 6비트로 만들 수 있는 경우의 수(64가지)와 딱 맞아떨어집니다. 그래서 6비트 한 덩어리 = 글자 한 개로 일대일 대응됩니다.

**변환 과정 (3바이트 → 4글자)**

여기서 숫자 트릭이 나옵니다. 8과 6의 최소공배수가 24라, **3바이트(8×3=24비트)를 4글자(6×4=24비트)로** 바꾸는 게 기본 단위입니다.

예시로 글자 "Man"을 변환해보겠습니다.

**1단계 — 각 글자를 8비트 2진수로**

```
M = 77  → 01001101
a = 97  → 01100001
n = 110 → 01101110
```

**2단계 — 24비트를 이어붙임**

```
01001101 01100001 01101110
→ 010011010110000101101110 (24비트 한 줄)
```

**3단계 — 6비트씩 다시 자름 (여기가 핵심)**

```
010011 010110 000101 101110
```

8비트 경계를 무시하고 6비트로 새로 끊는 게 포인트입니다.

**4단계 — 각 6비트를 십진수로, 그걸 글자표에서 찾음**

```
010011 = 19 → T
010110 = 22 → W
000101 = 5  → F
101110 = 46 → u
```

그래서 "Man" → **"TWFu"**가 됩니다.

**글자표(인덱스 → 글자)는 고정되어 있습니다**

```
0~25  : A B C ... Z
26~51 : a b c ... z
52~61 : 0 1 2 ... 9
62    : +
63    : /
```

이 표는 표준으로 정해져 있어서, 누가 변환하든 같은 결과가 나옵니다.

**왜 크기가 33% 커지는지도 여기서 보입니다**

3바이트(24비트)가 4글자(4바이트, 32비트)가 됐죠. 3 → 4니까 약 33% 증가입니다. 원본 8비트를 6비트로 쪼개면서 글자 수가 늘어나는 게 원인입니다. 앞서 말한 "37% 정도 커진다"가 바로 이 비율(정확히는 약 33%, 여기에 패딩과 줄바꿈이 붙으면 조금 더)입니다.

**나누어떨어지지 않을 때 — 패딩(=)**

원본이 3바이트로 딱 안 떨어지면(예: 1바이트, 2바이트가 남으면), 6비트를 채우려고 0을 보충하고, 모자란 글자 자리는 `=` 기호로 메웁니다. 그래서 Base64 결과 끝에 `=`나 `==`가 붙어있는 걸 종종 보게 됩니다. 이 `=`는 "여기는 실제 데이터가 아니라 빈 자리 메움"이라는 표시입니다.

**복원(디코딩)은 반대로**

받는 쪽은 이 과정을 역으로 합니다. 글자 → 6비트 → 24비트로 이어붙임 → 8비트씩 다시 잘라서 원래 바이트로 복원. `=` 부분은 버립니다. 변환 규칙이 고정이라 원본이 그대로 정확히 복구됩니다(손실 없음).

### HTTP/1.1의 고질적인 문제 : HOL

- 헤더가 너무 무거워짐!(Head Of Line Blocking)
	- 큐에 패킷이 너무 많아서 지연됨.

# 10. HTTP/2와 HTTP/3의 차이

- SPDY프로토콜을 개발해서 만듦 -> 2.0

## HTTP/2
![[Pasted image 20260619093300.png]]
- 바이너리 프레이밍은 **Application 계층 박스 안에** 들어있습니다. TCP나 IP 같은 별도 계층 사이에 새로 생긴 게 아니라, **HTTP라는 애플리케이션 프로토콜 내부에** 추가된 처리 단계입니다.
> HTTP가 데이터를 전송 계층(TCP)으로 내려보내기 직전에 거치는 변환 단계가 HTTP 내부에 새로 생겼다
- 헤더 부분 → **HEADERS frame** (주황 화살표: 위 박스의 헤더들이 여기로)
- 본문 부분 → **DATA frame** (초록 화살표: {"msg":"hello"}가 여기로)

### 바이너리 포맷 계층 -> <mark class="hltr-yellow">멀티플렉싱</mark> 가능!
- <mark class="hltr-yellow">단일 TCP 연결이 여러 스트림에서 여러 HTTP 요청과 응답을 비동기적으로 보낼 수 있음!</mark>
	- HOL 해결
	- 기존에는 앞에 엄청 큰 요청(응답)이 있으면 그 이후의 요청들이 밀리게 된다.

만약 한 연결에 요청 A와 요청 B를 동시에 섞어서 흘려보낸다고 칩시다. 받는 쪽에 글자들이 이렇게 도착합니다.

```
POST /a ... {"msg":"hello"} POST /b ... {"data":"world"}
```

받는 쪽 입장에서 이 글자 스트림을 보면, **어디까지가 A고 어디부터가 B인지 알 방법이 없습니다.** 텍스트에는 "이 글자는 A 소속, 저 글자는 B 소속"이라는 꼬리표가 없거든요. 그냥 글자가 주르륵 이어질 뿐입니다.

그래서 텍스트 방식에서는 섞으면 뒤죽박죽이 되니까, **어쩔 수 없이 하나씩 순서대로** 보낼 수밖에 없었습니다. A 끝나고 B, B 끝나고 C. 이게 멀티플렉싱이 불가능했던 진짜 이유입니다.

**프레임이 이 문제를 푸는 방식: 모든 조각에 꼬리표를 붙인다**

바이너리 프레이밍이 하는 일이 정확히 이 한계를 겨냥합니다. 메시지를 작은 프레임으로 쪼개되, **각 프레임 머리에 "내가 누구 소속인지"를 적은 꼬리표(헤더)를 붙입니다.**

각 프레임에 들어가는 핵심 정보:

- **스트림 ID**: 이 프레임이 몇 번 요청 소속인지 (A는 1번, B는 3번 식)
- **프레임 종류**: 헤더 프레임인지 데이터 프레임인지
- **길이**: 이 프레임이 몇 바이트인지
이제 한 연결에 여러 요청의 프레임을 섞어 보내도 이렇게 됩니다.

```
[ID:1 헤더][ID:3 헤더][ID:1 데이터][ID:3 데이터][ID:1 데이터]...
```

받는 쪽은 각 프레임의 ID만 보면 됩니다. "ID 1번짜리들만 모으면 요청 A, ID 3번짜리들만 모으면 요청 B"라고 정확히 분류해서 다시 조립할 수 있습니다. 섞여 있어도 꼬리표가 있으니 안 헷갈립니다.

# 1. 왜 디자인 패턴을 쓸까?

- 문제를 해결하기 위한 하나의 규약/약속
	- ~한 경우에는 ~게 코드를 짜자!
- 라이브러리나 프로그램의 기반이 됨
	- 간편 로그인 구현 -> 다양한 서비스 로그인 -> 전략패턴!
	- Spring -> MVS, Vue.js -> MVVM
- 일종의 개발자들간의 약속이 될 수 있음
	- 협업 Good

### 디자인 패턴의 종류
1. <span style="background:#fff88f">생성패턴</span> - 객체 생성 방법 
	ex) **싱글톤**, **팩토리**, 추상팩토리, 빌더, 프로토타입패턴...etc
	
2. <span style="background:#fff88f">구조패턴</span> - 객체, 클래스 등 큰 구조를 만들 떄 유연하고 효율적으로 만드는 방법
	ex) **프록시**, 어댑터, 브리지, 복합체, 데코레이터, 퍼사드, 플라이웨이트패턴...etc
	
3. <span style="background:#fff88f">행동패턴</span> - 객체나 클래스 간의 알고리즘, 책임 할당에 관한 디자인패턴
	ex) **이터레이터**, **옵저버**, **전략**, 책임 연쇄...etc
	어떤 식으로 객체나 클래스들이 행동할거야?! 순회할거야!?

#### 유명한 프레임워크/라이브러리의 디자인패턴
1. Spring AOP : Proxy 패턴
	- 대상 객체를 감싸는 프록시를 만들어서 로킹, 트랜잭션을 주입
	- 어드바이스 체인은 Chain of Resposibility 패턴 -> 여러 인터셉터가 순차적으로 호출
2. Spring Security : Chain of Responsibility 패턴
	- FilterChainProxy 안에서 여러 보안 필터(인증, 인가, CSRF 등등)의 <u>순서대로</u> 요청을 처리한다.
	- 인증 처리하는 패턴은 "전략패턴" -> AuthenticationProvider를 교체해서 DB인증, LDAP, OAuth 등 다양한 방식을 적용할 수 있음.
3. Spring 전반
	- 팩토리 / 추상 팩토리 : BeanFactory, ApplicationContext가 객체 생성을 담당
	- 싱글톤 : Bean 객체는 싱글톤으로 생성됨. 컨테이너가 관리함.
	- 템플릿 메서드 : JdbcTemplate, RestTemplate가 대표적. 공통 프름은 고정 -> 변하는 부분만 콜백으로 위임
	- 프록시 : 트랜잭션, 캐싱, 보안 부분에서는 비즈니스 로직 앞단에 프록시 객체가 요청을 가로채서 해당 작업을 먼저 수행하고 비즈니스 로직을 수행하도록 한다.

-> Spring이 프록시 패턴을 쓰는 이유가 뭘까? -> 원본 코드를 건드리지 않고 기능(로깅, 트랜잭션)을 추가하기 위해서. 

**다른 백엔드 관련**
- **Express.js / FastAPI / 대부분의 웹 미들웨어 구조**: **Chain of Responsibility**. 미들웨어를 순차 통과시키는 구조는 Spring Security 필터 체인과 본질적으로 같습니다.
- **Hibernate / JPA**: **Proxy**(지연 로딩용), **Unit of Work**(영속성 컨텍스트), **Data Mapper**(엔티티 ↔ 테이블 매핑). 영속성 컨텍스트가 변경을 추적했다가 한 번에 반영하는 것이 Unit of Work입니다.
- **Kafka / 메시지 큐 계열**: **Observer / Pub-Sub 패턴**. 발행자와 구독자를 분리합니다.

# 2. 라이브러리 vs 프레임워크

### 공통점
- 개발자들이 공통으로 사용하고 싶은 기능을 모듈화 해둔 것!
- 정해진 규칙(코드의 규칙)으로 개발

### 차이점
- 라이브러리
	- 자유도가 높음
	- 그냥 내 코드에 import해서 가져다가 쓰면 되는거임.
- 프레임워크
	- 자유도 낮음
	- 프레임워크의 개발방식(디자인 패턴)을 지키면서 개발을 해야함
		- 예를 들어서 Spring을 쓰면 Controller, Service, Repository가 없을 수!가 없음
		- Spring Security를 쓰면 무조건 정해진 클래스를 정의해야함 -> 정해놓은 보안 체인을 구현하는 방식임

# 3. 싱글톤 패턴

- 하나의 클래스는 하나의 인스턴스만 가진다!
	- 주로 DB 연결에 많이 사용함 -> 하나의 서버에서 하나의 DB에 여러개의 연결을 하면 어떻게 되겠나.. -> 동시에 같은 row를 update할 수도 있음 -> 하나의 연결만 있어야함
		- 따라서 예를 들어서 DBConnector라는 클래스가 있으면 그건 딱 하나만 있어야함.

### 장점
1. 인스턴스 생성 비용 Down(당연히..)
2. 주문 요청을 처리하기 위한 클래스가 계속 만들어지면..
	- 주문 요청을 처리하는 클래스는..
		- 디스크랑도 연결
		- 네트워크 통신 연결
		- DB에도 연결
		- ...등등 많은 작업을 해야지 동작하는 클래스임
	- 그럼 당연히 서버에 부하가 많이 생김

# 단점
1. 의존성이 높아짐 -> 인스턴스를 하나만 사용 -> 해당 인스턴스로 모든 작업을 해야함 -> 다른 코드들이 해당 코드에 의존하는 경우가 높아짐
	- 즉 싱글톤 클래스 A가 바뀌게 되면 -> 이에 영향을 받는 B 클래스도 바뀌어야함
2. TDD의 걸림돌 -> 단위 테스트는 모든 테스트가 독립적(어떤 순서로 실행해도 상관없음)
	싱글톤 패턴은 각 테스트 마다 "독립적인" 테스트를 만들기 어려움
	- 테스트의 순서에 따라서 테스트의 결과가 달라질 수도 있음
	- 앞선 테스트에 의해서 변화가 생기면 뒤의 테스트에 영향을 미침

#### Spring은 싱글톤 -> 어떻게 단점을 해결할까?

1. 의존성
	1. 의존성의 문제는 뭘까? A 코드가 바뀌면 B도 바뀌어야 한다는 것이다
	2. 그럼 이 문제를 어떻게 해결할까?
	- Spring은 구현체 클래스를 의존하지 않음 -> 무조건 추상화되어 있는 인터페이스를 의존하게 설계된다.
	- 또한 DI(의존성 주입)을 하기 때문에 B는 구체적인 클래스가 아닌 인터페이스에 의존하면서 해당 의존성 자체를 프레임워크에서 주입하기 때문에, B는 인터페이스만 알고 있고, 컨테이너가 알아서 런타임에 구현체를 주입함.
	- 따라서, 특정 코드에 의존하지 않기 때문에 이 문제를 해결할 수 있음
2. TDD 문제(JUnit -> Java 테스트 프레임워크)
	- 왜 TDD에서 문제일까? -> 싱글톤은 전역 -> 테스트를 하나 돌리면서 바뀌는 하나의 인스턴스가 다른 테스트에도 영향을 미침
	- DI를 하기 때문에 진짜 싱글톤 인스턴스가 아니라 가짜(Mock)객체를 넣을 수 있음!

# 3. 팩토리 패턴
- 상속 관계에서 상위가 뼈대
- 하위가 구체

그냥 Spring 그 자체임 -> 상위 인터페이스(인터페이스 자체가 그냥 뼈대) -> 하위 구체 클래스

#### 그렇다고 추상화가 목적은 아님!! 생성의 책임을 하나가 지게 하는 것이 목적임
```java
// 직접 생성 (결합도 높음)
Animal a = new Dog();   // 호출자가 Dog라는 구체 클래스를 알아야 함

// 팩토리 사용
Animal a = AnimalFactory.create("dog");  // 호출자는 Dog를 몰라도 됨
```
- 어떤 객체를 만들지를 호출하는 쪽에서 알지 못하게 함.

**언제 팩토리가 정당한가**
팩토리는 공짜가 아님! 클래스가 하나 늘고 코드가 복잡해짐. 아래 같은 경우에 사용하는 것.
- 생성 로직이 복잡하다 (조건에 따라 다른 객체, 초기화 과정이 길다)
- 어떤 구현체를 만들지 **런타임에 결정**된다
- 생성 코드가 여러 곳에 중복되어 한 곳으로 모으고 싶다.

#### Spring에서의 팩토리 패턴
`@Controller`, `@Service` 같은 어노테이션을 붙이면, Spring이 시작할 때 그 클래스들을 **직접 인스턴스화해서 컨테이너 안에 보관**합니다. 우리는 그 보관된 객체를 꺼내 쓸 뿐입니다.

순서는 이렇습니다.
```
1. 스캔   : @Controller 등이 붙은 클래스를 모두 찾는다
2. 등록   : "이런 객체를 만들어야 한다"는 설계도(BeanDefinition)를 등록한다
3. 생성   : 설계도를 보고 컨테이너가 실제로 new 한다
4. 주입   : 그 객체가 필요로 하는 다른 객체를 찾아 넣어준다
5. 보관   : 완성된 객체를 싱글톤 레지스트리에 저장한다
```
<mark class="hltr-yellow">핵심은 **3번에서 우리가 아니라 컨테이너가 `new`를 한다**는 점입니다.</mark>

- **Abstract Factory**: **관련된 객체들의 묶음(family)**을 생성합니다. 예: "윈도우용 버튼+체크박스" 세트 vs "맥용 버튼+체크박스" 세트.

# 4. 이터레이터

- 반복자(Iterator)를 통해서
- 컨테이너 요소들을 하나하나 접근하는 디자인 패턴

### 장점
- 각기 다른 자료구조를 똑같은 인터페이스로 순회할 수 있음
- 컨테이너(동일한 요소들을 담아놓은 집합 - )

**1. Spring Data의 `Streamable` / `Page`**

Spring Data JPA에서 조회 결과를 순회할 때 이터레이터 추상화를 씁니다.

```java
public interface UserRepository extends JpaRepository<User, Long> {
    Streamable<User> findByActiveTrue();
}

// 사용하는 쪽
Streamable<User> users = userRepository.findByActiveTrue();
for (User user : users) {   // 내부 구현을 몰라도 순회
    process(user);
}
```

`Streamable`은 `Iterable`을 확장한 인터페이스입니다. 결과가 메모리의 List인지, DB 커서를 통해 흘러오는지 호출자는 신경 쓸 필요가 없습니다.

### 쓸데없이 알아본 내용 -> Java의 Iterator와 Spring의 Streamable
### 1. `Iterable<T>` — Java 표준 (java.lang)

이터레이터 패턴의 출발점입니다.
```java
public interface Iterable<T> {

    // 핵심: 이 컬렉션을 순회할 Iterator를 반환한다
    Iterator<T> iterator();

    // Java 8에서 추가된 default 메서드 (직접 구현 안 해도 됨)
    default void forEach(Consumer<? super T> action) {
        Objects.requireNonNull(action);
        for (T t : this) {
            action.accept(t);
        }
    }

    // Java 8에서 추가, 병렬 처리·Stream 변환의 기반
    default Spliterator<T> spliterator() {
        return Spliterators.spliteratorUnknownSize(iterator(), 0);
    }
}
```
- 핵심 : 추상 메서드가 `iterator()` 하나뿐 -> T타입을 읽을 수 있는 Iterator를 반환한다.

### 2. `Streamable<T>` — Spring Data (org.springframework.data.util)

이제 Spring 쪽입니다. 깃허브 원본 기준으로, 선언부는 이렇습니다.

```java
@FunctionalInterface
public interface Streamable<T> extends Iterable<T>, Supplier<Stream<T>> {
    ...
}
```
- Iterable과 `Supplier<Stream<T>>`를 동시에 확장합니다. 즉  **"순회도 되고(Iterable), Stream으로도 변환되는(Supplier) 대상"** 이라는 뜻입니다. Iterable을 더 쉽게 스트림으로 다루게 해주는 인터페이스입니다.

### 3. `? super T` — 하한 경계 (T 또는 그 부모)


```java
default void forEach(Consumer<? super T> action)
```

`? super T`는 **"T이거나, T의 부모 타입"**이라는 뜻입니다. super는 "상위(부모)"를 가리킵니다.

상속 관계를 예로 들겠습니다.

```
Object          (최상위 부모)
   ▲
 Animal         (부모)
   ▲
  Dog           (자식)  ← 여기가 T라고 하자
```

`T = Dog`일 때 `? super T`(= `? super Dog`)가 허용하는 타입은 이렇습니다.

```
Dog     ✅ (자기 자신)
Animal  ✅ (Dog의 부모)
Object  ✅ (더 위 부모)
Cat     ❌ (Dog의 부모가 아님)
```

**왜 부모까지 허용하는가.** `forEach`는 컬렉션의 각 요소(`Dog`)를 꺼내서 `action`에 **넘겨주는(소비하는)** 역할입니다. 그런데 `Dog`를 처리할 수 있는 함수는 `Dog` 전용 함수만이 아닙니다.

```java
List<Dog> dogs = ...;

// Dog를 받는 함수 — 당연히 가능
Consumer<Dog> dogPrinter = dog -> System.out.println(dog.bark());

// Animal을 받는 함수 — 이것도 가능해야 한다!
Consumer<Animal> animalPrinter = animal -> System.out.println(animal.getName());

dogs.forEach(dogPrinter);     // OK
dogs.forEach(animalPrinter);  // 이것도 OK여야 자연스럽다
```

`Dog`는 `Animal`이기도 하므로, **`Animal`을 처리하는 함수에 `Dog`를 넘기는 건 문제없습니다.** "동물을 출력하는 함수"에 "개"를 줘도 잘 작동합니다. `? super T`가 이걸 허용해줍니다.

만약 `Consumer<T>`라고만 했다면 `Consumer<Animal>`을 넘길 수 없어 불필요하게 제약이 생깁니다. `? super T`로 **더 일반적인 함수도 받아들일 수 있게** 유연성을 준 것입니다.

### 4. 짝꿍 개념: `? extends T` — 상한 경계 (T 또는 그 자식)

`super`를 이해했으면 반대편도 알아야 짝이 맞습니다. `forEach`엔 없지만 자바 어디에나 나옵니다.

`? extends T`는 **"T이거나, T의 자식 타입"**입니다.

```
T = Animal 일 때, ? extends Animal:
Animal  ✅ (자기 자신)
Dog     ✅ (Animal의 자식)
Cat     ✅ (Animal의 자식)
Object  ❌ (Animal의 부모라서 안 됨)
```

이건 **데이터를 꺼내 읽을 때(생산)** 씁니다.
```java
// "Animal이거나 그 자식들"이 든 리스트를 받는다
void printAll(List<? extends Animal> animals) {
    for (Animal a : animals) {   // 꺼내면 최소한 Animal임이 보장됨
        System.out.println(a.getName());
    }
}

printAll(dogList);  // List<Dog> 넘기기 가능
printAll(catList);  // List<Cat> 넘기기 가능
```

### <mark class="hltr-yellow">핵심 원칙: PECS</mark>

이 둘을 언제 쓰는지 외우는 유명한 약어가 있습니다. <mark class="hltr-yellow">**PECS: Producer Extends, Consumer Super.**</mark>

- 데이터를 **꺼내 읽기만**(생산, Producer) → `? extends T`
- 데이터를 **받아 넣기만**(소비, Consumer) → `? super T`

`forEach`를 다시 보면 명확합니다.

```java
default void forEach(Consumer<? super T> action)
//                   ↑          ↑
//          이름부터 Consumer    그래서 super
```

`Consumer`는 값을 **받아서 소비**하는 역할이니, PECS 원칙대로 `super`를 씁니다. 이름과 문법이 일관됩니다.

# 5. DI와 DIP

- 메인 모듈(우리가 짠 메인 코드)가 직접 다른 하위 모듈에 의존성을 주입하지 않고
- 중간에 의존성 주입자가 '간접적으로' 의존성을 주입

```css
**<개인적인 생각>**
- 그냥 일반적으로 내가 코드를 짜게되면 거의 모든 코드가 서로를 의존하게 짜게 된다.
```

MVC MVP MVVM 패턴의 예시를 중학생도 이해할 수 있게 설명해줘

오전 12:38

좋습니다. 이 세 패턴은 비유 하나로 잡으면 훨씬 쉽습니다. **식당**에 비유해서 설명하겠습니다.

### 먼저 공통점: 왜 이런 걸 나누나

세 패턴 모두 목적이 같습니다. **"화면(보이는 것)"과 "데이터/로직(속 내용)"을 섞지 말고 분리하자.**

왜냐면 다 섞어 짜면, 버튼 색 하나 바꾸려다 데이터 처리 로직까지 건드리게 되고, 코드가 엉망이 되기 때문입니다. 그래서 역할을 나눕니다. 셋의 차이는 **"이 분리를 어떻게 하느냐"**일 뿐입니다.

공통으로 등장하는 세 조각부터 알아두면 됩니다.

- **Model (모델)**: 데이터와 그 데이터를 다루는 규칙. (식당의 **주방·창고**)
- **View (뷰)**: 사용자가 보는 화면. (식당의 **메뉴판·테이블**)
- 그리고 이 둘을 **연결하는 중간 역할** — 이 중간 역할의 이름과 방식이 세 패턴을 가릅니다.

### 1. MVC — 중간에 "직원(Controller)"이 있다

**M**odel - **V**iew - **C**ontroller

식당으로 보면 이렇습니다.

- **손님(사용자)**이 메뉴판(View)을 보고 주문합니다.
- 주문을 **직원(Controller)**이 받습니다.
- 직원이 **주방(Model)**에 "이거 만들어주세요" 하고 전달합니다.
- 주방이 요리를 만들면, 그 결과가 다시 화면(View)에 나옵니다.

```
손님 ──주문──▶ 직원(Controller) ──전달──▶ 주방(Model)
                                              │
화면(View) ◀──────────결과 반영──────────────┘
```

핵심은 **직원(Controller)이 교통정리를 한다**는 것입니다. 손님의 요청을 받아서 주방에 넘기고, 결과를 화면에 띄웁니다.

**어디서 쓰나**: 웹 백엔드. 앞서 계속 얘기한 **Spring MVC**가 이것입니다. 브라우저(View)가 요청하면 Controller가 받아 Service/Model에 넘기고, 결과를 화면으로 돌려줍니다.

### 2. MVP — 중간 역할이 "매니저(Presenter)"로 더 똑똑해진다

**M**odel - **V**iew - **P**resenter

MVC와 비슷한데, 중간 역할이 **훨씬 많은 일을 직접 처리**합니다.

식당으로 보면, 직원이 아니라 **유능한 매니저(Presenter)**가 붙은 셈입니다.

- 메뉴판(View)은 정말 **아무 생각이 없습니다.** 그냥 보여주기만 합니다.
- 손님이 뭘 누르면, 메뉴판은 "손님이 이거 눌렀어요"라고 매니저에게 **그대로 전달만** 합니다.
- **매니저(Presenter)**가 모든 판단을 합니다. 주방(Model)에서 데이터를 가져오고, 계산하고, "화면에 이렇게 보여줘"라고 메뉴판에 **하나하나 지시**합니다.

```
메뉴판(View) ◀──"이렇게 보여줘"라고 일일이 지시── 매니저(Presenter) ──▶ 주방(Model)
     │                                                  ▲
     └──────"손님이 눌렀어요"라고 보고───────────────────┘
```

**MVC와 차이**: MVC의 View는 자기가 직접 Model을 좀 들여다보기도 하는데, MVP의 View는 **완전히 멍청합니다.** 모든 걸 Presenter가 대신 해주고, View는 Presenter가 시키는 대로만 합니다. View와 Model이 **직접 만나지 않습니다.** (반드시 매니저를 거칩니다.)

**어디서 쓰나**: 예전 안드로이드 앱 개발에서 많이 썼습니다. 화면 테스트를 쉽게 하려고 View를 최대한 비워둔 구조입니다.

### 3. MVVM — 중간 역할이 "자동 연결 장치(ViewModel)"가 된다

**M**odel - **V**iew - **V**iew**M**odel

여기가 가장 신기한 부분입니다. 중간 역할이 사람처럼 일일이 지시하는 게 아니라, **화면과 데이터가 자동으로 연결**됩니다.

식당으로 비유하면, **메뉴판이 전광판으로 바뀐 것**입니다.

- 주방(Model)의 재료가 떨어지면, 전광판(View)의 "품절" 표시가 **자동으로 켜집니다.**
- 누가 가서 바꾸는 게 아니라, **데이터와 화면이 실시간으로 묶여(binding) 있어서** 한쪽이 바뀌면 다른 쪽이 알아서 따라 바뀝니다.

이 "자동 연결"을 담당하는 게 **ViewModel**입니다.

```
        ┌──── 자동 양방향 연결 (Data Binding) ────┐
        ▼                                        ▼
화면(View)  ◀──────────────────────────▶  ViewModel ──▶ 주방(Model)
   "데이터가 바뀌면 화면도 자동으로 바뀜"
   "화면을 조작하면 데이터도 자동으로 바뀜"
```

**MVP와 차이**: MVP는 매니저(Presenter)가 "이거 보여줘, 저거 바꿔" 하고 **일일이 손으로 지시**합니다. MVVM은 그 지시를 **자동화**합니다. ViewModel의 데이터만 바꾸면 화면이 알아서 갱신됩니다. 개발자가 "화면을 업데이트하라"는 코드를 직접 안 써도 됩니다.

**어디서 쓰나**: 현대 프론트엔드. **Vue, 안드로이드의 Jetpack, React**(개념적으로 유사) 등. 앞서 첫 대화에서 React를 Observer 패턴과 연결했는데, 이 "데이터가 바뀌면 화면이 자동으로 바뀐다"는 게 바로 그 연장선입니다.

### 한눈에 비교

세 가지를 "중간 역할이 얼마나, 어떻게 일하는가"로 보면 명확합니다.

|패턴|중간 역할|화면(View)은|핵심 차이|
|---|---|---|---|
|**MVC**|Controller (직원)|어느 정도 똑똑함|직원이 요청을 받아 분배|
|**MVP**|Presenter (매니저)|완전히 멍청함|매니저가 모든 걸 일일이 지시|
|**MVVM**|ViewModel (자동연결)|데이터에 묶여 있음|화면과 데이터가 자동 동기화|

흐름으로 외우면 이렇습니다. **MVC → MVP → MVVM** 순으로, 중간 역할이 점점 더 똑똑해지고, 화면 업데이트가 점점 더 자동화됩니다.

### 비판적으로 짚을 점

중학생도 이해하도록 단순화했으니, 한 가지는 솔직히 말씀드려야 합니다.

**실무에서 이 셋의 경계는 비유처럼 깔끔하지 않습니다.** 특히 현대 프레임워크는 이 패턴들을 변형하고 섞어 씁니다. 예를 들어 React를 두고 "MVVM이다 / 아니다"로 논쟁이 있을 만큼, 딱 잘라 분류하기 어렵습니다. 그러니 **"이 프레임워크 = 이 패턴"으로 1:1 암기하는 건 위험**합니다. 지금까지 다른 패턴들에서 계속 나온 교훈과 똑같습니다.

이 세 패턴에서 진짜 가져갈 핵심은 분류 자체가 아니라 이것입니다. **"화면과 데이터/로직을 분리하고, 그 둘을 잇는 중간 역할을 어떻게 설계할 것인가"** — 이게 본질이고, MVC/MVP/MVVM은 그 질문에 대한 세 가지 답일 뿐입니다.

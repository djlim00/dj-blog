# 1. 세션 기반 인증방식

- HTTP는 무상태성(로그인을 어떻게 기억하지?)

-  <mark style="background: #FFF3A3A6;">세션</mark> : 서버와 클라이언트의 연결이 활성화된 상태
-  <mark style="background: #FFF3A3A6;">세션ID</mark> : 웹 서버 또는 DB에 저장되는 클라이언트에 대한 유니크한 ID

1. 로그인 -> 세션id생성 -> 서버에서 암호화된 세션id를 쿠키로 전달
2. 클라이언트가 세션 id를 쿠키에 담아서 요청을 보냄

- DB에 세션id를 저장하는 경우 : 직렬화, 역직렬화 비용 발생
- WAS에 저장하는 경우 : 사용자가 많아지면 서버 과부화 심해짐

# 2. 토큰 기반 인증방식

- 토큰 : 유저의 상태를 저장
- 서버는 Stateless하게(세션 유지할 필요 없음)
- 인증로직 -> 따로 서버를 두고 관리해야함(안그러면 전부 로그인이 불가능해짐)

1. JWT토큰 생성(Access토큰, Refresh토큰)
2. 사용자가 요청에 Access Token을 헤더에 넣어서 보낸다.

### JWT

1. Header : 토큰 유형, 알고리즘
2. Payload : 정보들, 토큰 발급자, 유효기간
3. Signature : 헤더와 payload를 기반으로 생성된 값

- **장점**
	- 별도의 인증 저장소가 필요 없음(토큰에 다 있으니까)
	- 다른 토큰들에 비해서 가벼움
	- 디코딩하면(Base64) JSON이 나옴
- **단점**
	- 토큰 과부화 -> 서버가 토큰 보는게 부담
	- 토큰 탈취

- Access 토큰은 짧게, Refresh토큰은 길게!

# 3. HTTP 상태코드

#### 1xx(정보)
- 서버가 요청 잘 받았고, 프로세스 이어가겠음
	- 100 : 계속함을 의미

#### 2xx(성공)
- 요청 잘 받아서 클라이언트에 데이터 성공적으로 보냄!
	- 200 OK : 요청 성공적으로 처리
	- 201 Created : 성공처리, 새로운 리소스 생성

#### 3xx(리다이렉션)
- 요청 완료를 위해서 추가 작업이 필요함
	- 301 Moved Permanently : 리소스 URI가 변경되었음(변경된 URI를 301이랑 함께 줘야함)
		- 보통 뭐 주소가 바뀐 경우

#### 4xx(클라이언트 오류)
- 클리언트의 요청이 잘못된 경우
	- 401 Bad Request : 클라이언트 요청이 이상함
	- 401 Unauthorized : 인증 불가
	- 404 Not Found : 요청한 컨텐츠 없음

#### 5xx(서버 오류)
- 서버가 요청 처리를 못함
	- 500 Internal Server Error : 서버에 오휴
	- 502 Bad Gateway : 게이트웨이, 프록시 서버 오류
	- 504 Gateway Timeout : Timeout날때까지 요청을 처리 못함

# 4. GET과 POST의 차이

#### GET 
- GET은 url을 기반으로 데이터 요구(쿼리 파마미터)
- 성공 -> 200
- 민감한 정보는 전달하지 않음

#### POST
- HTTP Body로 데이터를 전달
- 201반환 -> 생성 시, 200반환 -> 생성은 안함
- 민감한 정보 전달

#### PUT
- 데이터 전체 교체 -> 없으면 생성

### PATCH
- 일부분만 수정

# 6. 전이중화 통신

- 양쪽 장치가 동시에 송수신할 수 있는 방식
- 😥 과거의 반이중화 통신은 회선이 충돌하는지 확인을 해야했음..
#### 유선 LAN
- 동축케이블, 광케이블 등을 기반으로 송수신

#### 무선 LAN
- <mark class="hltr-yellow">반이중화 통신 사용</mark>
	- 동시에 수신 불가능(한번에 한 방향만 통신 가능)
	- 전이중화는 회선이 두개임
- WIFI
	- 전자기기들이 무선 LAN 신호에 연결
	- 유선 LAN을 AP 또는 공유기에 꽂아서 무선 LAN 만듦
- 2.4GHz
	- 장애물에 강함
	- 속도가 낮음
	- 호환성이 좋음(오래된 노트북, 휴대폰)
	- 커버리지가 높음
-  5GHz
	- 장애물에 약함
	- 속도가 빠름
	- 호환성이 좋음
	- 커버리지가 낮음

# 2. Split-Brain
![[Pasted image 20260613101821.png|427]]
- 네트워크 분단이 순진한(naive) 리더 선출 프로토콜을 만나면 발생!
- 두 절반이 연결이 끊기면 -> leader가 조용함 -> 새로운 leader 선출

**[다이어그램: Split-Brain]**
- 양쪽 절반이 각자 리더를 선출하고 서로 충돌하는 쓰기를 받아들임
- 클러스터 절반 A: PRIMARY — Leader A — user.balance = $100
- 클러스터 절반 B: PRIMARY — Leader B — user.balance = $50
- 분단이 복구되면: 어느 값이 이기는가? 두 개의 유효한 히스토리는 깔끔하게 병합될 수 없음

- 처음에는 둘다 받아들이고 쓰고 있으니까 잘 돌아가는 것 같음
	- 근데 다시 회복이 되서 연결이 되면 두개가 충돌함(절대 내부적으로 통합 불가능)
	- 같은 row에 두개의 다른 leader로 부터 받은 내용이 작성됨
	- 둘 중 하나를 선택해서 merge를 할 수가 없음(알아서)

# 두가지 레이어로 해결 가능
1. <mark class="hltr-yellow">**쿼럼 기반 리더 선출**:</mark> 분단의 과반수 쪽만 새 리더를 선출할 수 있게 하여, 소수파 분단이 자기 멋대로 프라이머리를 옹립하지 못하게 합니다.
2. <mark class="hltr-yellow">**펜싱 토큰(fencing token)**:</mark> 한 단계 더 나아가, 새 리더마다 단조 증가하는 번호를 발급하고, 더 높은 번호의 리더가 존재하는 순간 폐위된 리더가 제출한 작업은 모두 거부합니다. 옛 리더가 여전히 살아 있고 자신이 폐위된 줄 모르더라도, 그 쓰기 작업은 피해를 입히기 전에 차단됩니다.

스플릿 브레인은 네트워크 분단 없이도 발생할 수 있습니다. 리더가 긴 가비지 컬렉션 사이클로 30초간 멈추면, 피어들은 리더가 죽었다고 선언하고 후임자를 선출할 수 있는데, 정작 원래 리더는 몇 초 뒤 깨어나 작업을 재개하기 직전일 수 있습니다. 클러스터의 관점에서, 충분히 긴 멈춤(pause)은 분단과 구별이 불가능합니다.

![[Pasted image 20260613104217.png|415]]
**[다이어그램: Partial Failures — 똑같아 보이는 네 가지 결과]**
- 클라이언트 → 요청 → 서버 / 응답은... 올 수도 안 올 수도?
- 구별 불가능한 네 가지 결과:
    1. **도중에 유실** — 요청이 서버에 도착조차 못 함
    2. **먼저 죽음** — 서버가 처리 전에 크래시
    3. **응답 유실** — 작업은 성공했으나 응답이 유실됨

- 타임아웃은 "실패했다"가 아니라 **"모른다"**를 의미. 3번 케이스에서 재시도하면 요청이 두 번 처리됨. 멱등성(idempotency)이 중요한 이유.

# 4. Gray Failures

- 모든게 정상적이라고 뜨는데, 실제 해야하는 작업에 실패하는 경우
	- 개발팀이 만든 헬스체크 같은 내부점검을 통과하는데 느림..
![[Pasted image 20260613104931.png|376]]
전형적인 사례는 쓰기 자체는 성공하지만 완료까지 30초가 걸리는 디스크입니다. 프로세스는 살아 있고, OS 수준에서 디스크는 정상으로 보고되고, 디스크를 직접 건드리지 않는 헬스 체크는 전부 초록불입니다. 하지만 그 디스크를 거치는 실제 쿼리는 전부 타임아웃이 나고 있고, 다운스트림 서비스들은 이 느린 이웃 하나 때문에 자기네 성공률이 무너지는 것을 지켜보게 됩니다.

노드가 "살아 있으면서 고장난" 상태가 되는 경로는 다양합니다:

- 네트워크 카드가 한 방향으로만 패킷을 버려서, 보내는 건 되는데 받는 게 점점 안 되는 경우
- 프로세스가 긴 가비지 컬렉션에 빠져 한 번에 10초씩 응답을 멈추는 경우
- 설정 변경으로 응답 내용이 미묘하게 깨졌는데 상태 코드는 계속 200인 경우
- 메모리 누수로 캐시가 굶주려서, 프로세스는 한 번도 죽지 않았는데 캐시 적중률만 폭락하는 경우

헬스 체크가 이런 장애를 놓치는 이유는, 헬스 체크는 자신이 측정하도록 설계된 것만 측정하기 때문입니다. 프로세스에 핑을 보내는 체크는 "프로세스가 돌고 있다"는 것만 알려줄 뿐, "프로세스가 실제로 쓸모 있는 일을 하고 있는가"는 알려주지 않습니다.

해결책은 메타데이터가 아니라 **실제 워크로드를 검사하는 관측(observability)**입니다. 실제 코드 경로를 그대로 타는 합성 트랜잭션(synthetic transaction), 이웃 노드들이 서로의 동작을 감시하는 피어 기반 탐지, 그리고 고객 관점에서 측정하는 종단 간(end-to-end) 지연 시간 측정 같은 접근이 여기에 해당합니다.

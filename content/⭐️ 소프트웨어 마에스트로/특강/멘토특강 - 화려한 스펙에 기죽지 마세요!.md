## 장진영 멘토님(IOS) - "어떻게 나의 무기를 만들 수 있을까?"

## 1. 
- 일단 모든 것들은 있긴한데 -> 괜찮은 서비스는 딱히 없다! 이미 있어도 선택할 수 있다.
- 우리가 전문가거나, 관심이 참 많은 분야에 대해서 선택을 하는 것이 가장 좋다!
	- 특히 학생 신분이라면, 학생들에게 관심이 있다던지 대학생이 고민할만할 것들을 하는 것이 가장 좋을 수 있다.
- 연인들 쿠폰 주고 받기 -> 밥 다섯번 사주기 쿠폰
- 트렌드 맞춰서 하자! - 적어도 뒤쳐지지는 말자 - 예를 들어서 뭐 공유 서비스 이런건 진부함
## 2. 
- 요즘은 아무래도 AI가 맞다! -> 너무 뒤쳐지면 심사위원의 눈에 들어오지도 않는다.
	- 트렌드를 쫒아가다보면 기획의 타당성을 떨어진다
	- 이도저도 아니면 사회적인 뭔가가 있어야한다.(사회를 돕는 서비스)
		- 시니어, 장애, 불우이웃(윗분들이 좋아함, 설명을 꼭 할 필요는 없음)
## 3. 
- <mark class="hltr-yellow">주제는 타겟팅은 진짜 확실하게 해야함.(최대한 작게 잡고 만든 다음에, 나중에 확장성 있게)</mark> 
	- 8~9월까지 MVP를 만들 때는 타겟팅은 진짜 확실하게 해야한다.
	- UX/UI는 진짜 타겟층에 확실하게 하면 되는 문제다.
	- 그래야지 프로젝트 기간동안 흔들리지 않는다(회의 리소스 낭비를 줄일 수 있음).
## 4. 
- <mark class="hltr-yellow">주제 속에 나만의 기술적 챌린지를 심자</mark>
	- ex) 강아지 산책 -> Core Location 심화 활용
	- ex) 식단 관리 -> 대용량 영양소 데이토 처리 및 이미지 인식 기술 도입
	- ex) 동화 생성 앱 : 생성형 AI 연동 및 이미지/텍스트 조화 최적화
- <mark class="hltr-yellow">메인 기능을 무조건 하나씩 잡아야함.</mark>
	- 특정한 사람한테 메인이 몰리지 않도록 -> 각자의 포트폴리오를 완성하기 위해서
## 5. 
- R&R(Role & Responsibility)은 확실하게
- <u>맹목적 비난 금지, 논리 환영!</u> - '코드와 문서'로 대화
- 형식적인 회고는 No! : 좋은 점(keep), 아쉬운 점(Problem), 시도할 점(Try)은 짧고 굵게!
## 6. 
- 우리 나라 사람들은 예쁜 것들을 좋아한다 -> 외주로라도 구해야함!
- 디자인에 취해서 본질을 놓쳐서는 안된다!
## 7. 
- <mark class="hltr-yellow">덜어내기의 미학</mark>
	- 로그인 같은 없으면 엔트리가 안되지만, 무조건 필요한 것들은 최대한 짧게 해야한다.
	- "이 기능 없으면 배포 못 한다 싶은 것들만 남기기"
## 8. 
- 억지스러운 독창성 보다는 시스템 표준 UI를 따르자! 
- HIG(Human Interface Guidelines) 준수

# 2. 어떻게 기록을 할 것 인가?

- <mark class="hltr-yellow">이슈를 생성하고 커밋 로그와 연결하는 습관이 실무의 시작이다</mark> -> 이슈 넘버와 커밋을 연결

## 2. Commit, PR 어떻게 할 지 정하자
- Commit message랑 PR이랑 Claud Code랑 Jira랑 죄다 연동을 시켜두자.

## 3. QA 기간은 최소 3일은 잡아서 하자.
- 계획 -> 개발 -> QA -> 고치기 : 싸이클에 익숙해지자.
- <mark class="hltr-yellow">QA 이슈를 와르르 만들어서 Tag에 Major, Minor, Critical, Trivial(이거 우선순위 별로 QA 쳐내는 경험을 해봤다.) => 어필요소임!!</mark>

## 4. 로그 심기!!!
1. 로그 다이어트가 핵심!!
	- 너무 많은 로그는 소음
	- 운영 초반에는 카테고리(기능)별로 우선순위를 정하고 최대 개수를 정해서 남기기
		- 이게 어떻게 운영에 도움이 되는지를 생각하자.(카테고리별로 3~4개 정도)
	- 시각화
	- 로그에 따라서 기능개선을 어떻게 했는지(~데이터를 심어서 보니까 ~하다고 나와서 ~게 바꾸니까 ~만큼 효과가 이었다.)
2. 프론트단에서 A/B 테스트를 한다던지 뭐 이런 것들도 괜찮음.
3. 운영할 때는 진짜 고객 문의가 많이 들어오는데, 크래시가 나기 직전까지 유저가 클릭한 지점을 저장 -> 어떤 경로로 크래시가 되는 것에 들어왔느지를 확인한다.
	- 앱 내에서 숨은 디버깅 공간에서 로그를 뽑을 수 있음,, 뭐 이런 식으로

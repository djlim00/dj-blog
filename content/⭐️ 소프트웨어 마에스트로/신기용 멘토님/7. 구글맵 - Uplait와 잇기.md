## 1. 지도 101

#### 측위 시스템
- 지구(구) 표면 상의 위치를 표현하는 체계
- 위경도 기반 측위 시스템
![[Pasted image 20260626111531.png|297]]
- 위도(lat) : 북/남
- 경도(lon) : 서/북 

#### 그럼 이 3차원의 구를 2차원 평면에 두는 것 -> 지도 투영법(도법)

- 하지만, 각각의 방법은 왜곡이 발생할 수 밖에 없음.
	- 구글 -> 웹 메르카토르 기법 

#### 지오코딩(Geocoding)
- 주소를 측위 시스템으로 ex) 성남시 분당구 판교로50 -> {36.xxx, 20.xx}
- 역 지오코딩 -> 위경도를 주소로

- 인터폴레이션을 이용해서 수행할 수 있음 ex) GIS(Geographic information System)

#### 지오해싱(Geohashing)
- 지도 위 특정 영역을 영문자와 숫자로 구성된 짧은 문자열에 대응시키는 인코딩 체계.
- 2차원의 평면 공간으로 표현된 지리적 영역 위의 격자를 더 작은 격자로 재귀적으로 분할
	- 격자 -> 정사각형 or 사각형
	- 재귀적으로 분할된 더 작은 격자 -> 0~3사이의 번호를 부여
>wydm6 → 강남역 근처 
>wydm7 → 그 바로 옆 칸 
>wydkx → 조금 떨어진 곳

- 격자 칸 경계에 걸친 두 지점은 실제로 매우 가까워도 앞글자가 완전히 달라질 수 있습니다. 
	- 예를 들어 칸 경계를 사이에 두고 10m 떨어진 두 점이 `wydm6`과 `wydkx`처럼 공통 접두사가 짧아질 수 있습니다. 
	- 그래서 "근처 검색"을 제대로 하려면 **중심 칸 + 주변 8칸까지 함께 조회**해야 누락이 안 생깁니다.

# 2. 지도 표시

- 가장 기본 -> <mark class="hltr-yellow">타일</mark>
- 클라이언트는 사용자가 보려는 영역에 관계된 타일만 다운받아 모자이크처럼 이어 붙인 다음 화면에 뿌린다.
- 지도 확대/축소
	- 확대 수준에 따른 다른 종류의 타일을 준비

#### 경로 안내 알고리즘을 위한 도로 데이터 처리

- 대부분 다익스트라 or A* 경로 탐색 알고리즘의 변종
- 교차로를 노드(node), 도로는 노드를 잇는 선(edge)로 표현하는 그래프 자료 구로를 가정
	![[Pasted image 20260628172857.png]]
- 알고리즘 성능 -> 그래프 크기에 아주 민감함.
	- 전세계 도로망을 하나의 그래프로 표현 -> 메모리 Up, 성능 Down
	- <mark class="hltr-yellow">그래프를 관리 가능한 단위로 분할해야함.</mark>
	- 타일 기반 분할법과 매우 유사하게 접근(지오 해싱)

- 타일 기반 분할법과 매우 유사하게 접근(지오 해싱)
	- 그래프를 작게 나눔
	- 각각의 격자 -> 경로 안내 타일(routing tile)
	- 경로 안내 타일은 연결된 다른 타일에 대한 참조(reference)를 유지 -> 이래야지 연결된 타일들을 지나갈 때 보일 더 큰 도로망 그래프를 만들 수 있음
	![[Pasted image 20260628173257.png]]
>위의 그림의 routing tile은 특정한 지역을 포괄 -> 지도의 타일과 유사
>그러나 지도 타일=PNG(이미지), routing tile=binray file(도로 데이터)

#### 계층적 경로 안내 타일
- 경로 안내 -> 필요한 수준의 구체성을 지닌 도로 데이터가 필요함(지도!!)
	- ex) 국토 종단 여행
		- 지번 수준 정밀도 타일은 너무 많은 시간이 걸림
		- 메모리 Up, 그래프 크기 Up
- 보통 구체성 정도를 상, 중, 하로 구분
	- 상 : 가장 구체성 높음, 지방도(local roads) 데이터
	- 중 : 관할구를 잇는 간선 도로 데이터
	- 하 : 도시와 주를 연결하는 고속도로 데이터
	![[Pasted image 20260628173608.png]]
	- 각 타일에는 다른 정밀도 타일로 연결되는 edge가 있을 수도 있다.
		- ex) 도로 A의 특정 지점을 나타내는 노드엣 도로 F의 특정 지점을 나타내는 노드 사이에 연결선이 있어야함.

#### 1. 세계 지도

- 확대 수준에 따른 지도 타일을 한 벌 씩 두어야함.
- 전부 보관 용량 -> 최대 확대 수준(최대로 zoom in) 했을 때의 타일 개수를 따지자.
	- 확대할 때마다 하나의 타일을 4장의 타일로 -> 21번 확대 -> 약 4.4조 개의 파일
	- 한장의 타일이 256x256픽셀 압축(PNG) 파일 -> 한 장당 100KB 저장 공간
	- 전부 저장하는데 440PB 만큼의 저장 공간
	![[Pasted image 20260628174101.png]]
- 그러나 지구의 90%는 인간이 살지 않는 자연 -> 압축 -> 보수적으로 80%~90% 용량 절감 가능
	- 88PB로 줄어든다! -> 대략 50PB(최대 zoom in 했을 때)
- <mark class="hltr-yellow">확대 수준이 1 줄어든면 1/4된다고 하면 총 약 67PB 정도 된다!</mark>

#### 3. 지도 표시
- 확대 수준별로 한벌씩 지도 타일을 저장하려면 수백 PB가 필요
	- 클라이언트가 가지고 있는 것은 실용적이지 않음.
	- 확대 수준에 따라서 서버에 요청애서 가져와야한다.

##### 클라이언트는 언제 지도 타일을 서버에서 가져오는가?
>1. 사용자가 지도를 확대 또는 이동 시키면서 주변을 탐색
>2. 경로 안내가 진행되는 동안 사용자의 위치가 현재 지도 타일을 벗어나 인접한 타일로 이동한다.

##### 다량의 지도 타일 데이터를 효율적으로 가져오는 방법 : 선택지1
- 현재 클라이언트가 보는 지도의 확대 수준에 근거해 필요한 지도 타일을 즉석에서 만들기.
	- 모든 지도 타일을 동적으로 만들기 -> 서버 클러스터 심각한 부하
	- 캐시 활용 어려움

##### 다량의 지도 타일 데이터를 효율적으로 가져오는 방법 : 선택지2
- 확대 수준별로 미리 만들어 둔 지도 타일을 클라이언트에 전달
- 지도 타일 영역 -> 지오해싱을 통해 만든 고정된(정적인) 사각형 격자.
	- <mark class="hltr-yellow">클라이언트가 필요한 지도 타일 -> 확대 수준에 근거해서 지도 타일 집합을 만들고 -> 각 위치를 지오해시 URL로 변환</mark>
- 미리 만들어둔 정적 CDN을 통해서 서비스.
![[Pasted image 20260628180851.png]]
- 사용자에게 가장 가까운 POP(Pont Of Presence)에서 파일을 서비스
- CDN에 캐싱하기도 좋음

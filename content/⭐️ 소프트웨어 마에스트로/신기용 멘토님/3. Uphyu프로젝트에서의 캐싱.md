- 같은 요청을 어떻게 같은 키로 인식할까?
- 사용자 화면의 bounds를 쓰게 되면, 조금만 움직여도 같은 키가 만들어지지 않는다.

- 지오해시 문자열 -> 앞자리가 같으면 가까움 -> 격자 칸 하나가 곧 캐시 키가 될 수 있음.

> - 사용자 화면 → 덮는 지오해시 타일 목록 = ["wydm6", "wydm7", "wydmd", ...] 
> - 각 타일 = 캐시 키 하나

- 타일이 너무 크면(너무 앞자리만 캐싱) -> 캐싱된 타일 하나가 너무 비대해짐
	- 조금만 움직여도 타일이 바뀌어 버림
- 너무 작으면 -> 조회 횟수가 늘어남

#### 지오해시의 대략적인 칸 크기

| 글자 수 | 칸 크기(가로 기준) | 비고              |
| ---- | ----------- | --------------- |
| 5글자  | 약 5km       | 구 단위, 너무 큼      |
| 6글자  | 약 1.2km     | 동 단위, 도시 지도에 적합 |
| 7글자  | 약 150m      | 너무 세밀, 타일 수 많아짐 |
- 6글자 전후가 괜찮을 듯.
- 하지만 강남 같이 빽빽한 곳은 많이 캐싱하고, 시골은 널널하게 캐싱하는 것이 좋음.

___

#### 1. 캐시 구조 : 타일별로 쪼개 저장
- 화면 전체를 통으로 캐싱하지 않고, 타일 단위로 잘게 나눠서 저장
	- 사용자 A가 본 영역과 사용자 B가 본 영역이 일부만 겹쳐도 그 겹치는 타일이 재사용됨.

```
Redis 키 구조:
  stores:geohash:wydm6  →  [그 타일 안의 제휴처 마커 JSON 배열]
  stores:geohash:wydm7  →  [...]
```

##### 사용자 요청 시 백엔드 동작
```
1. 화면 bounds → 덮는 타일 목록 계산 (예: 9개)
2. Redis에서 9개 키를 한 번에 조회 (MGET — 여러 키 일괄 조회)
3. 캐시에 있던 타일은 그대로 사용
4. 없던(미스) 타일만 PostGIS로 조회
5. 새로 조회한 타일을 Redis에 저장 (다음 사용자를 위해)
6. 전체를 합쳐 프론트에 응답
```
- 여기서 `MGET`으로 여러 타일을 한 번에 가져와야함 ->  타일마다 Redis를 따로 호출하면 네트워크 왕복이 타일 수만큼 발생!


#### 2. 캐미 미스를 채우는 방법

#### 1. 지연 로딩(요청이 올 때 채우기)

- 위의 백엔드 동작
- 구현 단순, 요청이 적은 지역은 캐싱이 없음
- 다만 첫 방문자는 DB 조회를 기다려야함

##### 2. 사전 계산(미리 채워두기)

- 제휴처 위치는 바뀌지 않음.
- 제휴처가 있는 모든 타일을 배치 작업으로 하루에 한 번 미리 계산에 Redis에 올려둔다.
	- 캐시 미스가 사실상 없어짐.
	- 사용자는 항상 캐시를 통해 즉시 응답 받음. -> 책에서의 CDN 내용.

#### 3. 캐시 무효화

- 데이터가 수정/변경 되었을 떄 캐싱 어떻게 갱신할거임?
- 해당 제휴처가 속한 타일의 캐시 키만 삭제(또는 재계산)
	- 제휴처 좌표 -> 지오해시 -> 키 하나만 무효화

```
관리자가 강남역 매장 추가
  → 그 매장 좌표 → 지오해시 "wydm6" 계산
  → Redis에서 stores:geohash:wydm6 키만 삭제 (또는 즉시 재계산)
  → 다음 조회 때 최신 데이터로 다시 채워짐
```

- 여기에 TTL 정도 걸어두기 -> 24시간 만료 해두면, 무효화가 잘 안되도, 하루 안에 채워짐.

# 2. 밀도 차등 캐싱 전략

- 강남구 처럼 제휴처가 많은 동네 vs 시골 같이 없는 동네
	- 둘다 같은 격자로 나누면 비효율 생김

> 강남구에서는 한 격자 안에 제휴처가 수백 개 들어갑니다. 이 격자 하나를 조회하면 마커 수백 개가 한꺼번에 쏟아져 캐시 값이 비대해지고, 프론트는 화면에 다 그리지도 못합니다.
> 
> 반대로 시골에서는 한 격자에 제휴처가 0~1개입니다. 화면 하나를 덮는 데 격자 수십 개가 필요한데 대부분이 비어 있어, 빈 격자를 조회하는 헛수고가 생깁니다.

#### 1. 가변 정밀도(쿼드트리 계열)
- 데이터가 더 많은 곳은 격자를 잘게, 적은 곳은 크게

##### 1. 쿼드트리
- <mark class="hltr-yellow">각 사분면이 관리 가능한 수의 점(제휴처)를 담을 수 있을 때 까지 재귀적으로 4개로 분할하는 트리 구조</mark>

- **단점**
	- 트리 순회 오버헤드
	- 1차원 인덱싱 불가
	- 샤딩이 어려움
	- 지오 해시는 격자 구조가 고정(정적) -> 쿼드트리는 트리 구조가 데이터에 따라 변해 캐시 키로 삼기가 까다로움

##### 2. 고정 지오해시를 유지하되, 밀도별로 정밀도를 다르게 적용
- 지오해시를 그대로 사용하면서, 자체적으로 지역 등급에 따라 글자 수(지호해시 prefix개수)만 바꿔 쓰는 방법

```
- 밀도 높음(강남, 홍대, 명동 등)  → 지오해시 7글자(약 150m) 사용
                                   격자를 잘게 쪼개 한 칸당 제휴처 수를 제한
- 밀도 보통(일반 시내)            → 지오해시 6글자(약 1.2km)
- 밀도 낮음(시골, 외곽)           → 지오해시 5글자(약 5km)
                                   넓은 격자 하나로 듬성한 제휴처를 묶음
```

- 이렇게 하면 강남은 잘게 나뉘어 한 격자의 마커 수가 적정하게 유지되고, 시골은 큰 격자 하나로 묶여 빈 격자 조회가 없어짐

# Baton에서 적용할만한 방법

### 1. 룰 필터를 LLM 앞단에 둔다
- 이모티콘, 잡담, 인사 -> 업무 신호 없음 -> LLM에 넘기지 않는다.
	- 특히 Slack

### 2. 클레임 자체가 토큰 압축
- 문서를 통째로 비교하지 않고 -> 주어/속성/값... 6개의 필드로 압축해서 비교한다 -> 이게 모순 탐지에서의 토큰 절감

### 3. 모순 판정의 룰 선처리 -> 모든 문서의 클레임을 보는 경우(N^2) 없에기
- 범위/시점/단일값 예외를 미리 룰로 거른다
- 해당 룰에 의해 추려진 애들끼만 모순을 판정한다.

### 4. 속성 사전 누적
- 처음에는 새 속성마다 LLM으로 단일/복수 판정을 하지만, 한 번 판정한 건 사전에 쌓아서 재사용한다.
	- 사용이 누적되면 LLM 호출이 줄어든다.

### 5. 임베딩으로 후보를 추리기만 한다!
- 벡터 검색 결과를 전부 LLM 태우지 말고, "어느 위키를 볼 것인가?" 후보군을 추리는데 쓴다!
- 검색 경로랑 생성 경로를 분리한다.

# Baton에서의 부하는 어느정도 될까(굉장히 추정)

- 대규모 시스템 설계에서 하는 방법으로 한번 클로드로 돌려봤습니다!

```
회사 규모:        직원 100명 (중소기업)
직원당 일일 흔적:  슬랙 메시지 50개 + 문서/커밋 5개 ≈ 55건
                 → 회사 전체 일일 5,500건
업무 신호 비율:    슬랙 메시지의 약 20%만 업무 사실 포함
                 (잡담·인사·이모티콘이 80%)
휴직자:           동시 1~2명 (100명 기준 현실적)
```
라고 잡아보자.

#### 1. 인제스트(위키 생성)
```
일일 원천 데이터:     5,500건
초당 평균:         5,500 ÷ 86,400초 ≈ 0.06건/초
```
- QPS가 1도 안됨 -> 부하 문제 없음
- <mark class="hltr-yellow">만약에 룰 필터를 걸어서 80%가 버려지면 하루에 1100건 정도가 LLM을 탐.</mark>

#### 2. 린트(모순 탐지)
#### 3-1. 비교 횟수: N² 회피의 효과를 숫자로

```
일일 신규 소스 클레임:  업무 신호 1,100건 × 평균 2개 사실 ≈ 2,200개/일
1년 누적:             2,200 × 250 영업일 ≈ 55만 개
```

만약 **순진하게 전부 1:1 비교**하면("단순한 접근")

```
55만² ≈ 3,025억 번 비교
→ LLM 호출당 0.5초 잡으면 약 4,800년
→ 명백히 불가능
```

- 설계 문서의 **N² 폭발**입니다 -> 범위 축소 해야함!!

#### 3-2. 범위 축소 후 실제 배치 부하

<mark class="hltr-yellow">밤 배치는 **변경된 위키만** 본다!</mark>

```
하루 변경된 위키 페이지:   신규 클레임 2,200개가 건드리는 위키
                       → 클레임당 평균 1.5페이지 ≈ 3,300페이지 변경
                       (중복 제거하면 실제 수백 페이지 수준)

변경 위키당 후보 클레임:   벡터 검색 top-k로 추림 → 페이지당 약 5~10개 후보

총 비교 쌍:             3,300페이지 × 평균 7쌍 ≈ 23,000쌍
```

여기서 4단계 룰 게이트가 작동합니다.

```
23,000 비교 쌍
  ↓ 룰1~4 (주어·속성·범위·시점·단일값) 로 90% 걸러냄
약 2,300쌍만 애매 → LLM 판정 행
```

#### 3-3. 배치가 시간 안에 끝나는가

```
LLM 호출 약 2,300건
순차 처리:    2,300 × 0.5초 ≈ 19분
병렬 10개:    약 2분
```

**중소기업 기준으로는 병렬 없이 순차로도 밤 배치 시간(예: 새벽 2시~6시, 4시간) 안에 여유롭게 끝납니다.** 부하가 터지는 구간이 아닙니다.

### 그래서 부하는 언제 문제가 되는가?

이 분석의 진짜 결론입니다. 중소기업 100명 기준으로는 **부하가 문제가 안 됩니다.** 그러면 백엔드 어필이 약해 보일 수 있는데, 면접 책 방식대로 **규모를 키워보면** 어디서 깨지는지 드러납니다.

```
직원 100명 → LLM 판정 일일 2,300건       (순차 19분, 문제없음)
직원 1,000명 → 일일 약 23,000건          (순차 3시간, 배치 빠듯)
직원 10,000명 → 일일 약 23만 건          (순차 32시간, 하룻밤 초과 = 붕괴)
```

**1만 명 규모에서 단일 배치가 하룻밤을 넘깁니다.** 바로 여기서 설계 문서의 백엔드 기법들이 필요해집니다.

- **멀티레벨 우선순위 큐**: 휴직자 자료(P0)부터 처리해 4시간 안에 "중요한 것은 끝낸다"
- **병렬 처리 + 페이지 락**: 10~50배 병렬화로 배치 시간 압축
- **증분 검사**: 전체가 아닌 변경분만 → 23만을 실제 수만으로 축소
- **배치 API(50% 할인)**: 밤 배치 전체를 비실시간으로 묶어 비용 절반

즉 발표 논리는 **"중소기업에선 단순 구조로 충분하지만, 규모가 커지면 우선순위 큐·병렬화·증분 검사가 필수가 된다. 우리는 그 확장 지점을 미리 설계했다"**가 됩니다. 면접 책이 "GPS를 매초 보내면 3백만 QPS인데, 모아 보내면 20만으로 준다"고 한 것과 같은 서사 구조입니다.

# 멀티모듈 구조

1. CORE, COMMON 모듈은 삭제
	- Core & Common 모듈을 사용하는 의존성 프로젝트는 커넥션 풀을 할당 받음(Build 효율 저하)
	- 일부 중복을 허용하더라도, 삭제하는게 나은 선택

2. 멀티 모듈 그룹 경계 나누기
>1. BOOT (자주 변함) — 실제 서버. api, batch, admin 
>2. INFRA (가끔 변함) — 외부 연동. 결제, 영상, 사진 
>3. CLOUD (드물게 변함) — 인프라. gateway, config, discovery 
>4. DATA (도메인) — 데이터·도메인. user, meta, chart


3. Services는 어느 BOOT, DATA 양쪽 모두 책임과 역할에 맞게 각각 구현되어야함.
	- BOOT (이벤트 발생) -> DATA (처리)

```markdown
[기존 - Service 한 곳에 몰빵]
UserService 하나에
- HTTP 요청 검증 (BOOT 관심사)
- DB 트랜잭션 처리 (DATA 관심사)
가 다 들어감 → 책임 과다

[변경 - 양쪽에 역할 맞게 분리]
BOOT 쪽 Service  — 요청 받기, 이벤트 발생, 흐름 제어
DATA 쪽 Service  — 실제 데이터 처리, 트랜잭션

BOOT(이벤트 발생) → DATA(처리)
```

4. 좋은 설계는 부서 조직이 할상 일정하지 않듯이, 언제는 MAS에서 모놀리식으로, 모놀리식에서 MSA로 넘어갈 수 있게 하는 구조다.

5. SUMMARY
    - [WHY] 왜 멀티 모듈 프로젝트 구조가 중요할까요?
        - 잘못 구성하면 나중에 변경하기 고통스럽다.
        - 프로젝트 초기에 이루어져야 하는 일련의 설계 과정이다.
        - 개발 생산성에 막대한 영향을 미친다.
        - 서비스 장애와 밀접한 관련이 있다.
    - [WHAT] 무엇을 기준으로 멀티 모듈 프로젝트 구조를 나눠야 할까요?
        - 경계 안에서 의미를 가질 수 있는 그룹을 정의하는(나누는) 것이 가장 중요하다. (Bounded Context)
        - 역할, 책임, 협력 관계가 올바른지 다시 한번 생각한다.
        - BOOT(Server), INFRA, DATA(Domain), SYSTEM(Cloud)
    - [HOW] 어떻게 실전 멀티 모듈 프로젝트를 구현해야 할까요?
        - 프로젝트가 커지고 있다면 다시 경게를 나누고 그 기준으로 소스 저장소를 분리한다.
        - INFRA(외부) 라이브러리에는 DATA 관련 구현을 지향한다. (Anticoruption Layer)
        - 서비스 구현은 각자 역할에 맞게 각각 구현될 수 있다. (공통으로 한쪽에 구현되지 않는다.)
        - 시스템 레벨 구현이 실제 서비스 애플리케이션과 밀접하게 연관되지 않도록 격리하거나 전환(Istio)한다.



> 모듈을 잘 정의해놓으면(각각 build.gradle을 따로), 나중에 이걸 하나로 묶어서 모놀리식 배포 or MSA로 각각 배포를 정하면 된다.

```java
baton  (루트 프로젝트)
├─ settings.gradle          ← 모듈 목록 등록 (가장 중요)
├─ build.gradle             ← 루트 공통 설정
│
├─ baton-query-api          ← 모듈 (인텔리제이가 별도 모듈로 인식)
│  ├─ build.gradle          ← 이 모듈의 빌드 파일
│  └─ src/main/java/
│     └─ com/baton/query/   ← 이 안은 패키지로 나눔
│        ├─ search/
│        └─ answer/
│
├─ baton-claim              ← 또 다른 모듈
│  ├─ build.gradle
│  └─ src/main/java/
│     └─ com/baton/claim/   ← 이 안도 패키지
│        ├─ extract/
│        └─ model/
│
└─ baton-slack              ← 또 다른 모듈
   ├─ build.gradle
   └─ src/main/java/
      └─ com/baton/slack/
```

# 멀티 모듈 구조 설계하기

- <mark class="hltr-yellow">정해진 형식이 있다기 보다는, 경우에 따라서 다양하게 구조화될 수 있다.</mark>
- 일단 시작하고, 조금씩 고쳐나가는 방식으로 성장해보자!

```java
예시                  바통
──────                ──────
project-api      →    baton-api      (질의·답변 API)
project-domain   →    baton-domain   (클레임·위키 엔티티, DB)
project-common   →    baton-common   (예외, 응답코드, 유틸)
project-batch    →    baton-batch    (린트 야간 배치)
project-external →    baton-external (슬랙, 드라이브, 임베딩, OCR)


baton/
├─ settings.gradle
│
├─ baton-api          (실행 가능, 배포 단위)
│  — 질의 받기, 답변 생성, Swagger, Security(JWT)
│  — domain, common, external 의존
│
├─ baton-batch        (실행 가능, 배포 단위)
│  — 린트 야간 배치 스케줄러
│  — domain, common, external 의존
│
├─ baton-domain       (부품)
│  — 클레임/위키 entity, repository
│  — PostgreSQL, JPA, QueryDSL, pgvector
│  — common 의존
│
├─ baton-external     (부품)
│  — 슬랙/드라이브 커넥터, 임베딩 API, OCR
│  — common 의존
│
└─ baton-common       (부품)
   — 예외, 응답코드, 클레임 스키마(6필드), 유틸
```

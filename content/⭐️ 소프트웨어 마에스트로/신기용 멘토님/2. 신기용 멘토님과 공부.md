 # 1. NoSQL에서의 CAP이론

- <mark class="hltr-yellow">일관성, 가용성, 파티션 감내 -> <u>모두 만족하는 분산 시스템을 설계하는 것은 불가능하다는 정리</u></mark>

![[Pasted image 20251120175308.png]]
![[Pasted image 20251120175329.png]]

이제 세가지 중에 두가지를 만족하는 경우를 아래와 같이 나눌 수 있다.

- **CP시스템** : 일관성과 파티션 감내를 지원하는 key-value 저장소. 가용성을 희생
- **AP시스템** : 가용성과 파티션 감내를 지원하는 key-value  저장소. 데이터 일관성 희생
- **CA시스템** : 일관성과 가용성을 지원하는 key-value 저장소. 파티션 감내는 희생. 그러나 통상 네트워크 장애는 피할 수 없는 일로 여겨지므로, 분산 시스템은 반드시 파티션 문제를 감내할 수 있도록 설계해야한다. 따라서 <font color="#f79646">CA시스템은 존재하지 않는다. => 장애가 생겼을 때, 동작이 멈추는 시스템을 만드는 것은 말이 안됨.</font>


#### 이상적 상태
- 네트워크가 파티션되는 상황이 절대로 일어나지 않는다.
- n1에 기록된 데이터는 자동적으로 n2와 n3에 복제된다. 데이터 일관성과 가용성도 만족된다.
![[Pasted image 20251120175616.png]]

#### 실세계의 분산 시스템
- 파티션 문제를 피할 수 없다(네트워크 장애를 피할 수는 없음).
- 우리는 그렇다면 일관성과 가용성 사이에서 하나를 선택해야한다.
![[Pasted image 20251120175652.png]]
- 그림의 경우, n3에 장애가 발생해서 n1과 n2에 데이터가 전달되지 않음.
	- n1과 n2는 n3에서 전달하지 않은 데이터 즉, 오래된 사본을 가지고 있음.

1. 일관성 선택(CP시스템) 
	- 세 서버 사이에 생길 수 있는 데이터 불일치 문제를 피하기 위해서 n1과 n2에 대해서 쓰기 연산을 중단한다면.. -> 응답을 받을 수가 없기 때문에 <font color="#f79646">가용성이 깨짐</font> 
	- 은행권 시스템은 데이터 일관성을 양보하지 않음.
		- ex) 온라인 뱅킹 시스템에 계좌 최신 정보를 출력하지 못하면 큰 문제임..
		- 네트워크 파티션 때문에 일관성이 깨질 수 있는 상황이 발생하면 -> 이런 시스템은 상황이 해결될 때까지는 오류를 반환해야함.

2. 가용성 선택(AP시스템)
	- 설사 낡은 데이터를 반환할지라도, 계속 읽기 연산을 허용해야함.
	- 아울러 n1과 n2는 계속해서 쓰기 연산을 허용할 것.
		- P문제가 해결된 후에 새 데이터를 n3에 전송함.

<mark class="hltr-yellow">=> 즉 요구사항에 맞게 CAP 정리를 적용해야한다.</mark>

#### 그렇다면 NoSQL에서는 CAP가 어떻게 적용될까?

- 우선 NoSQL이 등장한 이유 -> RDBMS는 ACID라는 강력한 일관성을 보장한다.(CA를 기본으로 잡고감, 그렇지만 P가 발생하면 C, A를 지키기가 어려움)
	- 데이터가 커지면 -> 서버를 분산해야하는데.. -> RDBMS는 이런 분산이 어려움
	- ACID를 지키려면, 노드들이 통신을 계속 연결해야함 -> 비용이 큼(느림)
		- 한 노드가 멈추면 전부 멈춰야함.

- NoSQL은 다름 -> ACID라는 강한 일관성을 포기 -> 대신 분산이 편해지고, 한대가 죽어도 작동하도록
	- <mark class="hltr-yellow">기본적으로 AP를 기본으로 잡고 간다!</mark>
	- ACID대신에, BASE
		- BA : Basically Avaiable(기본적으로 가용)
		- S : Soft state(상태 변화가 가능)
		- E : Eventually consistent(결과적으로는 일관)

| 제품              | 분류  | 특징                                                                                      |
| --------------- | --- | --------------------------------------------------------------------------------------- |
| MongoDB         | CP  | 기본은 CP. 쓰기는 항상 Primary 노드 하나에만 가서 일관성을 지킴. Primary가 죽으면 새 Primary 뽑을 때까지 잠시 멈춤(가용성 희생). |
| HBase           | CP  | Google Bigtable 기반. 강한 일관성 우선. 한 영역(region)을 담당하는 서버가 죽으면 그 영역은 잠시 응답 못 함.              |

| Riak            | AP  | Cassandra와 비슷한 철학. AP 우선.                                                               |


# 2. Raft 알고리즘 (분산 합의 알고리즘)

- 기존의 분산 합의 알고리즘인 Paxos가 너무 구현이 어려움 -> 구현이 쉬운 버전이 RAFT다.
- **사실 현대 분산 시스템의 표준**

- 시스템에서 합의를 이루어 노드의 리더를 선출해내는 알고리즘
	- 합의 -> 노드들이 투표를 해서 리더를 선출
- <mark class="hltr-yellow">분산 환경에서는 상태(데이터)를 저장해야함</mark> -> 이때 사용함
	- 살아있는 노드들이 같은 값에 동의하게 하는 알고리즘(리더의 값을 자신들도 따르게 하는)

#### 알고리즘

#### Raft알고리즘 : 노드의 상태
1. Leader : 클러스터에 딱 하나. 모든 클라이언트 요청을 받고, 다른 노드에 데이터를 퍼트림.
2. Follower : 리더 말을 듣고 따라간다. 리더가 보내는 데이터를 받아 저장만 함.
3. Candidate : Leader가 죽었다고 판단 -> "내가 새 리더가 되겠다" (임시상태)

#### 합의 문제를 아래 세개로 쪼갠다.

#### 1. **Leader Election**
- 평소에는 리더가 팔로워들에게 하트비트를 보냄(나 살아있어..)
- 하트비트가 안옴 -> 팔로워는 죽었다고 판단 -> Follwer들이 Candidate가 됨
	- - Candidate가 자기 자신에게 한 표를 던집니다.
	- 다른 모든 노드에게 "나한테 투표해달라"고 요청합니다.
	- 다른 노드들은 **이번 선거에서 아직 투표 안 했으면** 표를 줍니다.
	- <mark class="hltr-yellow">**과반수(majority)** 표를 받으면 새 리더가 됩니다.</mark>
	- 새 리더는 즉시 하트비트를 보내기 시작합니다.
##### 분할 투표 문제 — 랜덤 타임아웃으로 해결!
만약 여러 팔로워가 거의 동시에 "리더 죽었다"고 판단하면, 여러 명이 동시에 Candidate가 됩니다. 그러면 표가 갈려서 아무도 과반수를 못 받고 선거가 실패합니다.

Raft의 해결책이 단순하면서 우아합니다. **타임아웃 시간을 랜덤으로** 둡니다(150~300ms 사이). 그러면 누군가 먼저 시간이 끝나 Candidate가 되고, 다른 노드가 시간 끝나기 전에 표를 모아 리더가 됩니다. 동시에 시작될 확률이 확 줄어듭니다. 만약 그래도 동시에 시작돼서 실패하면 다음 라운드에서 다시 랜덤 타임아웃으로 재시도합니다.
> **용어: Term (임기)** Raft에서 시간을 나누는 단위입니다. 한 번의 선거와 그 결과로 뽑힌 리더가 다스리는 기간이 하나의 Term입니다. Term은 1, 2, 3...으로 증가하기만 합니다. 누군가 Term 5의 메시지를 보냈는데 다른 노드가 Term 7이라면, **더 높은 Term이 이깁니다.** 이게 Raft가 옛 리더의 메시지를 무시하는 핵심 장치입니다.

- <mark class="hltr-yellow">이때 리더선출과 하트비트를 위해서는 RPC통신을 함 -> 굉장히 빠름</mark>

#### 2. **Log Replication**
리더가 정해지면, 모든 클라이언트 요청이 리더로 갑니다. 리더는 이렇게 처리합니다.

1. 클라이언트가 "X = 5로 저장해줘"라고 요청
2. 리더가 자기 **로그(log)**에 이 명령을 추가합니다. 아직 진짜 저장은 안 한 상태.
3. 리더가 모든 팔로워에게 "이 명령을 너희도 로그에 추가해라"라고 보냅니다.
4. <mark class="hltr-yellow">**과반수 팔로워가 "추가했다"고 응답**</mark>하면, 리더는 그 명령을 **commit**(확정)합니다.
5. 리더가 자기 상태 머신에 적용("X = 5"를 실제로 저장)하고 클라이언트에 응답.
6. 다음 하트비트에서 팔로워들에게 "이 명령 commit 됐다"고 알려줍니다. 팔로워들도 적용합니다.

- 팔로워가 리더와 로그가 안 맞으면, **리더 로그가 정답**이고 팔로워가 자기 로그를 덮어씀.

#### 3. **Safety**
만약 옛날 정보를 가진 노드가 리더가 되면, commit된 데이터가 사라질 수 있습니다. Raft는 이걸 막는 규칙을 둡니다.

**선거 제한 (Election Restriction)**
- 투표 요청을 받은 노드는, <mark class="hltr-yellow">**요청자가 자기보다 로그가 최신이거나 같지 않으면 투표하지 않습니다.**</mark> 즉 옛날 정보를 가진 Candidate는 표를 못 얻습니다.
- 이 규칙 덕분에 새 리더는 항상 **commit된 모든 로그를 갖고 있는 노드 중에서** 뽑힙니다. 그러면 commit된 데이터가 사라질 수 없습니다.

- **term :** 0부터 시작하는 term은 현재 리더의 임기를 보관하는 정보입니다. 이 정보를 기반으로 해당 로그가 최신 상태인지 구버전의 상태인지 확인할 수 있습니다.   
    **예를 들어서 리더 서버가 현재 3이고 이 상태를 공유하는데 만약 2인 서버가 있다면 이 서버는 아직 로그가 전파되지 않았다는 의미이므로 구버전이라는 의미입니다.** 

- **index :** 이 index는 kafka로 치면 <mark class="hltr-yellow">offset과 비슷한 개념</mark>입니다. 현재 리더가 처리하고있는 명령의 순서이죠. 이 index가 작다는 이야기는 구버전이라는 의미입니다. 왜냐하면 현재 자신이 보유한 index가 높으면 그만큼 최신이라는 의미이기 때문이죠.

- 자신보다 term이 같거나 높은 후보자에게 투표를 던진다. 
- 만약 term이 같다면 index가 높은 후보자에게 투표한다. 
- term이 낮거나 term과 index모두 자신보다 낮은 후보자에게는 투표하지 않는다.

### CAP 관점에서
CAP 이론과 연결하면, <mark class="hltr-yellow">**Raft는 CP를 선택한 알고리즘**</mark>

- 분할이 일어나서 과반수가 안 되는 쪽은 **응답을 거부**(가용성 희생).
- 대신 commit된 데이터는 **절대 잃지 않음**(일관성 보장).

그래서 Raft를 쓰는 시스템들(etcd, Consul, CockroachDB)이 다 CP 진영임. 은행 잔액, 쿠버네티스 클러스터 상태, 분산 트랜잭션처럼 **틀린 데이터를 주느니 잠시 멈추는 게 나은** 영역에 적합함.


# 3. JPA 낙관적 락과 펜싱토큰 관계

#### 1. 동시성 제어의 두 가지 큰 흐름

여러 사용자가 동시에 같은 데이터를 건드릴 때 일어나는 문제를 막는 방법은 크게 두 가지!

1. **비관적 락 (Pessimistic Lock)** — "충돌이 자주 일어날 것이다"라고 비관적으로 본다는 뜻입니다. 데이터를 건드리기 전에 **미리 락을 걸어** 다른 사람이 못 건드리게 막습니다. 안전하지만 락을 기다리는 시간 때문에 <u>성능이 떨어집니다.</u>

2. **낙관적 락 (Optimistic Lock)** — "충돌이 거의 안 일어날 것이다"라고 낙관적으로 본다는 뜻입니다. 일단 락 없이 작업하고, **저장하는 순간에만** 다른 사람이 먼저 바꿨는지 확인합니다. 빠르지만 충돌이 나면 작업을 다시 해야 합니다.

#### 2. JPA의 낙관적 락

- 버전 번호(or 타임스탬프)로 구현
```java
@Entity
public class Account {
    @Id
    private Long id;
    private Long balance;
    
    @Version
    private Long version;  // 낙관적 락용 버전 번호
}
```
- 업데이트가 발생하면 version++
- <mark class="hltr-yellow">다른 요청이 현재의 version보다 낮은 버전으로 요청을 보내면 -> `OptimisticLockException` 던짐</mark>

#### 문제
1. 같은 DB 안에서만 동작 -> 분산 시스템에서는 못 씀.
2. 작업이 빠르게 끝내야만 함! -> "읽고 -> 처리하고 -> 저장"이 짧은 트랜잭션에서 끝나야함 -> 읽은 다음에 한참 다른 일을 하다가 저장하면 -> 충돌할 가능성이 높아짐
3. <mark class="hltr-yellow">분산 환경에서 리더가 바뀌는 상황을 못 다룸 -> 펜싱토큰!!!</mark>

### 3. 펜싱 토큰 — 어떻게 동작하나

펜싱 토큰(Fencing Token)은 **분산 락**의 안전성 문제를 푸는 메커니즘입니다. JPA 낙관적 락과는 완전히 다른 층위에서 동작합니다.

> **용어: 분산 락 (Distributed Lock)** 여러 서버가 같은 자원을 동시에 건드리지 못하게, **DB 밖의 별도 시스템**(Redis, ZooKeeper 등)에 락을 두는 것입니다. 예를 들어 5개 서버가 같은 파일을 동시에 못 쓰게 하려고 Redis에 "이 파일은 지금 서버 A가 쓰고 있다"는 락을 둡니다.

```
시점 1: 클라이언트 A가 Redis에서 락 획득
시점 2: A가 파일 쓰기 작업 시작
시점 3: A가 GC(가비지 컬렉션) 때문에 갑자기 멈춤. 15초 정지.
        그 사이 락의 TTL이 만료됨.
시점 4: 클라이언트 B가 락을 새로 획득.
시점 5: B가 파일 쓰기 시작.
시점 6: A가 GC에서 깨어남. 자기가 락 가졌다고 생각하고 계속 쓰기 시도.
시점 7: A와 B가 동시에 파일을 쓴다. 충돌. 데이터 깨짐.
```
- **즉 A가 자신은 락이 없다는 것을 모름..!(GC로 멈췄으니까)**

펜싱 토큰은 이걸 이렇게 해결합니다. <mark class="hltr-yellow">**락을 줄 때마다 단조 증가하는 숫자를 같이 준다**</mark>

```
A가 락 획득 → 토큰 33을 받음
A가 파일 쓰기 시도 → "토큰 33으로 쓰겠다"고 알림
(이 사이 락 만료, B가 락 획득 → 토큰 34를 받음)
B가 파일 쓰기 → "토큰 34로 쓰겠다"고 알림. 저장소는 마지막 토큰을 34로 기록.
A가 GC에서 깨어나 쓰기 시도 → "토큰 33으로 쓰겠다"
저장소: "내가 본 마지막 토큰은 34인데, 33은 이전 거네. 거부."
```
- <mark class="hltr-yellow">저장 대상(DB, 파일 시스템)이 토큰을 검증</mark>
#### 토큰의 조건
1. **단조 증가 (monotonically increasing)** — 락을 새로 얻을 때마다 숫자가 항상 커집니다. 줄어들거나 같아지면 안 됩니다.
2. **저장소가 검증** — 자원을 받는 쪽이 "마지막에 본 토큰보다 작은 토큰은 거부"하는 로직을 가져야 합니다.

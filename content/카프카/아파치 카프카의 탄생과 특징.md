![[Pasted image 20260701114319.png|501]]
- 무수히 많은 데이터가 생성되고, 이에 알맞은 애플리케이션에 연결을 해야함!
- 처음에는 연동이 많지 않았지만 -> 점점 거대해지면서 데이터 전송 라인이 너무 복잡해졌다..


![[Pasted image 20260701114404.png|481]]
- 이를 관리하게 위해서 링크드인의 데이터팀은 새로운 시스템을 만들었고
- 이게 곧 아파치 카프카(Apache Kafka)이다.
- <mark class="hltr-yellow">각각의 애플리케이션을 연결하는 것이 아니라, 한 곳에 모라 처리할 수 있도록 중앙집중화했다.</mark>

![[Pasted image 20260701114508.png|467]]
- 기존에는 한쪽에 문제가 생기면 연동되어 있는 다른 한쪽에도 영향이 끼쳤음.
- 그러나 카프카는 의존도 문제도 해결할 수 있다.
- 일종의 FIFO 방식의 큐 자료구조와 유사하다.
	- 데이터는 보내는 쪽이 '프로듀서', 가져가는 쪽이 '컨슈머'다.

# 카프카의 특징

## 1. 높은 처리량

![[Pasted image 20260701115122.png|425]]

- 카프카는 (프로듀서 -> 브로커), (브로커 -> 컨슈머) 모두 <mark class="hltr-yellow">데이터를 묶어서 전송한다.</mark>
	- 많은 양의 데이터를 송수신할 때 맺어지는 네트워크 비용을 아낄 수 있다.
	- 같은 시간에 더 많은 양의 데이터를 보낼 수 있음(통신횟수가 줄어드므로)
- '파티션' 단위를 통해서, 동일 목적의 데이터를 여러 파티션에 분배하고 병렬처리할 수 있다.
	- 파티션만큼 컨슈머도 늘릴 수 있다(처리량 증가)

## 2. 확장성

![[Pasted image 20260701115204.png|488]]

- 데이터는 가변적으로 들어오기 때문에, 안정적인 <mark class="hltr-yellow">scale-out, scale-in</mark>이 되어야 한다.
- 카프카 클러스터의 브로커의 개수를 조절하면서 가능하다!

## 3. 영속성

![[Pasted image 20260701115319.png|529]]

- 특이하게 메모리가 아니라 <mark class="hltr-yellow">파일 시스템에 저장한다.</mark>
	- 보편적으로 파일 시스템은 느리지만, 카프카는 운영체제 레벨에서 파일 시스템을 최대한 활용.
	- 파일 I/O 성능 향상을 위해 페이지 캐시(Page Cache) 영역을 메모리에 따로 생성해서 사용.
		- 한번 읽은 파일 내용을 메모리에 저장시켰다가 다시 사용하는 방식 -> 처리량이 높다.
- 디스크 기반이기 때문에, 프로세스를 재시작해도 다시 처리가능(휘발X)

## 4. 고가용성

![[Pasted image 20260701115505.png|551]]

- 카프카 클러스터 -> 3개 이상의 서버들로 운영.
- 무중단 데이터 처리 가능
	- replication


### 정리
> 1. 카프카는 복잡한 데이터 흐름을 단순화해준다.(중앙집중)
> 2. 카프카는 높은 처리량, 확장성, 영속성, 고가용성을 지닌다.

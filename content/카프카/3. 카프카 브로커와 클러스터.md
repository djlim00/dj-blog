## 1. 카프카 브로커·클러스터·주키퍼

![[Pasted image 20260702184452.png]]
- 카프카 브로커 : 카프카 클라이언트와 데이터를 주고 받기 위해 사용하는 주체.
	- 데이터를 분산 저장하여, 장애가 발생하더라도 안전하게 사용할 수 있도록 해주는 애플리케이션
- 한개의 서버 == 하나의 카프카 브로커 프로세스
- 카프카 브로커 서버 1대로도 기본 실행 가능
	- <mark class="hltr-yellow">데이터를 분산처리 하기 위해 3대 이사으이 브로커 서버를 1개의 클러스터로 묶어서 운영</mark>


## 2. 여러개의 카프카 클러스가 연결된 주키퍼

![[Pasted image 20260702184623.png]]

- 카프카 클러스터를 실행하기 위해서 주키퍼가 필요함
- 주키퍼의 서로 다른 znode에 클러스터 지정
- root znode에 각 클러스별 znode를 생성 -> 클러스터 실행시 root가 아닌 하위 znode 설정
- 가프카 3.0부터는 주키퍼가 없어도 클러스터 동작 가능

> **주키퍼(ZooKeeper)**
> - 분산 시스템에서의 서버 상태 관리를 해주는 Apache 오픈소스 소프트웨어

## 3. 브로커의 역할 - 컨트롤러, 데이터 삭제
#### 컨트롤러
- 다수의 브로커 중 한 대가 컨트롤러(리더)
	- 다른 브로커 상태 체크
	- 브로커가 빠지면, 해당 브로커가 존재하는 리더 파티션 재분배
	- 브로커의 상태가 비정상이면 빠르게 클러스터를 빼내는 것이 중요
	- 컨트롤러가 이상이 생기면, 다른 브로커가 컨트롤러

#### 데이터 삭제
- 카프카는 다른 메시징 플랫폼과 다르게 컨슈머가 데이터를 가져가더라도, 토픽의 데이터는 삭제되지 않음.
- 컨슈머나 프로듀서가 데이터 삭제 요청 불가능
- 오직 브로커만 데이터를 삭제
- 데이터 삭제 -> 파일 단위(로그 세그먼트)
	- 로그 세그먼트는 다수의 데이터가 있기 때문에 일반적인 DB 처럼 특정 데이터 선별 삭제 불가능

## 4. 브로커의 역할 - 컨슈머 오프셋 저장, 코디네이터
#### 컨슈머 오프셋 저장
- 컨슈머 그룹은 토픽이 특정 파티션으로부터 데이터를 가져가서 처리
	- 이 파티션의 어느 레코드까지 가져갔는지 확인 -> 오프셋을 커밋해 둠.
	-  _ cmsumer_offsets 토픽에 저장

#### 그룹 코디네이터
- 코디네이터는 컨슈머 그룹의 상태 체크
- 파티션을 컨슈머와 매칭되도록 분배
- 컨슈머가 컨슈머 그룹에서 빠지면, 매칭되지 않은 파티션을 정상 동작하는 컨슈머로 할당 -> 끊임없이 데이터 처리
	- 이 과정을 리밸런스(rebalance)라고 함

## 5. 브로커의 역할 - 데이터의 저장
- config/server.properties의 log.dir 옵션에 지정한 디렉토리에 저장
	- 토픽이름+파티션 번호로 하뤼 디렉토리 생성

```text
/kafka-data/
└── ingest-events-0/          ← 토픽명-파티션번호, 폴더 하나
    ├── 00000000000000000000.log
    ├── 00000000000000000000.index
    ├── 00000000000000524288.log
    ├── 00000000000000524288.index
    └── ...
```

- 세그먼트는 `.log` 확장자로 저장이 된다.
	- 세그먼트 파일 내부 내용은 `레코드`

## 6. 세그먼트

- **세그먼트** — 로그(파티션)를 파일 단위로 잘라놓은 조각. 로그는 세그먼트 여러 개로 구성(여러개의 `.log`파일)

#### 액티브 세그먼트
![[Pasted image 20260702195953.png]]

- 파일 읽기 쓰기가 일어나고 있는 세그먼트
	- 삭제 혹은 압축의 대상이 되지 않음

#### 세그먼트의 삭제

- retenrion.ms : 세그먼트 보유 최대 기간, 기본값은 7일
- <mark class="hltr-yellow">카프카에서는 데이터를 세그먼트 단위로 삭제함 -> 레코드 단위로 개별 삭제는 불가능</mark>
	- 레코드의 값들은 수정이 불가능함
	- 데이터를 적재할 때(프로듀서) 또는 데이터를 사용할 때(컨슈머) 데이터를 검증해야한다

#### 압축, 테일/헤드 영역

![[Pasted image 20260702200250.png]]

- 토픽 압축 정책은 오래된 데이터를 삭제한다.
- 일부 레코드만 삭제된다.
	- 액티브 세그먼트는 당연히 대상이 아님!

![[Pasted image 20260702200315.png]]
- 테일 영역 : 압축이 된 레코드 - 클린(clean)
- 헤드 영역 : 압축 전 레코드 - 더티(dirty), 중복이 있음.

## 7. 브로커의 역할 - 복제

![[Pasted image 20260702200444.png]]

- 파티션은 각각 읽기/쓰기 작업을 하는 브로커가 대표 파티션을 가짐
	- 나머지는 해당 파티션의 복사본(replica)를 저장
- 복제 개수 만큼 저장 용량이늘어남 -> <mark class="hltr-yellow">하지만 데이터를 안전하게 사용할 수 있게 해줌</mark>
	- 해킹, 디스크 오류, 네트워크 연결 장애에 대비할 수 있음

#### 브로커에 장애가 발생하면?

![[Pasted image 20260702200626.png]]

- 해당 브로커의 리더 파티션을 사용 못함 -> 팔로워 파티션 중에 하나가 리더 파티션의 지위를 넘겨 받음.

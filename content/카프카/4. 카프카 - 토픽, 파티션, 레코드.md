## 토픽과 파티션

![[Pasted image 20260702200946.png]]

- **토픽** : 카프카에서 데이터를 구분하기 위한 단위
- <mark class="hltr-yellow">토픽 1개에는 1개 이상의 파티션</mark>
	- 파티션에 들어간 데이터가 레코드
	- 이런 레코드들을 일정 단위로 자른게 세그먼트
- FIFO(하지만 pop하지 않고 저장한다)

#### 토픽에서의 파티션 분배

![[Pasted image 20260702201005.png]]

- <mark class="hltr-yellow">파티션의 개수에 따라서 브로커에 Round-Robin으로 분배한다(현재는 Sticky 알고리즘으로 바뀌었다)</mark>
	- 카프카 클라이언트는 리더 파티션이 있는 브로커와 통신

![[Pasted image 20260702201131.png]]
- 특정 브로커에 파티션이 몰리면 kafka-reassign-partitions.sh로 재분배 가능

#### 파티션 개수를 줄일 수는 없다!

- 따라서 파티션 개수를 늘리는 작업은 신중해야한다
	- 오직 토픽 삭제 밖에 답이 없음
- <mark class="hltr-yellow">줄이게 되면, 여러 브로커에서 데이터를 취합하고 다시 정렬하는 복잡한 과정이 필요하기 때문에</mark>

## 2. 레코드

![[Pasted image 20260702201348.png]]
- 파티션 내부의 세그먼트를 구성하는 실제 데이터 값들

#### 1. timestamp
- 스트림 프로세싱에 활용

#### 2. offest
- 프로듀서가 생성한 레코드에는 없음 -> 브로커에 적재되면서 생김
- 0에서 1씩 증가
- <mark class="hltr-yellow">offset값을 기준으로, 완료된 데이터와 앞으로 처리해야할 데이터를 구분</mark>
	- 파티션 별로 고유한 오프셋을 가지기 때문에, 컨슈머에서 중복 처리를 방지한다.

#### 3. header
- key/value 데이터 추가
- 데이터 프로세싱에 참고할만한 정보를 담아서 사용

#### 4. key
- <mark class="hltr-yellow">처리하는 메시지 값을 분류하기 위한 용도</mark>
	- == 파티셔닝
- 지정하지 않으면 null값

ex) 사용자ID, 주문ID, 문서ID

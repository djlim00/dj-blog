# 2. 람다 아키텍처

![[Pasted image 20260701120546.png|519]]

1. **배치 레이어** : 배치 데이터를 모아서 특정 시간, 타이밍 마다 일괄 처리
2. **서빙 레이어** : 가공된 데이터를 데이터 사용자, 서비스 애플케이션이 사용할 수 있도록 데이터가 저장되는 공간
3. **스피드 레이어** : 낮은 지연으로 분석이 필요한 경우

## 람다 아키텍처의 한계

- 데이터 처리 방식을 명확히 나눔에 따라 발생하는 단점
1. **데이터 분석/처리 로직이 2벌로 각각 레이어에 생김**
2. **배치 데이터와 실시간 데이터의 융합이 유연하지 못함**

# 3. 카파 아키텍처

![[Pasted image 20260701121230.png|536]]


- 람다 아키텍처의 한계를 극복하고자 Jay Kreps(카프카를 최초로 고안 한 개발자, 현 Confluent CEO)가 제안한 아키텍처.
- 람다 아키텍처의 단점인 '로직의 파편화', '디버깅', '배포', '운영 분리' 이슈를 제거
- <mark class="hltr-yellow">스피드 레이어에서 데이터를 모두 처리한다(배치 레이어 X)</mark>

## 카파 아키텍처의 활용

![[Pasted image 20260701121417.png]]

- 로그(log)는 **배치 데이터를 스트림 데이터로** 표현하기에 적합함.
	- 일반적인 데이터 플랫폼에서는 배치 데이터 -> 스냅샷(통으로 저장)
	- <mark class="hltr-yellow">그러나 change log를 사용하면, 시간 순서대로 기록하여서, 각 시점을 따로 크게크게 저장하기 않아도 배치 데이터를 표현할 수 있음!(change log를 시간대로 자르면 그게 배치 데이터)</mark>

# 4. 스트리밍 데이터 레이크

![[Pasted image 20260701121758.png|437]]
- 2020년 Jay Kreps는 서빙 레이어까지 제거한 스트리밍 데이터 레이크를 제안.
	- 카파 아키텍처에서는 데이터를 사용하는 고객을 위해 스트림 데이터를 서빙 데이터에 저장
- 근데 만약 카프카(스피드 레이어)에서 오랜 기간 저장할 수 있다면?
	- 서빙 레이어는 필요없어짐
	- 이중으로 관리하는 리소스 줄어듦

![[Pasted image 20260701122054.png|349]]

- **그러나 아직까지는 카프카를 스트리밍 데이터 레이크로 사용하기에는 한계가 있음**
	- 자주 사용하는 데이터만 브로커에서 사용하고, 나머지는 오브젝트 스토리지 같은 곳에 저장하던가 하는 구분 작업이 필요하다.

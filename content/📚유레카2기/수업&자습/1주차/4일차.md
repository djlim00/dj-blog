# 3. 인터페이스(Interface)

#### 다형성 구현의 주된 기술! implements

- **A implements B** : B클래스에서 A클래스를 구현했다는 뜻이다.

> 🙉 인터페이스와 인터페이스 사이에는 **extends**를 사용할 수 있다..!

- 인터페이스는 완전한 실행코드를 가진 디폴트 메서드를 가질 수 있음.
	- 일반메서드는 애초에 선언이 안되는거고, 추상메서드는 완전한 실행코드를 가질 수가 없음!

### 정적 메소드를 인터페이스에서?

- 구현 객체가 없어도, 인터페이스만으로 호출 가능 -> 자동으로 public

### private 메소드를 인터페이스에서?

- 오직 디폴트 메소드 안에서만 호출가능하다.
	- 생각해보면 당연한게, 인터페이스 자체가 상속처럼 다른 애들한테 넘겨주는 애인데, 해당 클래스(인터페이스)안에서만 사용 가능한 private 메소드면, 어디다가 쓸까?
	- 오직 실행코드가 존재하는 인터페이스의 디폴트 메소드안에서만 쓸 수 있음!

### 인터페이스도 상속이 된다고?

- 인터페이스도 다른 인터페이스를 상속 가능 -> <u>인터페이스끼리는 다중 상속 가능</u>

```java 
public interface HOHO extends HAHA, HIHI{}

// 여기서 HAHA, HIHI는 모두 인터페이스다(그러니까 extends하고 여러개 나올 수 있는 것이지)
```


### 자동 타입 변환 - 할아버지로 캐스팅 가능
- 부모를 상속받은 자식은
- 부모가 인터페이스를 구현하고 있다면 -> 자식도 할아버지인 인터페이스로 캐스팅(형 변환)이 가능하다!
### 강제 타입 변환 - 인터페이스로 선언해서 자식으로 넣기
- 그냥 뭐 더 상위로 선언하고, 자식으로 넣을 때 강제로 변환하는 것!


- 한마디로 디폴트 메서드 - Override 하든말든 상관없음
- static - 인터페이스 안에서만 사용함
- <font color="#ffff00">추상 메서드 - 무조건 구현 해야함!!</font>


# <font color="#ff0000">인터페이스는 상속(extends)가 너무 연결성이 크고(단일 상속) abstract의 강제성을 극복하기 위해서 만들었다!</font>

# 4. 예외

- 예외 : 잘못된 사용 또는 코딩으로 인한 오류
- 에러와 달리 예외 처리를 통해 계속 실행상태 유지 가능
	- 일반 예외(exception) : 컴파일러가 예외처리 코드 여부를 검사하는 예외
	- 실행 예외(run-time exception) : 컴파일러가 예외 처리 코드 여부를 검사하지 않은 예외

#### try - catch - finally 블록으로 구성

- try : 안에 있는 코드의 예외를 처리하겠다는 범위
- catch : 어떤 종류의 예외를 어떻게 처리할지를 결정함
- finally : 최종적으로 수행되어야하는 작업을 지정

- catch문은 자세한 클래스(하위 클래스) -> 큰 클래스(상위 클래스) 순서로 적어야한다.

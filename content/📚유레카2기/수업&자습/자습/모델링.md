공학 -> 설계 -> 모델링 -> 추상화

앞서 말한 객체지향 프로그래밍에서도 객체는 상태와 행위를 가진 것의 추상체이다. 
즉 객체는 모델링의 결과인 것 이다.
그렇다면, 이러한 모델링은 어떤 과정을 통해 되는 것일까?

개념 모델링(초기) -> 논리 모델링(거의 완성) -> 물리 모델링(실질적 구현)

##### 개념 모델(비즈니스 모델)

- 현재 우리가 직면한 문제(기능)에서 필요한(중요한) 데이터를 간단하게 표현하는 것
	- ERD, UML 다이어그램(아래 링크 참조, class, use-case, sequence...etc)
	- https://seulhee030.tistory.com/56
- <font color="#ffff00">주요 엔티티와 그 엔티티의 주요 속성을 표현 -> 핵심 엔티티와의 관계 도출</font>
- 개념 모델에서 표현된 구조들은 유지되어야 함
![[Pasted image 20250122133809.png]]
- <font color="#ffff00">개념 모델에서는 회원울 보면 그냥 "신상상세정보"라고만 적고 넘어감</font>
- 관계도 대충만 모델링

##### 논리모델(Logical Model)

- 개념모델의 상세화
- 중요속성 외에 필요한 속성추가
- 누가, 어떻게 접근하는지 파악
- 완료 후에는 거의 완성된 모델 -> 더 이상의 엔티티/속성 삭제는 없음![[Pasted image 20250122133820.png]]
- <font color="#ffff00">회원의 모든 "신상상세정보"를 명시함</font>
- 관계 명확하게 명시


##### 물리 모델(Physical Model)

- 데이터베이스의 진짜 설계
- 물리적인 스키마를 만든다
- 개체명, 속성명, 데이터 형태, 길이, 영역값 등을 설계
- DBMS의 조건에 맞게 <u>성능 최적화</u>

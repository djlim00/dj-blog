 RAG라는 것은 어떤 llm이름이나 라이브버리라기 보다는 기존의 llm은 llm이 가지고 있는 내용을 토대로만 정보를 제공하지만 이것은 부정확하기 때문에 외부의 DB나 외부의 데이터를 이용해서 일종의 답지를 가지고 답변을 할 수 있도록, 즉 더 정확한 정보를 가진 인공지능을 구현하는 기술인거네. 
 RAG의 과정은

1. 사용자가 질문을 한다.

2. 임베딩 모델을 통해서 사용자의 질문을 벡터화 한다.

3. 이미 RAG를 위한 벡터 DB에는 외부의 데이터를 임베딩 모델을 통해서 청킹을 통해 저장되어 있는 정보들이 있다.

4. 그럼 벡터화된 사용자의 질문과 가장 유사한 데이터를 벡터 DB에서 찾는다.

5. 그럼 벡터 DB의 조회 결과와 사용자의 질문을 Chat GPT와 같은 llm에 모델에 입력을 한다.

6. llm 모델은 위의 두 정보를 조합해서 답변을 생성한다.

7. 답변을 사용자에게 출력한다.

| 보강 포인트                                     | 설명                                                    |
| ------------------------------------------ | ----------------------------------------------------- |
| **RAG는 구조/기법이지, 라이브러리나 모델 이름이 아님**         | RAG는 방법론                                              |
| **RAG = Retrieval + Generation**           | Retrieval(검색) + Generation(생성)을 결합한 이름                |
| **"청킹(chunking)" 과정**                      | 외부 데이터를 벡터 DB에 저장할 때, 문서를 의미 단위로 자르는 과정. 이것의 정확도가 중요. |
| **임베딩 모델 ≠ LLM**                           | 질문 임베딩에 사용하는 모델과, 답변 생성에 사용하는 LLM은 다른 모델.             |
| **GPT는 원래 외부 지식 참조 못함**                    | 오로지 학습된 모델 파라미터에만 의존함 → 그래서 RAG가 필요해짐                 |
| **보통 사용하는 LLM은 ChatGPT 계열 (GPT-3.5, 4 등)** | 또는 open-source LLM (예: LLaMA, Mistral 등)도 사용 가능       |


### 어떤 부분이 고려 사항인가?

1. 어떤 임베딩 모델을 사용할 것인지?(어떤 임베딩 모델 라이브러리를 사용할 것인가?)
2. 청킹을 어떻게 할 것인지?(이건 임베딩 모델에서 정하는 것인가?) 
3. 어떤 벡터 DB를 사용할 것인지?(어떤 부분을 고려해야하는 것인지? 임베딩 모델 라이브러리와의 호환성?) 
4. 어떤 llm을 써야 하는 것인지?(응답속도와 사용자 질문과 벡터 DB에서의 반환하는 정보를 합쳐서 최적의 정답을 할려 줄 수 있는 성능을 고려해야함)


## 1. **어떤 임베딩 모델을 사용할 것인가?**

| 선택 기준   | 설명                                   |
| ------- | ------------------------------------ |
| **정확도** | 문장의 의미를 얼마나 잘 벡터화하는지                 |
| **속도**  | 응답 시간 (실시간 챗봇이라면 중요)                 |
| **비용**  | OpenAI 모델은 과금, HuggingFace는 무료 사용 가능 |
| **호환성** | 사용하는 벡터 DB에 잘 연동되는지                  |
| **언어**  | 한국어, 다국어 지원 여부                       |

### 🔹 대표 임베딩 모델 예시

| `sentence-transformers`       | HuggingFace | 무료, 다양한 언어 지원, 로컬 사용 가능  |
| `bge-base-en`, `E5`, `Kobert` | HuggingFace | Open Source, 한글 특화 모델 있음 |

## 2. **청킹(Chunking)을 어떻게 할 것인가?**

|고려 사항|설명|
|---|---|
|**Chunk 단위 크기**|너무 작으면 정보 부족, 너무 크면 검색 정확도 낮음 (보통 100~500 tokens)|
|**Overlap 여부**|문단 간 맥락 보존 위해 10~50% 겹치게 자르기도 함|
|**자르는 기준**|문단 기준? 문장 기준? 또는 슬라이딩 윈도우 방식?|
|**임베딩과 분리됨**|✨ 중요한 점: **청킹은 임베딩 모델이 자동으로 해주는 게 아님!** 직접 분리해줘야 함|

### 🔹 청킹 도구

- LangChain의 `RecursiveCharacterTextSplitter` (많이 사용됨)
    
- 직접 구현 (예: 줄 수 기준 or 문장 기준으로 나누기)

## 3. **어떤 벡터 DB를 사용할 것인가?**

|고려 기준|설명|
|---|---|
|**성능 (속도, 정확도)**|검색 속도와 정확도 (Faiss < Weaviate < Milvus)|
|**배포/운영 편의성**|클라우드 사용 가능? 로컬 설치 어려운가?|
|**API or Python SDK 제공 여부**|코드 연동 쉬운지|
|**임베딩 모델과의 호환성**|float32 벡터, cosine/IP 유사도 등 지원 여부|

### 🔹 대표 벡터 DB

|이름|특징|난이도|
|---|---|---|
|**Milvus**|고성능, 대규모 서비스에 적합|중상|

|**Pinecone**|서버 없이 사용 가능 (유료)|하|
|**FAISS**|Meta 개발, 빠름, 파일 기반, 설치 쉬움|하|

> 🔰 처음 구현엔 **FAISS or Weaviate** 추천

## 4. **어떤 LLM을 사용할 것인가?**

|고려 사항|설명|
|---|---|
|**성능**|RAG에서는 hallucination이 적은 모델이 유리|
|**속도**|실시간 답변이 중요한가?|
|**비용**|OpenAI GPT-4는 정확하지만 비쌈|
|**context length**|RAG에서는 context length가 길수록 유리 (많은 reference를 넣을 수 있음)|

### 🔹 대표 LLM 비교

|모델|특징|가격/속도|비고|
|---|---|---|---|
|**GPT-4o**|성능 최고, context 길다|💸 느리고 비쌈|OpenAI API|
|**GPT-3.5 Turbo**|빠름, 싸고 성능 무난|👍 가성비|OpenAI API|
|**Claude 3**|긴 context, 비교적 안정적|중간|Anthropic API|
|**Mistral / LLaMA 3**|오픈소스, 로컬 가능|빠름|HuggingFace or Ollama|
|**KoAlpaca / KorGPT**|한글 특화|무료|실험용으로 적합|

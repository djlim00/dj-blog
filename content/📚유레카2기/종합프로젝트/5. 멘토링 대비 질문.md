# **기술 스택 관련 예상 질문**

### Q1. 왜 React를 선택했나요?
**A.** 사용자 맞춤형 UI 흐름이 중요한 서비스이기 때문에, 컴포넌트 기반으로 재사용성과 유지보수성이 뛰어난 React를 선택했습니다. 특히 로그인, 마이페이지, 추천 결과 등 UI가 계속 바뀌는 구조에서 React의 상태 관리와 렌더링 최적화 기능이 큰 도움이 됩니다.

---

### Q2. Axios가 아닌 fetch를 쓰지 않은 이유는 무엇인가요
**A.** Axios는 기본적으로 요청/응답에 대한 인터셉터 처리, 에러 핸들링, JSON 자동 파싱 기능을 제공해 fetch보다 코드가 간결하고 유지보수가 쉬웠습니다. 또한, JWT 토큰 재발급 로직 등 인증 흐름에 필요한 커스터마이징에도 유리했습니다.

---

### Q3. 백엔드에서 Spring Boot 3.x를 쓴 이유는?

**A.** Java 17과의 호환성과 더불어, Jakarta EE로의 전환 이후 최신 보안, 성능 최적화를 포함한 기능들을 사용할 수 있습니다. API 문서 자동화(Springdoc), 인증(JWT + Security), DB 관리(JPA + Liquibase)와도 자연스럽게 연동이 가능해 선택했습니다.

---

### Q4. 인증에 왜 JWT + OAuth2를 같이 사용했나요?
**A.** 소셜 로그인을 위해 OAuth2 기반의 인가 코드를 받아오고, 이후 자체 토큰(JWT)을 발급하는 구조를 사용했습니다. 이를 통해 무상태(stateless) 기반의 인증을 구현했고, 보안성과 확장성 모두 확보했습니다.

---

### Q5. PGVector를 왜 도입했나요?
**A.** OpenAI 임베딩 벡터를 기반으로 한 유사도 검색을 수행하는 데 있어서, 벡터 DB를 별도로 두는 대신 PostgreSQL 내에서 `pgvector` 확장을 활용했습니다. 이를 통해 운영 환경을 단순화하고, 관계형 데이터와의 결합 쿼리도 동시에 처리할 수 있어 효율적입니다.

---

# **구조 및 아키텍처 관련 예상 질문**

### Q6. RAG 기반의 추천 시스템은 어떻게 동작하나요?
**A.** RAG(Retrieval-Augmented Generation)는 사용자의 질문을 임베딩한 후, 벡터 유사도 검색을 통해 관련 요금제 문서를 찾아 LLM(OpenAI)에게 전달하는 방식입니다. 이렇게 하면 단순 생성보다 더 정확하고 컨텍스트 기반의 응답이 가능합니다.

---

### Q7. 왜 FastAPI를 AI 서버로 썼나요?
**A.** 챗봇 서버는 빠른 응답성과 Python 기반의 생태계 활용이 중요했기 때문에, 비동기 HTTP 처리에 강한 FastAPI를 선택했습니다. LangChain, OpenAI 연동, PGVector 조회까지 Python에서 모두 자연스럽게 처리됩니다.

---

### Q8. 전체 서비스 구조는 어떻게 연결되나요?

**A.**
- **Frontend** (React): 사용자 UI
- **Backend** (Spring Boot): 인증/추천 로직 처리, RDS 관리
- **AI Server** (FastAPI): OpenAI API + PGVector 기반의 추천 처리
- **Infra**: EC2 + RDS + Docker로 각각 배포 및 운영


---

### Q9. DB는 어떻게 버전 관리하고 있나요?
**A.** Liquibase를 활용해 DB 변경 사항을 `.xml` 스크립트로 관리하고 있습니다. 이를 통해 팀원이 동일한 DB 스키마를 유지할 수 있고, 배포 시점에서도 DB 일관성이 보장됩니다.

---

# **협업 및 DevOps 관련 예상 질문**

### Q10. 왜 Github Actions를 선택했나요?

**A.** GitHub와 바로 연동되어 있어 브랜치 push, PR 등 이벤트 기반으로 빌드/테스트/배포 워크플로우를 쉽게 구성할 수 있었습니다. 개발 속도가 빠른 프로젝트라 CI/CD 자동화는 필수였습니다.

---

### Q11. Jira는 어떻게 사용하고 있나요?

**A.**

- Task, Epic, Sprint 구분을 명확히 했습니다.
    
- GitHub 브랜치와 연결하여 Smart Commit 및 커밋 추적도 자동화했습니다.
    
- 각 스프린트(예: 1차 개발 스프린트)에 포함된 기능들을 Issue 단위로 추적하고 있습니다.


---

### Q12. 프론트는 어떻게 배포했나요?

**A.** 정적 파일 중심이므로 Vercel을 통해 React 앱을 배포했습니다. GitHub 연동만으로 Preview, Production 배포가 자동화되어 빠른 테스트가 가능했습니다.

# ERD 및 데이터베이스 관련 예상 질문
### **Q1. 왜 Plan을 Mobile/Internet/IPTV로 분리하고, 각각을 상속 구조처럼 만들었나요?**

- **의도**: 정규화 및 확장성 설계에 대한 이해 확인
    
- **답변**:  
    각 요금제는 공통적으로 `plan` 테이블에서 요금제 이름, 가격, 혜택 등 기본 속성을 공유하고,  
    서비스별(모바일/인터넷/IPTV)로 특화된 속성(예: `data`, `velocity`, `channel`)이 달라서 테이블을 분리했습니다.  
    이는 **단일 테이블 설계보다 정규화 수준이 높고, 유연한 확장과 유지보수에 유리**합니다.  
    예를 들어, 향후 '사물인터넷 요금제'가 추가될 경우에도 `plan_id` 기반으로 쉽게 연동할 수 있습니다.
    

---

### **Q2. 사용자와 관심태그(Interest)는 왜 중간 테이블로 설계했나요?**

- **의도**: 다대다 관계 처리에 대한 이해 확인
    
- **답변**:  
    하나의 사용자가 여러 태그에 관심을 가질 수 있고, 하나의 태그도 여러 사용자에게 적용될 수 있는 **다대다(N:M)** 구조입니다.  
    따라서 중간 테이블인 `Interest`를 통해 연결하고 있습니다.  
    이 구조를 통해 향후 유저 취향 기반 추천, 마케팅 타겟팅 기능 확장이 용이해집니다.
    

---

### **Q3. 커뮤니티 혜택(community_benefit)과 가격(community_benefit_price)은 왜 분리했나요?**

- **의도**: 정규화 vs. 비정규화 판단 기준 확인
    
- **답변**:  
    혜택 조건(`community_condition`)과 할인 구조(`headcount`, `youth_discount` 등)은 **속성이 다르고 반복될 수 있는 구조**이기 때문에 분리했습니다.  
    즉, 하나의 혜택에 대해 여러 인원 기준의 가격 정책이 가능하도록 설계한 것입니다.  
    이렇게 분리하면 각 요소를 개별 관리할 수 있고, 로직도 명확하게 나눌 수 있습니다.
    

---

### **Q4. 결합 요금제(plan_community)는 어떤 기준으로 모델링했나요?**

- **의도**: 중간 테이블 생성 목적 확인
    
- **답변**:  
    특정 요금제(plan)가 결합 혜택을 받을 수 있는 community_benefit과 N:M 관계이기 때문에 중간 테이블 `plan_community`로 연결했습니다.  
    결합 가능한 요금제 유형이 다양하고, 하나의 혜택이 여러 요금제에 적용될 수 있는 구조이므로 이 설계가 적절했습니다.
    

---

### **Q5. 왜 사용자 상태(status), 역할(role), 성별(gender)을 ENUM으로 저장했나요?**

- **의도**: ENUM 사용 이유 + 확장성 판단
    
- **답변**:  
    세 값 모두 **미리 정의된 제한된 상태 집합**이므로 ENUM을 사용했습니다.  
    예를 들어, `role`은 `USER`, `ADMIN`, `TMP_USER`로 고정되며,  
    ENUM은 오타 방지와 관리의 일관성 유지에 유리합니다.  
    또한, 코드 상에서 상수 enum으로 바로 매핑해 처리할 수 있어 유지보수성도 높습니다.
    

---

### **Q6. 계약 테이블(Contract)에서 `user_id`, `plan_id`, `community_id`를 다 같이 관리하는 이유는?**

- **의도**: 실제 도메인 반영 구조 판단
    
- **답변**:  
    실제 서비스에서 유저는 단순히 요금제만 계약하는 것이 아니라, 결합 혜택이 적용된 복합 구조로 계약합니다.  
    따라서 `user_id`와 `plan_id`, 그리고 해당 결합 혜택을 나타내는 `community_id`를 함께 저장했습니다.  
    이를 통해 사용자별 결합 계약 내역과 혜택 적용 여부를 하나의 테이블에서 바로 조회할 수 있습니다.

---

### **Q7. 왜 벡터(Vector) 테이블을 별도로 분리했나요?**

- **의도**: AI 추천 구조에 대한 이해 확인
    
- **답변**:  
    임베딩 기반의 추천을 구현하기 위해 요금제 설명(title, description)을 OpenAI 임베딩 벡터로 변환해 저장합니다.  
    `Vector` 테이블은 요금제 ID와 임베딩 벡터(Field)를 매핑하며, `PGVector` 확장을 통해 **벡터 유사도 검색이 가능**하도록 설계했습니다.
    

---

### **Q8. 즐겨찾기(bookmark)와 관심태그(Interest)는 어떻게 다르게 작동하나요?**

- **의도**: 비슷한 구조를 구분할 수 있는지 확인
    
- **답변**:  
    `Interest`는 사용자 취향 기반 마케팅과 추천 알고리즘에서 사용되고,  
    `Bookmark`는 사용자가 직접 관심 있는 요금제를 저장한 정보입니다.  
    즉, Interest는 **간접/패시브 데이터**, Bookmark는 **직접/액티브 행동 데이터**로 성격이 다릅니다.

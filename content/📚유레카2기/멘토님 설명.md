# 도메인 설계

- API 중에서 기계적으로 생성되는 것 뿐만 아니라, 프론트엔드의 요구사항에 의해서 생기는 API를 어떻게 해야하나?
	- 기본적으로 기능 기준으로 프론트엔드 개발자가 같이 해야하고, 이것도 결국 백엔드의 기준에서 API 설계가 이루어져야하고,,
	- 도메인에 따라서 달라지는 데이터 조작 -> 어디서 데이터를 가공해서 보여줘야하나.
		- ex) 조직 구성도 -> 도메인이 트리임 -> 이건 백엔드가 할 일이다.
		- ex) 도메인은 트리가 아닌데, UI 적으로 트리가 필요하다면 그건 프론트의 일이다.
	- 파라미터 설계
		- 프론트의 요구사항이 백엔드의 설계에 들어가면 안된다.(화면 기준이 아니라)
		- 리소스는 사용자에게 이게 동일한 리소스라는 것을 보장해야한다.
			- Post를 등록하는 것을 비동기로 뺀다고 하면, 아직 저장이 안된 상태에서 프론트엔드에 응답을 했을 때 "OK"로 보내면 안된다. 응답은 "진행 중"으로 와야한다.
			- 동기 비동기 뿐만 아니라, 조회할 때는 어떤건 MySQL(RDB)를 보고, 어떤건 Elastic search를 보고 이런 식으로 하면 안됨 -> 이렇게 다르게 있다면 다른 Controller에서 처리를 해야함.
			- API 설계만 봐도 내부를 유추할 수 있어야 함.

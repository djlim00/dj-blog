## 1. 안정 해시

- **안정 해시** : <mark class="hltr-yellow">해시 테이블 크기가 조정될 떄 평균적으로 오직 k/n개의 키만 재배치하는 기술</mark>
	- k = 키의 개수
	- n = 슬롯(slot)의 개수
-> 위에서 발생하는 대규모 캐시 미스를 방지할 수 있음.

### 1-1. 해시 공간과 해시 링

- 해시 함수 f로 SHA-1를 사용.
- 함수의 출력 값 범위는 x0, x1, x2, ... , xn 이라고 하자.
	- SHA-1의 해시 공간의 범위는 0 ~ 2^106 -1
	- x0 = 0, xn = 2^160 -1

## 3. 가상노드

- 실제 노드 또는 서버를 가리키는 노드(일종의 포인터)
- 하나의 서버는 링 위에 여러 개의 가상 노드를 가질 수 있음.
![[Pasted image 20251113152058.png]]
- 서버0은 3개의 가상노드 / 서버1은 3개의 가상노드 => 실제로는 훨씬 많은 가상노드.
- **그렇게 되면 하나의 서버는 여러개의 파티션을 관리해야한다.**
- **가상 노드의 개수를 늘리면 키의 분포는 점점 더 균등해짐**
	- 표준 편차(데이터가 어떻게 퍼져 나갔는지)가 작아져서, 데이터가 고르게 분포.
	- 100~200개의 가상 노드를 사용했을 경우, 표준 편차의 값은 평균의 5%(가상노드200개)~10%(가상노드100개)
	- 더 많은 가상노드를 사용하면, 이 정보를 가지고 있는 데이터 공간이 필요함. <mark class="hltr-yellow">(타협)</mark>

## 마치며

### 안정 해시의 이점
1. 서버가 추가되거나, 삭제될 때 재배치되는 키의 수가 최소화된다.
2. 데이터가 보다 균등하게 분포되므로, 수평적 규모 확장성을 달성하기 쉽다.
3. 핫스팟 문제를 줄인다. 특정 샤트에 대한 접근이 지나치게 빈번하면 서버 과부화.

##### ex) 디스코드 채팅 애플리케이션
##### ex) 아파치 카산드라 클러스터에서의 데이터 파티셔닝

# 🔥 챌린지 - 토론

1. Read-Heavy System에서 ConsistentHash의 add 및 remove 메서드에 synchronized 키워드를 사용하는 것과 RWLock을 사용하는 것 중 어느 것이 더 적합할까요? 동시성이 제어되지 않는다면 어떤 문제가 발생할 수 있을까요?
   :  우선 Read-Heavy System이라면 get() 메서드 호출을 통해 값을 읽어오는 연산이 가장 많을 것이다. 락을 적용하지 않으면, get()호출을 하면서 해시링은 순회하는 도중 add() 혹은 remove()연산이 발생하면서 get()의 결과로 가져오는 값이 달라지는 더티 리드가 발생할 수 있음. 
2. 1번에서 언급된 방법 외에도 Spring Boot과 같은 멀티 스레드 환경에서의 동시성 문제를 해결하기 위해서 어떤 방식을 적용해볼 수 있을까요?
	1. **sychronized는 하나의 프로세스에서만 동작함**
		- 서버가 한개 이상이면 여전히 동시성 문제가 해결이 안된다.
	2. **SpringDataJpa의 비관적 락 @Lock(LockModeType.PESSIMISTIC_WRITE)**
		- 어떤 쓰레드가 해당 row나 table에 대해서 락을 걸면 다른 스레드에서 접근하지 못하게 한다.
		- 비관적 락은 데이터 정합성을 보장하는 대신에, 조회할 때 매번 락을 사용하기 때문에 DB 부하가 걸림(느려짐)
	3. **낙관적 락 @Lock(LockModeType.OPTIMISTIC)**
		- 스레드 A,B가 동시에 version 1.0인 데이터에 접근하면 -> 스레드 A가 작업을 완료하여 데이터가 변경되면 데이터의 version은 2.0이 된다. -> 스레드B는 version 변경이 발생했으므로 새로운 version의 데이터를 조회한다.
		- @Version 애노테이션을 이용해서 version필드를 해당 데이터의 객체 필드에 추가해야함.
		- version 변경에 의한 재요청 로직도 직접 작성
		- 조회할 때 Lock을 사용하지 않아서 대체로 비관적 락보다 성능이 좋지만, version 충돌이 자주 발생하면(동일한 데이터에 접근이 많다면) 비관적락보다 성능이 느릴 수 있음.

   :
	1. 새로운 노드로 데이터를 복제하는 동안, 기존 노드로 읽기 요청을 처리
	   장점 : 서비스 중단 없음
	   단점 : 복제 비용, 데이터 중복
	2. 데이터 이동 중에 해당 키값을 저장해두었다가(큐) 이관이 끝나면 처리
	   장점 : 데이터 중복 없음(일관성)
	   단점 : 응답지연, 추가적인 저장공간(큐) 관리


## WorkBook

- **분산 캐시 시스템에서 가장 단순한 부하 분산 방식인 `serverIndex = hash(key) % N` 의 가장 큰 문제점은 무엇인가요?**
  : 새로운 노드가 추가/삭제 되면 N값이 바뀌게 되면서 대부분의 해시값이 새롭게 갱신되버림.
- **안정 해시란 무엇이며 어떤 근본적인 장점을 가지나요?**
  : 안정 해시란, 새로운 노드가 추가/삭제 되더라도 해시가 갱신되는 부분이 최소화될 수 있는 구조의 해시이다.
    - **해시 링 위에 서버와 키를 배치하는 과정을 설명해 보세요.**
	    : 서버는 해시함수에 따라서 부여받은 해시값에 따라서 해시링에 배치되게 된다. 이후 key값들도 동일한 해시함수에 의해서 해시값을 부여받고, 서버 해쉬값의 범위에 속하게 된다.
- **기본적인 안정 해시 알고리즘이 가질 수 있는 두 가지 문제점은 무엇인가요?**
  : 첫번째로는 해시범위가 균등하게 나누어지지 않는 문제가 있음. 어떤 서버(노드)는 많은 범위를, 어떤 서버는 작은 범위만 할당받을 수 있음. 두번째 문제는 첫번째 문제에서 파생되는 문제로, key가 균등하게 분포하지 않게 되면서 특정 해시범위에만 접근이 빈번해지는 문제가 이음.
- **안정 해시 링에서 가상 노드의 수를 늘리면 어떤 장점과 단점이 있을까요?**
  : 가상노드 수를 늘리게되면 노드의 해시값의 표준편차가 줄어들게 되어서, 균등하게 해시링에 분포하게 된다. 따라서 각각의 서버가 처리하는 key값이 균등하게 배분된다. 단점은 가상노드의 수가 늘어나게 되면서 가상노드와 실제 서버에 대한 정보를 저장하기 위한 저장공간이 늘어나게 된다.

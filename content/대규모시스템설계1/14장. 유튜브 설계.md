## 2단계 : 개략적 설계안 제시 및 동의 구하기

- CDN과 BLOB스토리지는 기존 클라우드 서비스를 활용한다.
	- **이유1 : 시스템 설게 면접은 모든 것을 밑바닥에 만드는 것이 아니다.**
	- 주어진 시간에 적절한 기술을 골라서 설계를 마치는 것이 중요(각각의 동작보다 중요)
		- 비디오 저장을 위해서 그냥 단순히 BLOB 저장소를 사용한다고 해도 괜찮다(저장소 구현 방법을 자세히 아는 것은 면접에는 중요하지 않다)
	- **이유2 : 규모 확장이 쉬운 BLOB 저장소나 CDN을 만드는 것은 너무 복잡하고, 비용이 많이 든다.**
		- 거대한 회사도 직접 구축하지 않음. 넷플릭스도 AWS씀.
![[Pasted image 20260223202613.png]]
- Client : 컴퓨터, 모바일, 스마트TV에서 유튜브 시청
- CDN : 비디오는 CDN에 저장(재생을 누르면 CDN에서 스트리밍)
- API서버 : 비디오 스트리밍을 제외한 모든 요청은 API 서버가 처리한다. 피드 추천, 비디오 업로드 URL 생성, 메타데이터DB와 캐시 갱신, 사용자 가입 등등이 API서버가 처리하는 작업

### 1. 비디오 업로드 절차
![[Pasted image 20260223202742.png]]
- 사용자 : 컴퓨터, 모바일, 스마트TV를 통해 유튜브를 시청
- **로드밸런서 : API서버 각각으로 고르게 요청을 분산하는 역할**
- API 서버 : 비디오 스트리밍을 제외한 모든 요청을 처리
- **메타데이터 데이터베이스 : 비디오의 메타데이터를 보관, 샤딩(Sharding)과 다중화(replication)를 적용하여 성능 및 가용성 요구사항을 충족**
- **원본 저장소 : 원본 비디오를 보관할 대형 이진 파일 저장소(BLOB) 시스템.(이진 데이터를 하나의 개채로 보관하는 DB관리 시스템)** 
- <font color="#ffc000">트랜스코딩 서버</font> : 비디오 트랜스코딩은 비디오 인코딩. 비디오의 포맷(MPEG, HLS)을 변환하는 절차. 단말이나 대역폭 요구사항에 맞는 최적의 비디오 스트림을 제공
- <font color="#ffc000">트랜스코딩 비디오 저장소</font> : 트랜스코딩이 완료된 비디오를 저장하는 BLOB 저장소
- CDN : 비디오를 캐시하는 역할. 사용자가 재생 버튼을 누르면 비디오 스트리밍은 CDN을 통해 이루어짐
- <font color="#ffc000">트랜스코딩 완료 큐</font> : 비디오 트랜스코딩 완료 이벤트를 갱신할 작업 서버
- <font color="#ffc000">트랜스코딩 완료 핸들러</font> : 트랜스코딩 완료 큐에서 이벤트 데이터를 꺼내어 메타데이터 캐시와 데이터베이스를 갱신할 작업 서버

## 3단계 : 상세 설계

- 업로드 / 스트리밍 -> 최적화 방법에 대해서 알아보자.

### 비디오 트랜스코딩

- 카메라or스마트폰으로 동영상을 찍으면 다른 단말과 호환이 되는 비트레이트와 포맷으로 저장되야함.
- 비트레이트가 높으면 -> 고화질 -> 재생하기 위해서 높은 성능의 컴퓨팅 파워가 필요 / 인터넷이 빨라야함.

#### 비디오 트랜스코딩이 중요한 이유

1. 원본 비디오는 용량이 크다 -> 초당 60프레임으로 녹화된 HD비디오는 수백 GB의 저장공간을 차지할 수도 있음
2. 상당수의 단말과 브라우저는 특정 종류의 비디오 포맷만 지원 -> 호환성 문제를 해결해야함(여러 포맷으로 인코딩)
3. 대역폭에 따라서 화질을 고화질/저화질 변하는게 바람직함.

#### 인코딩 포맷
- **컨테이너** : 비디오 파일, 오디오, 메타데이터를 담는 바구니. 포맷은 .avi, .mov, .mp4를 보면 알 수 있음.
- **코덱** : 화질은 보존하면서 파일 크기를 줄일 목적(압축/압축해제 알고리즘). J.264, VP9, HEVC

### 유향 비순환 그래프(DAG) 모델

- 각기 다른 유형의 비디오 프로세싱 파이프라인을 지원하면서 -> 처리 과정의 병렬성을 높이기 위해서 적절한 수준의 추상화를 도입해야함.
- 클라이언트 프로그래머로 하여금 task를 손수 정의할 수 있도록 해야함.
	- ex) 페이스북의 스트리밍 비디오 엔지은 유향 비순환 그래프(Directed Acyclic Graph) 프로그래밍 모델을 도입
-  <font color="#ffc000">즉 비디오 업로드(트랜스코딩) 과정을 유연하고 병렬적으로 하기 위해서 사용한다.</font>

### 비디오 트랜스코딩 아키텍처
![[Pasted image 20260224161649.png]]
#### 1. 전처리기 역할
1. **비디오 분할** : 비디오 스트림을 GOP라고 불리는 단위로 쪼갠다. GOP는 특정 순서로 배열된 프레임 그룹. 하나의 GOP는 독립적으로 재생 가능(몇 초 정도).
	- 오래된 단말/브라우저는 GOP단위의 비디오 분할 지원 X -> 그러면 전처리기가 하는 것임.
2. **DAG 생성** : 클라이언트 프로그래머가 작성한 설정 파일에 따라 DAG를 만든다. 
	- 14-12은 2개 노드와 1개 연결선으로 구성된 DAG의 사례.
	![[Pasted image 20260224161952.png]]
3. **데이터 캐시** : 전처리기는 분할된 비디오의 캐시이기도 하다. 안정성을 높이기 위해 전처리기는 GOP와 메타데이터를 임시 저장소에 보관한다. -> 인코딩에 실패하면 다시 임시보관소에서 꺼내서 인코딩 재개.

#### 2. DAG 스케쥴러의 역할
: DAG 그래프를 몇개 단계로 분할한 다음에 그 각각을 <u>자원 관리자</u>와 <u>작업 큐</u>에 넣는다.	

![[Pasted image 20260224162237.png]]
- 14-15는 하나의 DAG 그래프를 2개 작업으로 쪼갠 사례
- 1단계 -> 비디오, 오디오, 메타데이터 분리
- 2단계 -> 비디오 파일을 인코딩, 썸네일 추출, 오디오 파일 인코딩.

#### 3. 자원 관리자.
: 자원 배분을 효과적으로 수행하는 역할
: 아래 그림에는 세 개의 큐와 작업 스케줄러로 구성된다.

![[Pasted image 20260224162647.png]]
- 작업 큐 : 실행할 작업이 보관되어 있는 PQ
- 작업 서버 큐 : 작업 서버의 가용 상태 정보가 보관되어 있는 PQ
- 실행 큐 : 현재 실행 중인 작업 및 작업 서버 정보가 보관
- 작업 스케줄러 : 최적의 작업/서버 조합을 골라서 해당 작업 서버가 작업을 수행하도록 지시

![[Pasted image 20260224162808.png]]

#### 4. 작업 서버
: DAG에 정의된 작업을 수행한다. 
: 14-19 처럼, 작업 종류에 따라 작업 서버도 구분해서 관리
![[Pasted image 20260224162908.png|300]]

#### 5. 임시 저장소
: 다양한 저장소 시스템을 활용할 수 있음.(시스템에 따라서 데이터의 유형, 크기, 이용 빈도, 데이터 유효기간들이 달라짐.)
: ex) 메타데이터 -> 빈번히 조회 / 크기가 작업 -> 메모리의 캐싱
: ex) 비디오/오디오 -> BLOB 저장소

#### 6. 인코딩된 비디오

- 기존의 시스템의 절차는 이전 단계의 결과물을 입력으로 받아야지 출력이 나오는 구조

### 안전성 최적화: pre-signed URL

- 허가받은 사용자만 올바른 장소에 비디오를 업로드할 수 이게 하기 위해서 pre-signed 업로드 URL을 사용.
![[Pasted image 20260224164627.png]]
1. POST/upload 요청을 해서 pre-signed URL을 받는다. S3에서 부르는 말임.
2. URL 반환
3. URL을 통해서 업로드

### 안전성 최적화: 비디오 보호

- 원본 도난(저작권)

1. 디지털 저작권 관리(DRM) 시스템 도입
	- Apple의 FairPlay, 구글의 Widevine, MS의 PlayReady
2. AES 암호화 : 비디오를 암호화하고 접근 권한을 설정하는 방식. 재생 시에만 복호화된다.(허락된 사용자만 시청 가능)
3. 워터마크 : 비디오 위에 소유자 정보를 포함하는 이미지 오버레이(회사이름, 이름)

## 마무리

### 추가적인 논의 사항

#### API계층의 규모 확장성 확보 방안
- 무상태 서버 -> 수평적 규모 확장 가능

#### DB 계층의 규모 확장성
- 샤딩, 다중화

#### 라이브 스트리밍
- 실시간으로 녹화하고 방송(업로드, 인코딩, 스트리밍) 이라서 비슷한 부분이 많음.

- **차이점**
	- 응답지연이 더 낮아야함.(스트리밍 프로토콜 선정에 유의해야함)
	- 병렬화 필요성은 떨어짐 -> 작은 단위의 데이터를 실시간으로 빨리 처리해야함.
	- 오류 처리 방식이 달라야함 -> 너무 많은 시간이 걸리는 방법은 X

#### 비디오 삭제
- 저작권 위반, 선정적, 불법적 비디오 삭제
- 업로드 과정에서 식별 or 사용자 신고

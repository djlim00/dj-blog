# 1. Docker 개념 정리

- 내가 생각하는 Docker는 모든 것에 대한 JAR/WAR 파일 같은 느낌. 우리가 돌리고 있는 어떤 환경 자체를 그냥 통채로 떠서 이동시키는 기술 -> 생각보다 별거 아닌거 같은데 왜 이제야 나타난거지?

## 1. Image(이미지)
- **정적인 실행 패키지**
- Spring 서버의 JAR 파일, JDK, OS 환경 등을 **미리 정의한 설정(Dockerfile)에 따라 하나의 이미지**로 만든다.
- 예: `my-spring-app:latest` 같은 이름을 가질 수 있음.

## 2. Container(컨테이너)
- 이미지로부터 **실행된 인스턴스** (실제 구동되는 상태)
- 여러 개의 컨테이너가 동시에 실행될 수 있음 (포트 다르게 하면 Spring 서버도 병렬 가능)

## 3. Volume(볼륨)
- **컨테이너 내부 데이터는 휘발성이라 재시작하면 사라짐**
- 그래서 로그, DB 파일, 설정 파일 등 **지속적인 저장이 필요한 데이터는 호스트와 연결된 디스크**에 저장해야 함 → 이걸 Volume이라 해
	- <font color="#ff0000">그래서 이걸 안지우면 새로 키고 싶어서 컨테이너랑 이미지를 아무리 지워도 다시 키면 예전이랑 똑같음..😡</font>

# 2. EC2에서 Docker를 쓰면 뭐가 좋나용?

#### 1. EC2에 Java 설치, JDK 설정, Gradle 빌드 등 복잡한 작업 필요 없음
#### 2. Dockerfile 한 번 만들어두면 **EC2에서 명령 한 줄로 배포 가능**
#### 3. 로컬에서 실행되는 그대로 EC2에서도 실행됨 → 환경 차이 문제 제거
- 더이상 제 컴퓨터에서는 잘 돌아가는데.. 이런거 안됨
#### 4. 여러 버전, 포트를 컨테이너로 나눠 구동 가능 (예: 테스트용, 운영용)

- 걍 미친거지 미친거야..

# 3.  Dockerfile 작성

- 자 이제 본격적으로 Docker로 배포를 해보자.

```Dockerfile title:"Dockerfile"
# 1. 베이스 이미지 선택 (JDK 포함된)
FROM openjdk:17-jdk

# 2. JAR 파일 복사
COPY build/libs/your-app.jar app.jar

# 3. 컨테이너 실행 시 JAR 실행
ENTRYPOINT ["java", "-jar", "/app.jar"]
```
- `./gradlew bootJar` 를 통해서 빌드 파일을 먼저 만들어야 한다!

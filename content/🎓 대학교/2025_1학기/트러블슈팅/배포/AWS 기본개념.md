# 1. VPC (Virtual Private Cloud)

![[Pasted image 20250612221351.png|300]]

- 논리적으로 격리된 사용자 전용 가상 네트워크.
- AZ(가용영역)이 여러개 있는 형태
- **즉 AWS의 리소스들이 위치할 네트워크 망**
- 만약 EC2, RDS, S3와 같은 리소스들이 물리적으로 하나의 데이터 센터에만 위치해 있다면, 해당 센터가 문제가 생기면 서비스가 전체적으로 다운타임이 생김
- **But 가상네트워크인 VPC를 이용해서, 물리적으로 다른 곳에 위치하미나 같은 사설망 IP 대역에 위치하게 만들어서 리소스들끼리 통신할 수 있게 해주는 기술**

# 2. Subnet
![[Pasted image 20250612221541.png]]
- VPC의 영역안에서 망을 더 쪼개는 행위
- **VPC와 달리 지역적으로 나누기 때문에 여러 AZ에 걸쳐있는 것이 아니라 단일 AZ에 위치한다.**

- <mark style="background: #BBFABBA6;">대표적으로 보안과 네트워크 브로드캐스트의 영역을 축소시켜서 통신 성능과 보안을 지키기 위함</mark>
	- 처음 AWS 계정을 생성하면 기본으로 잡힌 VPC 안에 4개의 Public subnet이 잡혀있음.


# 2-1. Public Subnet

- <mark style="background: #BBFABBA6;">외부에서 접근이 가능한 네트워크 영역</mark>
	- 서브넷이 인터넷 게이트웨이로 향하는 라우팅이 있는 라우팅 테이블과 연결되는 경우

- <u>인터넷 게이트웨이(igw)는 VPC의 구성요소로써 VPC와 인터넷 간에 통신</u>을 할 수 있게 만들어주는 역할을 함.
	- 즉 서브넷이 바라보는 라우팅 테이블에 인터넷 게이트웨이가 등록되어있음
	- <mark style="background: #BBFABBA6;">해당 서브넷이 위치한 리소스들은 공인IP를 가질 수 있음</mark>


# 2-2. Private Subnet

![[Pasted image 20250612222706.png]]

- <mark style="background: #BBFABBA6;">외부에서 다이렉트로 접근이 불가능한 네트워크 영역</mark>
	- private subnet의 라우팅 테이블에 인터넷게이트웨이가 등록되어있지 않음. -> **리소스들이 외부와의 연결이 불가능**

ex) 만약 private 서브넷에 DB가 있다고 하면, DB버전 업그레이드를 위해서 인터넷 연결이 필요할 수도 있음
- NAT 게이트웨이를 이용해서 **내부에서 외부로만 접근이 가능하게 할 수 있음**
- 여기서 외부 -> VPC 영역이 다른 것 / 같은 VPC에 잇으면 private IP로 접근 가능

1. NAT 게이트웨이 설정 방법은 **퍼블릭 서브넷에 NAT 게이트웨이에 EIP를 할당하여 생성 후 프라이빗 서브넷의 라우팅 테이블에 등록**해주면됩니다.
2. 즉, 프라이빗 서브넷에서 라우팅 테이블을 보고 인터넷으로 나가는 규칙 (0.0.0.0)에 대해서는 NAT 게이트웨이를 타고 **EIP(공인 IP)** 로 변환 후 igw를 통해 인터넷으로 나가는것입니다.

# RDS

- 관계형 데이터베이스를 간편하게 클라우드에서 설정, 운영, 확장이 가능하도록 지원하는 웹 서비스
- DB 설치, 모니터링, 백업, 알람 등 관리를 대신 해주며
- 하드웨어 프로비저닝, 데이터베이스 설정, 패치 및 작업과 같은 운영 작업을 자동화하여 비용이 효율적이고 크기 조정 가능한 DB서비스를 제공한다.

- 물론 EC2가 컴퓨터니까, EC2 인스턴스에 직접 DB를 설치해서 사용해도 되긴한다 -> 근데 직접 구축하여 운영할 때 보다 자동으로 많은 것을 자동화 할 수 이어서 애용한다.

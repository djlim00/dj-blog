# 1. 코드형 인프라(IaC) & CloudFormation

## IaC(Infrastructure as Code) 개념

코드를 사용해 인프라(AWS 리소스, 네트워크, IAM 등)를 정의하고 자동으로 생성·관리하는 방식.  
→ “콘솔로 클릭하는 작업을 코드로 반복·재현 가능하게 만든 것”

### 장점

- **Stability** : 사람 손실 줄여 안정적인 인프라 구성
- **Scalability** : 여러 계정·리전에 동일 인프라를 재생성
- **Visibility** : 모든 인프라가 파일로 기록
- **Security** : 변경 이력, 코드 리뷰 가능
- **Transactional** : CFN 스택 단위 생성 → 실패 시 rollback 가능

---

## AWS CloudFormation(CFN)

IaC의 AWS 대표 서비스.  
템플릿(JSON/YAML) → CFN에 제출 → 리소스 자동 생성.

###  템플릿 구조

- `AWSTemplateFormatVersion`
- `Description`
- `Parameters` (외부 입력: VPC ID, Subnet ID 등)
- `Mappings`, `Conditions`
- `Resources` **가장 중요**
- `Outputs` (S3 버킷 이름, ARN 등 외부로 노출되는 값)

---

## CloudFormation 동작 흐름

1. 템플릿 작성 (S3 업로드 포함 가능)
2. CFN 스택 생성
3. 필요 시 Change Set(변경 요약) 확인
4. 리소스 자동 생성
5. 스택 삭제하면 모든 리소스 자동 삭제

---

## 실습 내용 정리 – CloudFormation

### ① S3 버킷 생성 템플릿

- 기본적인 S3 생성 예시를 통해 CFN 구조 학습.

### ② SNS → Lambda 구독 구조(CFN 템플릿)

- SNS Topic 생성
- Lambda 함수 생성
- Topic과 Lambda 연결(SNS → Lambda event)
- IAM Role, Permission 자동 정의

---

## AWS SAM(Serverless Application Model)

CloudFormation 기반의 서버리스 특화 프레임워크.
### 특징

- Lambda, API Gateway, DynamoDB 같은 **서버리스 자원**을 간단한 문법으로 정의
- `sam build` → 패키징
- `sam deploy` → CFN 스택 생성

### 예시: SNS → Lambda SAM 템플릿

CloudFormation 대비 훨씬 간결하게 SNS + Lambda 자동 구성 가능.

# 2️. 서버리스(Serverless) & FaaS(Function as a Service)

## 서버리스(Serverless)란?

> “서버는 존재하지만, 개발자가 서버를 관리하지 않는 컴퓨팅 방식”

### 장점

- 서버 관리 부담 없음
- 자동 확장(오토스케일링)
- 사용한 만큼만 비용 지불(pay-per-use)
- 단순한 배포 → 비즈니스 로직 개발 집중 가능

### 대표 서비스

- Compute: **AWS Lambda**, Fargate
- Storage: **S3**
- Message Broker: SNS, SQS, Kinesis
- Authentication: Cognito
- Database: DynamoDB, Aurora Serverless
- API Gateway

---

## FaaS(Function as a Service)

- 이벤트 기반으로 작동
- 작은 단위의 함수(Function)를 호출
- API Gateway / S3 / CloudWatch 등에서 트리거 가능
- 서버 관리 필요 없음

---

## AWS Lambda 특징

- On-demand 실행
- 언어 지원 다양(Python, Node.js, Go, Java 등)
- 자동 확장
- 짧은 실행 시간 제한
- 콜드스타트 존재(특히 Java)

---

# 3️. DynamoDB 기본 개념

## DynamoDB 특징

- 완전관리형 NoSQL DB
- 빠른 성능, 무한 확장
- 테이블 기반
- Primary Key 구조:
    - Partition Key(HASH)
    - Sort Key(RANGE)
- Secondary Index(GSI, LSI) 지원
- ACID보다 BASE 모델(고가용성 중심)

---

## 주요 연산

###  Query

- Partition Key 기반으로 빠른 조회
- Sort Key 조건 가능
- 인덱스 통해 다양한 조회 가능

### Scan

- 테이블 전체 스캔 → 비용 비싸고 느림
- 대규모 사용 지양

# 4️. 데이터 파이프라인(Data Pipeline)

## 데이터 파이프라인 구성 요소

1. **데이터 수집(Ingestion)**  
    → API Gateway / SDK / 로그 등
2. **중간 저장소**  
    → S3, DynamoDB, RDS
3. **전송 매체**  
    → Kinesis, SQS, SNS
4. **가공·처리(Compute)**  
    → Lambda, Glue, Batch, Step Functions
5. **분석(Analytics)**  
    → Athena, Redshift, Elasticsearch
6. **시각화(Visualization)**  
    → QuickSight, Grafana

---

## 광고 유입 경로 분석 파이프라인(실습)

### 목표
SNS 광고(Instagram, Facebook, Twitter)에서 들어오는 사용자 유입 경로 로그 수집·분석하기.
### 구성

1. 사용자가 광고 링크 클릭 → API Gateway `/home` 호출
2. Lambda(FunnelLoggingFunction)가 Referer 기반 주소 분석
3. DynamoDB(FunnelAnalysis)에 저장
4. API Gateway `/funnel/{proxy}` 호출 시
5. FunnelQueryingFunction이 DynamoDB에서 count 조회 → 반환

### DynamoDB 테이블

- uid
- timestamp
- referer
- funnel
- userinfo

여기에 **유입경로 매핑 테이블, 인덱스(GSI/LSI)** 를 구성하여 Query 성능 최적화.

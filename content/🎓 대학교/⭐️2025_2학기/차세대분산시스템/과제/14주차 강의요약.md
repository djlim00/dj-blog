# 하이퍼바이저와 하드웨어 가상화 정리

## 1. 하이퍼바이저의 종류

하이퍼바이저는 가상 머신을 생성하고 관리하는 소프트웨어 계층으로, 구현 방식에 따라 여러 유형으로 나뉜다.

### 1-1. Type 0 하이퍼바이저 (Firmware 기반)
![[Pasted image 20251206205737.png]]
- 하드웨어 제조사가 펌웨어 수준에서 제공하는 가상화 기능
- 운영체제는 특별한 수정 없이 그대로 동작하며, 게스트 OS별로 하드웨어를 물리적으로 분리해주는 방식 
- 기능은 비교적 제한적이지만 안정성이 높고, 가상화 위에 다시 가상화를 얹는 구조도 가능
	- IBM LPARs, Oracle LDOMs 

### 1-2. Type 1 하이퍼바이저 (Bare Metal)
![[Pasted image 20251206205929.png|300]]
- 운영체제처럼 보이지만 실제로는 하드웨어 위에서 직접 실행되는 가상화 전용 OS
- 커널 모드에서 동작하며, 모든 하드웨어 자원을 제어하고 게스트 OS를 관리
	- 게스트는 가상 환경에서 동작하는 것을 인지하지 못한다.  
- 데이터센터 : Xen, VMware ESXi, Citrix XenServer

### 1-3. Type 2 하이퍼바이저 (Hosted)

- 일반 운영체제 위에서 애플리케이션 형태로 실행되는 구조
- 커널 모드 기반과 유저 모드 기반으로 나뉘며, 후자의 경우 운영체제는 VMM이 어떤 역할을 하는지조차 모름
- 성능은 Type 1보다 떨어지지만 설치와 사용이 쉽고, 호스트 OS를 수정할 필요가 없다는 장점
- ex) VirtualBox, VMware Workstation

## 2. 가상화 방식 분류: 특권 명령 처리 방법 기준

가상화를 구현할 때 중요한 문제는 게스트 OS가 실행하는 특권 명령을 어떻게 처리하느냐임!
![[Pasted image 20251206205956.png]]

### 2-1. 에뮬레이션과 직접 실행

에뮬레이션은 게스트의 명령어 집합(예: ARM)을 실제 하드웨어(x86)에서 실행할 수 있도록 번역하는 방식이다.  
1. 호환성이 높지만 성능이 떨어진다.  
2. 반대로 직접 실행 방식은 게스트와 호스트의 아키텍처가 동일할 때 사용되며, 번역 과정을 생략해 성능이 높다.  
3. 특권 명령을 안전하게 처리할 수 있어야 한다.

### 2-2. Trap & Emulation

1. 전통적인 가상화 구현 방식으로, 하이퍼바이저는 특권 모드에서 실행되고 게스트 운영체제는 비특권 모드에서 실행된다.  
2. 게스트가 특권 명령을 실행하면 트랩이 발생하고 하이퍼바이저로 제어권이 넘어간 뒤, 해당 명령을 에뮬레이션해 처리한다.  
3. 하지만 x86 구조에는 트랩으로 감지되지 않는 민감 명령이 존재해 구현이 까다롭다.

## 3. CPU 직접 실행 기반 가상화: Binary Translation과 Hypercall

Trap & Emulation의 한계를 해결하기 위해 등장한 방식이다.

### 3-1. Binary Translation (Full Virtualization)

- 하이퍼바이저가 게스트 OS의 바이너리 코드를 스캔하여 문제될 수 있는 명령어(민감 명령어)를 안전한 코드로 변환한다.  
- 게스트 OS는 수정 없이 그대로 실행할 수 있다는 장점이 있지만, 변환 과정 때문에 구조가 복잡하고 성능 손실이 생긴다.

### 3-2. Hypercall (Para-Virtualization)

- 게스트 운영체제의 일부 코드를 수정하여, 특권 명령에 해당하는 부분을 하이퍼바이저 호출(hypercall)로 치환하는 방식이다.  
- 게스트 OS를 수정해야 한다는 단점이 있지만 성능 오버헤드가 낮다.  
- 현대 CPU의 하드웨어 가상화 기능(VT-x, AMD-V)이 도입된 후에는 게스트 OS 수정 없이도 유사한 성능을 구현할 수 있게 되었다.

---

## 4. 하드웨어 지원 가상화

1. Intel VT-x나 AMD-V 같은 CPU 기반 가상화 지원 기능이 등장하면서, 기존 소프트웨어 기반 가상화가 지닌 문제들이 해결되었다.  
2. CPU가 직접 하이퍼바이저와 게스트 OS를 위한 별도의 실행 모드를 제공하며, 민감한 명령어는 자동으로 트랩되도록 설계되어 있다.  
3. 덕분에 바이너리 변환이나 게스트 OS 수정 없이 고성능 가상화가 가능해졌고, KVM 같은 하이퍼바이저가 널리 사용되기 시작했다.

### 전체 구조(개념)

1. **데이터 수집 레이어**
    - **Lambda-Collector**
        - EventBridge(스케줄러)로 1시간/하루 단위 트리거
        - 서울시 공공데이터 API 호출 → 축제 데이터 가져오기
        - 데이터 전처리(필드 정리, 날짜 파싱, 중복 제거)
    - 저장 위치:
        - **RDS(PostgreSQL)** :
            - 정규화된 테이블 (축제, 장소, 사용자, 선호도, 즐겨찾기 등)
            - 앱에서 검색, 필터링, 즐겨찾기 관리, 추천 로그 저장
        - **MongoDB** :
            - RAG용 문서 저장 (축제 설명, 상세 텍스트, 태그, 카테고리 등)
            - 한 축제 = 한 document 형태
                
2. **임베딩 & 벡터 저장 레이어**
    - **Lambda-Embedder (또는 배치 스크립트)**
        - MongoDB에서 축제 문서 불러오기
        - “설명/상세정보” 텍스트 → 임베딩 생성(외부 LLM API 사용)
        - 임베딩 벡터를 **MongoDB document의 필드로 저장**
            - 예: `embedding: [0.0123, -0.5123, ...]`
    - 벡터 검색 방식
        - **쉬운 버전(과제용으로도 충분)**
            - Lambda/백엔드에서 MongoDB에서 모든(or 후보) 문서를 읽어온 뒤,  
                메모리에서 **코사인 유사도 계산 → 상위 N개 선택**
        - **고급 버전**
            - MongoDB Atlas Vector Search나,  
                별도 벡터 DB(pgvector, Pinecone 등) 사용
            - 과제면 “아키텍처 상으로는 벡터 DB로 확장 가능” 정도만 써줘도 충분해 보여.
                
3. **챗봇 / API 레이어**
    - **RESTful API 서버** (요구사항 충족용)
        - Spring Boot or FastAPI (ECS/EKS/EC2 어디든)
        - `/festivals`, `/recommendations`, `/users/{id}/preferences` 등은 여기서 제공
    - **챗봇 API (Lambda-Chatbot)**
        - API Gateway → Lambda-Chatbot 연결
        - 입력: `user_id`, `question`
        - 처리:
            1. 질문 임베딩 생성
            2. MongoDB에서 유사도 높은 축제 문서 Top-k 검색
            3. 그 문서를 context로 해서 LLM 호출 (RAG)
            4. 최종 답변 반환(JSON)

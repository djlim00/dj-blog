## 1장. 데이터 모델링의 이해

![[Pasted image 20251114122129.png]]
정답 : 3
- <mark class="hltr-yellow">일관성 저해</mark> : 사용자가 처리하는 프로세스 혹은 이와 관련된 프로그램과 테이블의 연계성을 높이는 것은, 데이터 모델이 업무 변경에 대해 취약하게 만드는 단점이 해당함
- 2번은 왜 맞는 말인가? - 데이터의 사용 프로세스와 분리하지 않으면, 사소한 업무(사용)에 의해서 수시로 변경될 수 있음. 즉 유연하지 않게 되버림.

![[Pasted image 20251114122408.png]]
정답 : 3,4
- <mark class="hltr-yellow">데이터모델링에서는 존재적 관계와 행위에 의한 관계를 구분하는 표기법은 <u>없음</u>. UML에서는 연관관계에 대해 다른 표기법을 가지고 표현하게 되어있다.</mark>
- 3번 : ERD에서는 존재/행위에 대한 관계 구분 X
- 4번 : UML에서는 연관/의존 관계에 대해서 실선/점선

![[Pasted image 20251114122836.png]]
정답 : 4
- 주식별자에 의해 엔터티내에 모든 인스턴스들이 유일하게 구분되어야한다.
- 주식별자를 구성하는 속성의 수는 유일성을 만족하는 최소의 수가 되어야한다.
- 지정된 주식별자의 값은 자주 변하지 않는 것이어야 한다.
- 주식별자가 지정이되면 반드시 값이 들어와야 한다.

![[Pasted image 20251114123006.png]]
정답 : 4
- 사번은 업무적으로 의미가 있는 식별자 -> <mark class="hltr-yellow">시스템적으로 부여된 인조식별자가 아님.</mark> (인조식별자는 업무적으로 의미가 없어야함.)
- 일반적으로 사원 인스턴스의 탄생과 함께 업무적으로 부여되는 사원 인스턴스의 본질적인 속성에 해당한다 할 수 있음 -> 본질 식별자

![[Pasted image 20251114123151.png]]
정답 : 4
- 식별관계 -> 강한 관계(부모없이는 자식이 있을 수 없음)
- 자식 독립적 PK -> 비식별
- 모든 관계 식별자 관계 -> SQL Where절에서 비교하는 항목이 증가.. -> 조인에 참여하는 테이블에 따라 SQL문장이 길어져 -> SQL문의 복잡성을 증가되는 것을 방지하기 위해 -> 비식별관계 고려
	- 모두 식별자면 모든 자식은 부모의 PK를 포함해서 PK를 만듦
	- 점점 PK의 속성이 많아짐(복합키)
	- 조인 조건(ON절)에 자꾸 AND가 붙게 됨.
- 부모엔터티의 주식별자를 자식엔터티에서 받아 손자엔터티까지 계속 흘려보내기는 식별관계

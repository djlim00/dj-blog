## 4. 우리는 왜 MSA를 하나용?

- MicroService Architecture => <mark class="hltr-yellow">모든 것은 확장성을 위해!</mark>
	- 서버 스팩의 확장
	- 개발속도의 확장
	- 기술스택의 확장

- gRPC 고성능 통신
- 이벤트 기반 아키텍처(EDA) -> Event Driven Architecture
	- 카프카, RabbitMQ
	- 비동기 통신

#### 사가 패턴
	- 하나의 큰 비즈니스를  도메인별로 분리함 -> 여러개의 로컬 트랜잭션의 연속으로 처리
	- 중간에 실패하게 되면 이미 실행된 트랜잭션을 취소 -> 보상 트랜잭션 실행
	- <mark class="hltr-yellow">데이터의 최종 일관성을 보장</mark> 

#### CQRS패턴
	- 명령(생성/수정/삭제)와 조회(읽기)의 책임과 모델을 분리
	- 즉 쓰기 트래픽을 나누어서 생각한다.
	- ReadOnly DB와 Write only DB 분리


- MSA를 하게 되면 통합 테스트 보다는  -> 단위 테스트를 강화하는 방향으로 간다. 
- 멱등성 원칙을 보장해서 설계해야한다.

## 1. 좋은 테스트는 뭐라고 생각하세요?

- 고객의 핵심 요구사항을(가능한 한) 모두 검증해야함.
	- 그래서 프로그램이 문제 없이 동작함을 설명할 수 있도록
		- 나(지금 -> 6개월 뒤 -> n년 뒤)
		- 고객(동료 -> 재무팀 -> 돈을 지불하는 소중한 사람들)
- 그것을 최선을 다해 단순 명료해야함
	- 프로그래밍 언어인데도 누구라도 한 눈에 이애할 수 있도록
	- 명확한 입출력
- 그것은 매우 빨리 해야 함.
- 그것을 자동으로 해야 함.
-> 구현은 fancy하고 어려워도 된다. <span style="background:#fff88f">테스트는 읽었을 때 이해하기 쉽게!</span>


## 2. 좋은 테스트와 제품의 관계

만약 제품이 좋지 않다면 좋은 테스트가 나올 수가 없음
- <u>테스트 하기 어럽다면</u>
	- 입력을 통해 결과를 예측할 수 없음(<mark style="background: #BBFABBA6;">void반환이면 관찰이 힘듦</mark>)
	- 평범한 레퍼런스 테스트 기술을 사용할 수 없음
	- 결과가 너무 느리게 나옴(자동화가 어려움)
- <u>테스트를 자동화하고 있지 않다면</u>


# 3. "만약 테스트 코드가 없다면, 당신의 코드가 고객의 요구사항대로 만들어졌음을 고객에게 무엇으로 설명할 것인가?"


## 3-1. 단위 테스트 : 테스트의 시작(과 끝) 😀

제품을 기능의 집합이라고 할 때, 그 "단위 기능" 하나를 테스트하는 것을 말합니다.

때문에 단위 테스트는
- 당신이 할 수 있는 가장 빠른 테스트
- 제품에 문제가 있는 부분을 단위 기능 수준으로 좁게 특정
- <u>제품 빌드, 패키징, 배포 없이 실행 가능</u>
- 수시로, 반복적으로 실행 가능
- 고객이 이미 사용 중인 제품에 영향을 미치지 않음
-> <span style="background:#fff88f">고객에게 제품을 보증할 수 있는 가장 전문적인 방법</span>

#### 테스트가 없다면 -> 내 컴퓨터에서는 잘 되던데요?


## 3-2. 좋은 프로젝트의 테스트

- **hamcrest** : 복잡한 테스트 질의를 위한 라이브러리(~로 시작하는 String 검증)

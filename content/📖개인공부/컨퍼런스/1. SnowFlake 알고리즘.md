## 2. 일반적인 ID 생성방법

1. auto increment -> DB Transaction 필요
2. UUID, GUID 로 만들어내는 Random 값
	- 중복 체크 필요 -> DB 체크
3. 복합키(유저ID + 이벤드ID + 이벤트 단계ID)
	- 키 복잡도 증가

## 3. 실패 경험

1. 게임 서버 개발에 복합 키 적용
	-  유저ID + 이벤트ID + 이벤트단계ID
	- 단일 객체의 인덱스 크기 증가 -> 인덱스 삽입, 대용량 DB에 부적함
	- <mark style="background: #BBFABBA6;">단일 키 기반으로 변경</mark>
2. 대량 작업이 필요한 툴에서 자동 증가값(AutoIncrement) 사용
	- 연관 테이블을 넣으려니 ID 마다 트랜잭션 발생 -> 심각한 성능 저하

# 2. SnowFlake 알고리즘 소개

1. 트위터에서 개발된 ID 생성 알고리즘
2. 64비트 정수로 중복없는 고유한 값 생성 가능

### 서버 통합 시에도 유용하다.
-> WorkerID는 서버 아이디이기 때문에 절대 ID가 중복되지 않음!

### 매직 넘버를 사용하는 이유
1. UnixTimestamp는 1970-01-01 부터 시작하는 값
2. 41비트 Timestamp는 약 70년의 사용기간을 가지므로 그냥 사용하면 15년 정도의 제한을 가짐
3. 따라서 낭비를 줄이기 위해서 개발 시작 시점을 기준으로 만듦
### 응용과 주의해야할 점

- <u>UnixTimestamp를 항상 UTC를 기준으로 해야하는 이유</u>
1. 글로버 서비스를 목표로 한다면 서버 TimeZone으로 인한 문제가 발생
2. 로컬시간을 기준으로하면 서버 시간에 따라 ID 값이 중구난방으로 생성되어 기준이 모호해짐
3. 통일된 Timezone을 기준으로 ID를 생성한다면 CreatedTime에 대한 문제가 해결

- <u>동시성의 제어</u>
1. 서버 성능을 고성능화 하기 위해선 병렬처리가 필수적
2. 이에 따른 동시성 이슈로 동일한 PK가 생성될 확률이 존대
3. <mark style="background: #BBFABBA6;">Mutex 등의 Lock관리를 통해 동시성 제어가 필수적</mark>
<mark style="background: #BBFABBA6;">-> 처음에는 티가 안나다가 나중에 크게 터짐..!</mark>

- 중복된 값을 만들지 않기 위한 생성 서비스의 단일화
1. 한개의 서비스에서 ID생성 인스턴스를 단일로 관리(싱글톤)
2. 생성된 1개의 ID는 어떠한 경우에도 중복되지 않음을 의미
-> 실수로 다른 테이블에 값을 조회하더라도 Unique하기 때문에 조회되지 않을
-> 다른 데이터가 잘못 연결되는 실수 방지

- <u>각 사용되는 bit를 줄이거나 늘여서 필요에 따른 영역을 조절하는 방법</u>
1. 적은 데이터로 긴 기간을 바라보는 경우 (1970-01-01 ~ 20xx)
2. 서버의 갯수가 많이 늘어날 수 없는게 명확한 경우
3. 1ms에 생성되는 데이터의 수가 많지 않은 경우
-> 상황에 맞게 비트수를 조절하자


## Primary key
- Clustered Index로 유일한 값이고 물리적으로 정렬된다.
- 물리적인으로 정렬되므로 순차적인 증가에 적합함.
- 자동으로 인덱스가 생성된다.

- <u>ID 생성기로 얻을 수 있는 장점(SnowFlake)</u>
1. 유일한 값이며, 순차 증가하므로 PK에 적합함
2. BigInt 단일 값으로 인덱스 생성에 적합
3. TimeStamp 기반이므로 ID 기준으로 조회시 기간 조회에 매우 유리(앞의 비트를 가지고 시간을 만들어서 조회가 가능하기 때문에)


- <u>UUID, GUID 와의 차이</u> 
1. 문자열 기반의 난수 채번방식으로 크기가 크고 인덱스 캐시 성능이 저하(VARCHAR를 사용하기 때문에 너무 큼)
2. 순차 생성이 아닌 나수 채번으로 삽입순서사 비순차적이므로 인덱스 재정렬 발생
3. 서버 머신기준으로 난수 채번이므로 중복이 발생할 수 있음
4. UUIDv7 등 최신 채번방식을 선택한다면, timestamp 기반으로 사실상 크게 다른 건 없음.
   -> 굳이 쓸 이유가 있을까 싶다는 것이지요

- Auto Increment와의 차이
1. Auto Increment의 경우 DB 기준으로 자동 채번
2. DB기준이므로 무조건적인 DB 통신이 필수(성능저하)
3. 동일한 채번 기준을 가지므로, 분산서버에 적합하지 않음
   -> 2개 이상의 DB 사용시 동일 채번 문제
4. 단일 Row 기준 채번이므로 대량 작업에 부적합
   -> Insert후 채번된 id를 select 해야하는 연관관계

- 대량 작업에 최적화
1. 이벤트로 대규모 아이템 지금시
2. item 테이블에 넣을 데이터 생성 후 BulkInsert
3. Item_option, item_log 등 연관 데이터 BulkInsert
-> <mark style="background: #BBFABBA6;">만약 ID를 AutoIncrement를 한다면, 단일 Insert 후 채번된 값을 기준으로 연관 테이블을 넣어야하므로, 엄청나게 많은 트랙잭션이 발생</mark>


- 😅 Insert를 할 때 단일이냐 벌크이냐는 엄청난 차이가 난다...
		- 최근 DB에서는 전부 지원을 한다. -> 저번에 해봄

# 결론

1. 결국 SnowFlake는 시간이 포함된 고유한 ID를 생성하는 알고리즘이다. 
2. 부호 1비트를 제외한 63비트의 값을 나누어 정보를 분리하여 저장한다.
3. 다중 서버가 1개의 DB를 이용하는 경우, 다중 서버가 다중DB를 저장하는 경우 모두 적합하게 사용 가능하다
4. 순차 증가하므로 단일 Primary Key 생성에 적합하다!

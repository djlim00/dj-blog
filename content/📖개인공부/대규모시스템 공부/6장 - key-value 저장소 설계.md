# key-value 저장소

- 비 관계형 데이터베이스
- 저장되는 값은 무조건 고유 식별자를 가져야함(key).
	- key값은 일반 텍스트 or 해시값 -> 짧을수록 성능 상의 이점이 있음.
- 저장되는 값은(value)
	- 문자열 || 리스트 || 객체
- ex) 아마존 다이나모, memcached, 레디스...2₩

## 문제 이해 및 설계 범위 확정

- key-value 쌍의 크기는 10KB 이하다.
- 큰 데이터 저장할 수 있어야 함.
- 높은 가용성 제공. 설사 시스템에 장애가 있더라도 빨리 응답
- 높은 규모 확장성 제공. 따라서 트래픽 양에 따라 자동적으로 서버 증설/삭제가 이루어져야함.
- 데이터 일관성 수준은 조정이 가능해야함.
- 응답 지연시간(latency)가 짧아야함.

## 분산 key-value 저장소

- a.k.k 분산 해시 테이블
- <font color="#f79646">CAP(Consistency, Availabilty, Partition Tolerance Theorem)</font>을 이해하고 있어야함.

### CAP 정리
- 일관성, 가용성, 파티션 감내 -> <u>모두 만족하는 분산 시스템을 설계하는 것은 불가능하다는 정리</u>

![[Pasted image 20251120175308.png]]
![[Pasted image 20251120175329.png]]

이제 세가지 중에 두가지를 만족하는 경우를 아래와 같이 나눌 수 있다.

- **CP시스템** : 일관성과 파티션 감내를 지원하는 key-value 저장소. 가용성을 희생
- **AP시스템** : 가용성과 파티션 감내를 지원하는 key-value  저장소. 데이터 일관성 희생
- **CA시스템** : 일관성과 가용성을 지원하는 key-value 저장소. 파티션 감내는 희생. 그러나 통상 네트워크 장애는 피할 수 없는 일로 여겨지므로, 분산 시스템은 반드시 파티션 문제를 감내할 수 있도록 설계해야한다. 따라서 <font color="#f79646">CA시스템은 존재하지 않는다. => 장애가 생겼을 때, 동작이 멈추는 시스템은 말이 안됨.</font>


#### 이상적 상태
- 네트워크가 파티션되는 상황이 절대로 일어나지 않는다.
- n1에 기록된 데이터는 자동적으로 n2와 n3에 복제된다. 데이터 일관성과 가용성도 만족된다.
![[Pasted image 20251120175616.png]]

#### 실세계의 분산 시스템
- 파티션 문제를 피할 수 없다(네트워크 장애를 피할 수는 없음).
- 우리는 그렇다면 일관성과 가용성 사이에서 하나를 선택해야한다.
![[Pasted image 20251120175652.png]]
- 그림의 경우, n3에 장애가 발생해서 n1과 n2에 데이터가 전달되지 않음.
	- n1과 n2는 n3에서 전달하지 않은 데이터 즉, 오래된 사본을 가지고 있음.

1. 일관성 선택(CP시스템) 
	- 세 서버 사이에 생길 수 있는 데이터 불일치 문제를 피하기 위해서 n1과 n2에 대해서 쓰기 연산을 중단한다면.. -> 응답을 받을 수가 없기 때문에 <font color="#f79646">가용성이 깨짐</font> 
	- 은행권 시스템은 데이터 일관성을 양보하지 않음.
		- ex) 온라인 뱅킹 시스템에 계좌 최신 정보를 출력하지 못하면 큰 문제임..
		- 네트워크 파티션 때문에 일관성이 깨질 수 있는 상황이 발생하면 -> 이런 시스템은 상황이 해결될 때까지는 오류를 반환해야함.

2. 가용성 선택(AP시스템)
	- 설사 낡은 데이터를 반환할지라도, 계속 읽기 연산을 허용해야함.
	- 아울러 n1과 n2는 계속해서 쓰기 연산을 허용할 것.
		- P문제가 해결된 후에 새 데이터를 n3에 전송함.

<mark class="hltr-yellow">=> 즉 요구사항에 맞게 CAP 정리를 적용해야한다.</mark>

### 2. 데이터 다중화

- 높은 가용성과 안정성을 확보하기 위해 데이터를 N개의 서버에 비동기적으로 다중화(Replication)할 필요가 있음.
- <mark class="hltr-yellow">N은 튜닝이 가능한 값</mark>
	- 어떤 키를 링 위에 배치한 후, 그 지점으로부터 시계 방향으로 링을 순회하면서 만나는 첫 N개 서버에 데이터 사본에 보관하는 것.
	- N=3이라면 아래 그림에서 key0의 value는 s1,s2,s3에 저장된다.
![[Pasted image 20251120180911.png]]
- 가상 노드 -> <font color="#f79646">선택한 N개의 노드가 실제 대응될 실제 물리 서버의 개수가 N개보다 작을 수 있음.(사실상 3개의 가상노드가 2개의 서버일 수도 있으니까)</font>
	- 따라서 중복되는 물리 서버를 선택하면 안된다.
	- 같은 데이터 센터 -> 정전, 네트워크 이슈와 같은 Disaster를 동시에 겪을 수도 있음.
- 다른 네트워크에 분산하고, 센터들은 고속 네트워크로 연결해야함.

### 3. 데이터 일관성

- 여러 노드에 다중화된 데이터는 적절하게 동기화가 되야함.
- <font color="#f79646">정족수 합의(Quorum Consensus) 프로토콜</font>을 사용하면, 읽기/쓰기 연산 모두 일관성을 보장.

> **관계된 정의**
> - N = 사본개수
> - W = 쓰기 연산에 대한 정족수. 쓰기 연산이 성공한 것으로 간주되려면 적어도 W개의 서버로부터 쓰기 연산이 성공했다는 응답을 받아야함.
> - R = 읽기 연산에 대한 정족수. 읽기 연산이 성공한 것으로 간주되려면 적어도 R개의 서버로부터 응답을 받아야함.

#### N=3인 경우의 그림
![[Pasted image 20251120181251.png]]
- <span style="background:#b1ffff">W=1은 데이터가 한 대 서버에만 기록되니다는 것은 아님(적어도 1개)</span>
	-  쓰기 연산이 성공했다고 판단하기 위해서 중재자(coordinator)는 최소 한대 서버로부터 쓰기 성공 응답을 받아야한다는 뜻.
		- <u>여기서 중재자(coordinate)는 클라이언트와 노드 사이의 프록시(proxy)</u>
	- 따라서 s1로 부터 성공 응답을 받았다면 s0,s2의 응답을 기다릴 필요가 없음.

W, R, N의 값을 정하는 것은 "응답지연"과 "일관성" 사이의 타협점을 찾는 전형적인 과정이다.
- W=1, R=1 구성은 응답속도는 빠름
	- W나 R 값이 늘어나면 일관성은 늘어나지만, 응답속도가 느려짐.

- <font color="#f79646">W+R > N 인 경우는 강한 일관성이 보장된다.</font> -> 일관성을 보증할 최신 데이터를 가진 노드가 최소 하나는 겹칠 것이기 때문이다.

### 4. 일관성 모델

1. **강한 일관성** : 모든 읽기 연산은 가장 최근에 갱신된 결과를 반환한다. 다시 말해서 클라이언트는 절대로 낡은 데이터를 보지 못한다.
2. **약한 일관성** : 읽기 연산은 가장 최근에 갱신된 결과를 반환하지 못할 수 있다.
3. **최종 일관성** : 약한 일관성의 한 형태, 갱신 결과가 결국에는 모든 사본에 반영(동기화)되는 모델.

- 강한 일관성 일반적인 방법 -> 모든 사본에 현재 쓰기 연산의 결과가 반영되기 전까지 읽기/쓰기 금지
	- 고가용성 시스템에는 별로.. (<font color="#f79646">요청처리가 중단</font>)
- <mark class="hltr-yellow">다이나모, 카산드라 -> 최종 일관성 모델</mark>
	- 연산이 병렬적으로 발생하면 시스템에 저장된 값의 일관성이 깨질 수 있음. -> <font color="#f79646">클라이언트가 해결해야함.</font>
	- 데이터의 버전 정보를 활용해 일관성이 깨진 데이터를 읽지 않도록 하는 기법을 사용

### 일시적 장애 처리

- <font color="#f79646">엄격한 정족수(Strict quorum)</font> 접근법을 쓰면, 앞서 "데이터 일관성" 절에서 설명한 대로, 읽기와 쓰기를 금지해야함.
- <font color="#f79646">느슨한 정족수(Slopy quorum)</font> 접근법을 쓰면, 이 조건을 완화하여 가용성을 높임.
	- 정족수 요구사항을 강제하지 않고, 쓰기 연산을 수행할 W개의 건강한 서버와 읽기 연산을 수행할 R개의 건강한 서버를 해시 링에서 고름.
	- 이때 장애 서버는 R과 W에서 무시됨.

잠시 다른 서버가 맡아서 처리한다 -> 장애 와중에 발생한 변경사항은 서버가 복구되었을 때 일괄 반영하여 데이터 일관성을 보존.
임시로 쓰기 연산을 처리한 서버에는 그에 대한 단서(hint)를 남긴다. => <mark class="hltr-yellow">임시위탁 기법(hinted handoff)</mark>

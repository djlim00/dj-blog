# 처리율 제한 장치

- 클라이언트 또는 서비스가 보내는 트래픽의 처리율을 제어하기 위한 장치.
	- HTTP에서 특정 기간 내에 전송되는 클라이언트의 요청 횟수 제한.
	- **API 요청 횟수가 임계치(threshold)를 넘어서면**
	  -> <mark class="hltr-yellow">추가도 도달한 모든 호출은 처리가 중단.</mark>

### 처리율 제한의 장점

1. **DoS 공격에 의한 자원고갈 방지 (추가 요청 처리 중단)**
	- 대형 IT기업은 거의 대부분의 API에 어떤 식으로든 처리율 제한이 있음.
		- ex) 트위터 -> 3시간 동안 300개의 트윗만 가능
		- ex) 구글 독스 -> API 사용자당 분당 300회의 read 요청만 허용
2. **비용 절감**
	- 우선순위가 높은 API에 많은 자원 할당
	- 또한 써드파티 API 사용료 지불 방지
		- ex) 신용확인
		- 신용카드 결제
		- 건강 상태 확인
3. **서버 과부화 방지**
	- 봇(bot)에서 오는 트래픽 or 사용자의 잘못된 이용 패턴 거르기

#### 고려사항
1. 프로그래밍 언어, 캐시 서비스 등 현재 사용하는 기술 스택을 점검하라. 현재 사용하는 기술스택이 서버 측 구현을 충분히 지원할 정도로 효율적인가?
2. 사업에 맞는 처리율 제한 알고리즘을 찾아라. *서버 측에서 구현하기로 했다면, 알고리즘은 자유롭게 선택할 수 있음.* 그러나 *제3 사업자가 제공하는 게이트웨이를 사용한다면 선택지는 제한될 수 있다.*
3. 현재 MSA에 기반하고 있고, 사용자 인증이나 IP허용목록 관리 등을 처리하기 위해 API 게이트웨이를 이미 설계에 포함시켰다면, 처리율 제한 기능 또한 게이트웨이에 포함시켜야 할 수도 있다. -> 현재의 구조를 생각하자.
4. 직접 만드는 것은 시간(돈)! 충분한 인력이 없다면 상용 API 게이트웨이를 사용하자.

## 처리율 제한 알고리즘의 종류

#### 종류
1. 토큰 버킷 
2. 누출 버킷
3. 고정 윈도 카운터
4. 이동 윈도 로그
5. 이동 윈도 카운터

### 토큰 버킷 알고리즘
- 보편적, 많은 이용 ex) 아마존, 스트라이프

![[Pasted image 20251106140602.png|500]]
![[Pasted image 20251106140852.png|500]]
- 한마디로, 주기적으로 채워지는 API 요청 가능 토큰이 있고, 너무 많은 요청이 들어오면 토큰이 없기 때문에 해당 요청을 받지 않는다.

> 해당 알고리즘은 2개의 parameter를 받는다
> 1. 버킷 크기 : 버킷에 담을 수 있는 토큰의 최대 개수
> 2. 토큰 공급률 : 초당 몇 개의 토큰이 공급되는가
>    
> - **통상적으로 API 엔드포인트마다 별도의 버킷을 둔다**
ex) 우리의 예시는 "포스팅", "친구 추가", "좋아요"가 있으니까 버킷이 3개다.
ex) IP주소별로 처리율 제한을 적용해야한다면, IP 주소마다 버킷을 하나
ex) 시스템 처리율 10000개 -> 모든 요청이 하나의 버킷을 공유해야함.

#### 장점
1. 구현이 쉬움
2. 메모리 사용이 효율적
3. 짧은 시간에 집중되는 트래픽도 처리 가능 -> 남은 토큰이 있기만 하면 요청을 시스템에 전달
#### 단점
1. 버킷 크기와 토큰 공급률 -> 튜닝이 어려움

### 누출 버킷 알고리즘
- 요청 처리율이 고정되어 있기 때문에 다르다.
- FIFO 큐로 구현
> 1. 큐에 빈자리가 있으면 큐에 요청을 추가
> 2. 큐가 가득 차 있는 경우 새 요청을 버림
> 3. 지정된 시간마다 큐에서 요청을 꺼내서 처리
![[Pasted image 20251106142103.png|500]]

> 두개의 parameter
> 1. 버킷 크기 : 큐 사이즈와 같은 값. 큐에는 처리될 항목들 보관
> 2. 처리율 : 지정된 시간당 몇 개의 항목을 처리할 것인가? 보통 초단위로 표현

#### 장점
1. 큐의 크기가 제한 -> 메모리 사용량 효율적
2. 고정된 처리율 -> 안정적 출력이 필요한 경우 적합
#### 단점
1. 단기간에 많은 트래픽이 몰리면 -> 큐에 오래된 요청이 쌓임
2. 요청들이 제때 처리 못하면 최신 요청들이 버려짐
3. 두개의 인자 -> 튜닝 어려움

### 고정 윈도 카운터 알고리즘

> 1. 타임라인을 고정된 간격의 윈도(window)로 나누고, 각 윈도마다 카운ㅌ(counter)를 붙임
> 2. 요청이 접수될 떄 마다 이 카운터의 값이 1증가
> 3. 카운터의 값이 임계치에 도달하면 새로운 요청을 윈도가 열릴 때까지 버려짐.

![[Pasted image 20251106142513.png|500]]

😡 <span style="background:#ff4d4f">가장 큰 문제는 윈도의 경계 부근에 순간적으로 많은 트래픽이 집중되면, 윈도에 할당된 양보다 더 많은 요청이 처리될 수 있음.</span>
![[Pasted image 20251106142633.png]]
- 분당 최대 5개의 요청만 허용 (카운터가 매분 초기화)
- 2:00:00과 2:01:00 사이에 5개의 요청이 들어오고, 2:00:01과 2:00:02사이에 5개가 들어오게 되면 -> 사실상 분당 10개를 처리한 셈이다.

#### 장점
1. 메모리 효율 좋음
2. 이해하기 쉬움
3. 윈도가 닫히는 시점에 카운터 초기화 -> <font color="#00b0f0">특정한 트래픽 패턴을 처리하게 적합</font>
#### 단점
1. 윈도 부근에 일시적으로 많은 트래픽 -> 처리한도보다 더 많은 양의 요청을 처리

### 이동 윈도 로깅 알고리즘

- 앞선 "고정 윈도 카운터 알고리즘"의 문제를 해결

> 1. 타임 스탬프를 추적 Redis의 정렬집합(sorted set) 같은 캐시에 보관
> 2. 새 요청이 오면 만료된 타임스탬프 제거(만료된 스탬프는 그 값이 현재 윈도의 시작 시점보다 오래된 타임스탬프를 말함)
> 3. 새 요청의 타임스탬프를 로그(log)에 추가
> 4. 로그의 크기가 허용치보다 같거나, 작으면 요청을 시스템에 전달
> 5. 허용치보다 크면 처리를 거부

![[Pasted image 20251106144053.png|500]]
![[Pasted image 20251106145431.png]]
- 작업들을 큐에 넣어두고 일정 시간동안 처리할 수 있는 양만 처리하고, 고정된 시간이 아니라, 범위를 가지고 와서 해당 범위 안에 있으면 처리된다.

#### 장점
1. 아주 정교하다 -> 어느 순간의 윈도를 봐도, 허용되는 요청의 개수는 처리율 한도를 넘지 않음
#### 단점
1. 다량의 메모리 사용(TIMESTAMP), 거부된 요청의 타임스탬프도 보관해야하니까,,

### 이동 윈도 카운터 알고리즘

- *고정 윈도 카운터 알고리즘+이동 윈도 로깅 알고리즘*
- 두 가지의 구현 접근법이 있음 -> 하나만 책에서 설명

![[Pasted image 20251106145818.png]]
- **처리율 한도가 분당 7개, 이전 1분 동안 5개의 요청, 그리고 현재 1분 동안 3개의 요청이 옴**
- 현재 1분의 30% 시점에 도착한 새 요청의 경우, 현재 윈도에 몇 개의 요청이 온 것으로 보고 처리해야할까..?

> 1. 현재 1분간의 요청수 + 직전 1분간의 요청 수 x 이동 윈도와 직전 1분이 겹치는 비율
> 2. 현재 윈도에 들어있는 요청 3+5x70% = 6.5개 (반올림 or *내림*) = 6
> 3. 따라서 한도가 7개니까, 이번 요청을 시스템으로 전달(이 다음부터는 안됨)

#### 장점
1. 이전 시간대의 평균 처리율에 따라 현재 윈도의 상태를 계산 -> 짧은 시간에 몰리는 트래픽에도 잘 대응
2. 메모리 효율이 좋음
#### 단점
1. 직전 시간대에 도착한 요청이 균등하게 분포되어 있다고 가정함 -> 다소 느슨함
2. 그러나 생각보다 문제가 심각하진 않음
	- CloudFlare가 실시한 실험 -> 40억개의 요청 가운데 시스템의 실제 상태와 맞지 안헥 버려진 요청은 0.003%에 불과함.(12000개긴하네..)

### 개략적인 아키텍처

- 알고리즘의 아이디어는 단순 -> 카운터를 추적 대상별로 두고(사용자별? IP주소별? API엔드포인트나 서비스별?) -> 임계치를 넘으면 거부

	- Redis

		- <mark class="hltr-yellow">EXPIRE</mark> : 카운터의 타임아웃 값 설정. 설정 시간이 지나면 자동 삭제

> 1. 클라이언트가 처리율 제한 미들웨어에게 요청을 보냄

### 분산 환경에서의 처리율 제한 장치의 구현

- 단일 서버면 구현이 어렵지 않음.
- But 여러 대의 서버와 병렬 스레드를 지원하도록 확장하는 것은 어려움

> 1. 경쟁 조건(race condition)
> 2. 동기화(synchronization)

#### 1. 경쟁조건(Race Condition)

처리율 제한 장치의 동작
1. Redis에 카운터의 값을 읽음(counter)
2. counter+1의 값이 임계치를 넘는지 확인
3. 넘지 않으면 Redis에 보관한 카운터 값+1

*병행성이 심하면 아래 그림과 같은 race condition 문제가 발생함*

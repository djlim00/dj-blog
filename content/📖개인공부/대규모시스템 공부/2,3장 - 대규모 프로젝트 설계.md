- 데이터 양이 엄청나게 커지지만 계산법은 기본에서 크게 벗어나지 않음

#### CPU 및 메모리 계층
| 구성요소                                | 시간     | 설명                                           |
| ----------------------------------- | ------ | -------------------------------------------- |
| **L1 캐시 접근**                        | ≈ 1ns  | CPU 내부에 있는 초고속 메모리. 자주 쓰는 데이터를 저장.           |
| **분기 예측 오류 (Branch misprediction)** | ≈ 3ns  | CPU가 if문 등 분기 예측을 틀렸을 때 파이프라인을 비워야 하는 시간 손실. |
| **L2 캐시 접근**                        | ≈ 4ns  | L1보다 조금 느린 캐시. 수 MB 단위.                      |
| **뮤텍스 잠금/해제 (Lock/Unlock)**         | ≈ 17ns | 스레드 동기화를 위한 최소 비용.                           

#### 메모리(디스크) 및 스토리지 계층
| 구성요소                   | 시간                    | 설명                                   |
| ---------------------- | --------------------- | ------------------------------------ |
| **주 메모리 접근 (DRAM)**    | ≈ 100ns               | CPU가 RAM에서 직접 읽을 때. CPU에 비해 100배 느림. |
| **SSD 내 임의 위치 데이터 읽기** | ≈ 16,000ns = 16μs     | 플래시 셀 접근 시간. 빠르지만 RAM보단 100배 이상 느림.  |
| **SSD에서 1MB 순차 읽기**    | ≈ 49,000ns = 49μs     | 순차 읽기는 임의 접근보다 빠름.                   |
| **디스크 탐색 (seek)**      | ≈ 2,000,000ns = 2ms   | HDD 헤드 이동 시간. SSD보다 훨씬 느림.           |
| **디스크에서 1MB 순차 읽기**    | ≈ 825,000ns = 0.825ms | HDD의 연속적 블록 읽기.                      |
| **메모리에서 1MB 순차 읽기**    | ≈ 3,000ns = 3μs       | RAM은 대용량 읽기도 빠름.                     |

#### 네트워크 및 시스템 통신
| 동작                       | 시간                      | 설명                        |
| ------------------------ | ----------------------- | ------------------------- |
| **동일 데이터센터 내 패킷 왕복 지연**  | ≈ 500,000ns = 0.5ms     | 서버 간 통신. 마이크로서비스 간 호출 수준. |
| **장거리 네트워크 (CA ↔ 네덜란드)** | ≈ 150,000,000ns = 150ms | 대륙 간 왕복 (인터넷 수준).         |

### 결론
1. 메모리는 빠르지만, 디스크는 아직도 느리다
2. 디스크 탐색(seek)은 가능한 피해라
3. 단순한 압축 알고리즘은 빠르다(Zippy)
4. 데이터를 인터넷 전송하기 전에 가능한 압축하라
5. 데이터 센터는 보통 여러 지역에 분산 -> 센터 간에 데이터 송수신은 시간이 걸린다.


## 3. 가용성과 관계된 수치

- **고가용성 시스템** : 오랜 기간 동안 지속적으로 중단 없이 운영될 수 있는 능력
	- 100%는 한번도 중단된 적이 없음(대부분 99%~100%)
- **SLA(Service LEvel Agreement)** : 서비스 사업자가 보편적으로 사용하는 용어, 서비스 사업자와 고객 사이에 맺어진 합의
	- 서비스 사업자가 제공하는 서비스의 가용기간(uptime)이 공식적으로 기술되어 있음
	- 아마존, 구글, MS는 99% 이상의 SLA를 제공
	- 보통 99.999 이런식으로 표현.

## 4. 예제 : 트위어 QPS와 저장소 요구량 추정

#### 가정
1. 월간 능동 사용자는 3억(300million)명
2. 50%의 사용자가 트위터를 매일 사용
3. 평균적으로 각 사용자는 매일 2건의 트윗을 올림
4. 미디어를 포함하는 트윗은 10%
5. 데이터는 5년간 보관

#### 추정
1. QPS(Query Per Second) 추정치
	- 일간 능동 사용자 : 3억 * 50% = 1.5억
	- <span style="background:#b1ffff">QPS = 1.5억 * 2트윗 / 24시간 / 3600초 = 약 3500</span>
	- 최대 QPS(Peek QPS) = 2 * QPS = 약 7000
2. 미디어 저장을 위한 저장소 요구량
	- 평균 트윗크기
		- tweet_id에 64바이트(무시 너무 작음)
		- 텍스트에 140바이트(무시 너무 작음)
		- 미디어에 1MB
	- 미디어 저장소 요구량 : 1.5억 * 2 * 10% * 1MB = 30TB/일
	- 5년간 미디어 저장 -> 30TB * 365 * 5 = 55PB

### Tip.
- 문제를 풀어나가는 절차 -> 올바른가? (문제해결 능력을 보고 싶은 것이다)
- **근사치를 활용하라**
	- 복잡한 계산은 무리다(시간낭비) -> 999877/9.1 이런건 그냥 1000000/10 이렇게 가면 된다.
- **가정(assumption)을 적어두자**
- **단위를 붙이자**
	- 5라고만 적으면 5KB인지, 5MB인지 알 수 없음. 단위를 붙이는 습관이 모호함을 없엔다.
- **주로 QPS, 최대 QPS, 저장소 요구량, 캐시 요구량, 서버 수를 추정하는 것.**

# 3장. 시스템 설계 면접 공략법

- 면접에서 너무 모호한 질문이 올 수 있다 -> ex) 유튜브를 설계해보라
	- 그렇지만 이 모든 것을 대답하라는 것은 아니다!
- <span style="background:#b1ffff">모호한 문제를 풀기 위해 협력하고, 그 해결책을 찾아내는 과정에 대한 시뮬레이션</span>
	- 설계 기술을 시연
	- 방어능력
	- 면접관의 피드백을 건설적인 방식으로 처리

1. 기술적인 측면 평가
2. 협력에 적합한 사람인지
3. 압박이 심한 상황도 잘 헤처 나가는 자질이 있는지
4. 모호한 문제를 건설적으로 해결할 능력이 있는지

- red flag도 놓치지 않는다
	- 설계의 순수성에 집착해서 나머지 타협적 결정을 무시하고 오버 엔지니어링
		- 시스템 전반의 비용이 올라감
	- 완고함, 편협함

## 효과적 면접을 위한 4단계 접근법

### 1. 문제 이해 및 설계 범위 확장
- 생각 없이 바로 답을 내는 것은 좋지 않다.
- 요구사항을 완벽히 이해하지 않고, 답을 내놓은 행위는 아주 부정적 -> 정답이 없기 때문
- <span style="background:#b1ffff">깊이 생각하고, 요구사항과 가정들을 분명히 하라.</span>
	- 올바른 질문, 적절한 가정, 시스템 구축을 위한 정보 수집

1. 구체적으로 어떤 기능을 만들어야 하나?
2. 제품 사용자 수는 얼마나 되나?
3. 회사의 규모는 얼마나 빨리 커질까? 3달, 6개월? 일년 뒤의 회사의 규모는?
4. 회사 주로 사용하는 기술 스택은 무엇인가? 설계를 단순화하기 위한 기존의 서비스는 뭐가 있을까?

#### 예제 : 뉴스 피드 시스템 설계

> <font color="#00b0f0">지원자</font> : 모바일 앱과 웹 중 어느 쪽을 지원해야하나? 아니면 둘 다 일까요?
> 면접관 : 둘 다 입니다.
> <font color="#00b0f0">지원자</font> : 가장 중요한 기능은 무엇인가요?
> 면접관 : 새로운 post를 올리고, 다른 친구의 뉴스 피드를 볼 수 있도록 하는 기능입니다.
> <font color="#00b0f0">지원자</font> : 이 뉴스의 피드는 역순으로 정렬인가요? 아니면 다른 특별한 정렬 기준이 있습니까? 제가 특별한 정렬 기준이 있느냐고 묻는 이유는, 피드에 올라갈 포스트마다 다른 가중치가 부여되야하는지 알고 싶어서 인데요, 가령 가까운 친구의 포스트가 사용자 그룹(user group)에 올라가는 포스트보다 더 중요하다거나.
> 면접관 : 문제를 단순하게 만들기 위해, 일단 시간 역순으로 정렬된다고 가정합시다.
> <font color="#00b0f0">지원자</font> : 한 사용자는 최대 몇 명의 사용자와 친구를 맺을 수 있나요?
> 면접관 : 5000명입니다.
> <font color="#00b0f0">지원자</font> : 사이트로 오는 트래픽 규모는 어느 정도 입니까?
> 면접관 : 일간 능동 사용자(DAU)는 천만 명입니다.
> <font color="#00b0f0">지원자</font> : 피드에 이미지나 비디오도 올라 올 수 있나요? 아니면 포스트는 그저 텍스트입니까?
> 면접관 : 이미지나 비디오 같은 미디어 파일도 포스트 할 수 있어야 합니다.

-> <span style="background:#b1ffff">다음과 같이 요구사항을 이해하고, 모호함을 없에는게 이 단계에서 가장 중요함</span>

### 2. 개략적인 설계안 제시 및 동의 구하기

1. 설계안에 대한 최초 청사진을 제시하고, 의견을 구하라. 
	- 면접관을 마치 팀원인 것 처럼 대하라. 훌륭한 면접관들은 지원자들과 대화하고 설계 과정에서 개입하기를 즐긴다.
2. 화이트보드나 종이에 핵심 컴포넌트를 포함하는 <mark class="hltr-yellow">다이어그램</mark>
	- 클라이언트(모바일/웹), API, 웹 서버, 데이터 저장소, 캐시,  CDN, 메시지 큐 같은 것들이 포함될 수 있다.
3. 최초 설계안이 제약사항을 만족하는가? (계산을 해보자)
	- 계산의 과정을 말로 설명 -> 추정이 필요하다면, 면접관에게 물어보기
	- 시스템의 구체적인 사용 사례도 몇 가지 살펴보자.
		- 에지케이스 발견
4. API 엔드 포인트나 DB 스키마 같은 경우는 너무 큰 서비스는 생략, 작은 서비스라면 괜찮음.

#### 예제 : 뉴스 피드 시스템 설계

뉴스 피드 시스템을 두가지 flow로 생각해보자.

1. **피드 발행** : 사용자가 포스트를 올리면 관련 데이터가 cache/DB에 기록되고, 해당 사용자의 친구(friend) 뉴스 피드에 뜨게 된다.
2. **피드 생성** : 어떤 사용자의 뉴스 피드는 해당 사용자 친구들의 포스트 시간 역순 정렬하여 만든다.

![[Pasted image 20251030164257.png]]
- 피드 발행은 write(상대적으로 적음)이고 피드 생성은 read(잦음)기 때문에 둘을 분리해야함.

### 3. 상세 설계

1-2단계를 거치면서 다음 목표가 달성되어 있을 것이다.

- **시스템에서 전반적으로 달성해야 할 목표와 기능 범위 확인**
- **전체 설계의 개략적 청사진 마련**
- **해당 청사진에 대한 면접관의 의견 청취**
- **상세 설계에서 집중해야 할 영역들 확인**

1. 설계 컴포넌트의 우선순위를 정하자(지정될 수도)
	- 선입금 개발자 면접 : 성능 특성 -> 병목 구간, 자원 요구량 추정치
	- 대부분 : 각각의 컴포넌트에 대해서 세부사항을 깊이 있게 설명
ex) 단축 URL 생성기
- 해시 함수 설계를 구체적으로
ex) 채팅 시스템
- 지연시간 줄이기, 온/오프라인 상태 표시

1. 불필요한 세부사항 때문에 좋은 시그널을 줄 기회를 날리지 말자.
ex) Facebook의 뉴스 피드 순위 EdgeRank 알고리즘을 상세히 설명한다던지..

### 4. 마무리

후속 질문이 들어올 수 있고, 스스로 추가 논의를 진행할 수도 있음

- 시스템 병목구간, 혹은 개선 가능한 지점을 찾아라
	- 무조건 개선할 점이 있음 -> 비판적 사고 능력을 보이자
- 다시 한번 요약한다.(환기 효과)
- 오류가 발생하면 어떤 오류? (서버 오휴, 네트워크 장애)
- 운영 이슈(매트릭 수집, 모니터링, 로깅, 배포)
- 미래의 확장 요구 -> 백만에서 천만의 사용자를 감당하기
- 미처 다루진 못한 개선사항

#### 해야할 것
1. 질문을 통한 확인. 나의 가정이 무조건 옳지 않다!
2. 문제의 요구사항 이해
3. 정답이나 최선은 없다. 
	- 스타트업과 중견기업의 설계안이 같을 수 없다.
4. 면접관이 사고 흐름을 이해할 수 있도록 하라(소통)
5. 여러 해법은 함께 제시하라
6. 개략적 설계에 동의한다면, 컴포넌트의 세부사항을 설명하라. 
	- 항상 제일 중요한 컴포넌트부터 진행하라.
7. 면접 관의 아이디어를 이끌어라.
8. 포기하지 말라.

#### 하지 말하야 할 것
1. 전형적인 면접 문제도 대비하지 않고 가지 말라.
2. <span style="background:#b1ffff">요구사항이나 가정들을 확실하게 하지 않고 설계를 제시하지 말라.</span>
3. 처음부터 특정 컴포넌트를 너무 자세히 설명하지 마라

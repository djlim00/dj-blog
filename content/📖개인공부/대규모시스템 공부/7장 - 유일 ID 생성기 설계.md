#### 황영사 금동불상

그냥 DB의 <span style="background:#b1ffff">auto_increment</span>를 사용하면 되는거 아닌가?

-> 분산 환경에서는 DB 서버 한대로는 그 요구를 감당할 수 없음
-> 여러 DB를 사용하면 delay가 발생

___
## 1단계. 문제 이해 및 설계 범위 확정

1. 적절한 질문을 통해 모호함을 없에고 설계 방향을 정해 나가자.

![[Pasted image 20251127105444.png]]

> **요구사항**
> 1. ID는 유일해야 함.
> 2. ID는 숫자로만 구성.
> 3. ID는 64비트로 표현할 수 있는 값.
> 4. ID는 발급 날짜에 따라 정렬 가능해야함.
> 5. 초당 10,000개의 ID를 만들 수 있어야 함.

___
## 2단계. 개략적 설계안 제시 및 동의 구하기

> 1. 다중 마스터 복제(muti-master replication)
> 2. UUID(Universally Unique Identifier)
> 3. 티켓 서버(ticket server)
> 4. 트위터 스노플레이크(twitter snowflake) 접근법

### 1. 다중 마스터 복제

![[Pasted image 20251127135005.png]]
- <span style="background:#b1ffff">DB의 auto_increment 기능을 활용!</span>
	- 다음 ID의 값을 구할 떄, 1만큼 증가가 아니라 <font color="#ffc000">k만큼 증가</font>시킨다.
- k = 현재 사용 중인 데이터베이스 서버의 수
	- 그림 7-2에서 서버가 k=2
	- 따라서 1->3->5 되는 것을 볼 수 있다.

#### 장점
1. 규모 확장성 문제를 어느 정도 해결 가능.
2. DB 수를 늘리면, 초당 생산 가능ID 수도 늘릴 수 있음.

#### 매우 중대한 단점
1. 여러 데이터 센터에 걸쳐 규모를 늘리기 어려움.
2. ID의 유일성은 보장 -> 그 값이 시간 흐름에 맞추어 커지도록 보장할 수 없음
	- 1->3->5에서 7을 추가했는데, 그거보다 훨씬 빨리 2->4->6->8 해버리면 8이 7보다 먼저 생성되는 상황이 된다.
3. 서버를 추가하거나 삭제할 때도 동작하게 만들기 어려움.

___
### 2. UUID

- 컴퓨터가 시스템에 저장되는 정보를 유일하게 식별하기 위한 <font color="#ffc000">128비트 수</font>
	- 충돌 가능성이 지극히 낮음. 
	- 서버와 독립적으로 생성 가능

![[Pasted image 20251127135640.png]]
- 웹 서버는 별도의 ID 생성기를 이용해 독립적으로 ID를 만든다.

#### 장점
1. UUID를 만드는 것은 매우 단순함. 서버 사이의 조율이 필요 없음(<u>동기화 필요없음</u>)
2. 각 서버가 자기가 쓸 ID를 알아서 만드는 구조 -> <u>규모 확장 쉬움</u>

#### 단점
1. ID가 128비트로 김(우리의 요구사항은 64비트)
2. <font color="#ffc000">ID를 시간순으로 정렬 불가능</font>
3. ID에 숫자가 아닌 값이 포함

___
### 3. 티켓 서버(ticket server)

- 플리커(Flicker)는 분산 기본 키(distributed primary key)를 만들기 위해 티켓 서버를 사용함.

![[Pasted image 20251127140407.png]]
- <font color="#ffc000">핵심은 auto_increment 기능을 갖춘 DB서버, 즉 티켓 서버를 중앙 집중형으로 하나만 사용함.</font>

#### 장점
1. 유일성이 보장되는 오직 숫자로만 구성된 ID를 쉽게 만들 수 이음.
2. 구현하기 쉽고, 중소 규모 애플리케이션에 적합함.
#### 단점
1. 티켓 서버가 SPOF(Single-Point-of-Failure)가 된다.
	- 서버에 장애가 발생하면, 모든 시스템이 영향 받음
	- 이걸 방지하게 위해 N개의 티켓 서버를 두면, 또 동기화 문제가 발생.

### 4. 트위터 스노플레이크 접근법

- 먼저 <span style="background:#b1ffff">Divide and Conquer</span>를 적용해보자.
	- 생성해야하는 ID의 구조를 여러 절(section)으로 분할

###### 64비트 ID의 구조
![[Pasted image 20251127141419.png]]
> 1. **sign 비트** : <font color="#ffc000">1비트</font> 할당. 지금은 쓰임새가 없지만, 나중을 위해 유보. 음수/양수 판단.
>    
> 2. **타임스탬프** : <font color="#ffc000">41비트</font> 할당. 기원 시각 이후로 몇 밀리초가 경과했는지를 나타내는 값. 본 설계안의 경우에는 기원 시각으로 트위터 스노플레이크 구현에 사용하는 값  1288834974657(Nov 04, 2010, 01:42:54 UTC에 해당)
>    
> 3. **데이터 센터 ID** : <font color="#ffc000">5비트</font> 할당, 따라서 32개의 데이터 센터 지원 가능
>    
> 4. **서버 ID** : <font color="#ffc000">5비트</font> 할당, 데이터 센터당 32개의 서버 사용 가능.
>    
> 5. **일련번호** : <font color="#ffc000">12</font>비트 할당, 각 서버에서는 ID를 생성할 때마다, 이 일련번호를 1만큼 증가시킨다. 이 값은 1밀리초가 경과할 때 마다 0으로 초기화 된다. (안그러면 12비트가 오버플로우 날 수 있음)

## 3단계. 상세 설계

트위터의 스노플레이크 접근법을 사용하여 보다 상세한 설계를 해보자.

![[Pasted image 20251127142642.png]]
- 데이터센터 ID와 서버 ID는 시스템이 시작할 때 결정되며, 일반적으로 시스템 운영 중에는 바뀌지 않는다. 
- 데이터센터 ID나 서버 ID를 잘못변경하게 되면, ID 충돌이 발생할 수 있으므로, 그런 작업을 할 때는 신주해야함.
- 타임스탬프나 일련번호는 ID생성기가 돌고 있는 중에 만들어지는 값이다.

### 타임스탬프

- <font color="#ffc000">가장 중요한 41비트</font>
- 시간이 흐름에 따라 점점 큰 값을 갖게 되므로(기원 시각으로부터 지난 얼마나 지났는지이므로), 결국 ID는 시간 순으로 정렬 가능함.
![[Pasted image 20251127143018.png]]
- 그림 7-7은 앞서 살펴본 ID 구졸ㄹ 따르는 값의 이진 표현 형태로부터 UTC 시각을 추출하는 예제.(41비트 이진수 -> 십진수 -> 기원시각에 더해서 시간 구하기)
	- 역으로 하면 41비트를 만들 수 있음.

#### 최대 시간
![[Pasted image 20251127143148.png]]

### 일련번호
- <font color="#ffc000">12비트</font>

## 4단계. 마무리

추가적으로 논의할 부분들이 있다.

### 1. 시계 동기화(Clock Synchronization)
ID 생성 서버가 전부 같은 시계를 사용한다고 가정했음.
하지만 이런 사정은 하나의 서버가 여러 코어에서 실행될 경우 유효하지 않을 수 있음.
여러 서버가 물리적으로 독립된 여러 장비에서 실행되는 경우에도 마찬가지.
- NTP(Network Time Protocol)은 이 문제를 해결하는 가장 보편적인 수단.

### 2. 각 절(section)의 길이 최적화
동시성이 낮고, 수명이 긴 애플리케이션이라면, 일련번호 절의 길이를 줄기ㅗ, 타임 스탬프 절의 길이를 늘리는 것이 효과적일 수도 있음.

### 3. 고가용성
ID 생성기는 필수 불가결(mission critical) 컴포넌트이므로 아주 높은 가용성을 제공해야한다.

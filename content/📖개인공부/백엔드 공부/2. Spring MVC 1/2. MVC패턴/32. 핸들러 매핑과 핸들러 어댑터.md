# 핸들러 매핑 & 핸들러 어댑터

## Controller 인터페이스
"과거 버전의 스프링 컨트롤러"
```java title:"과거의 스프링 컨트롤러 인터페이스"
public interface Controller {
ModelAndView handleRequest(HttpServletRequest request, HttpServletResponse
response) throws Exception;
}
```
- ModelAndView를 반환하는 handleRequest()메서드가 있음!


>참고
>Controller 인터페이스는 @Controller 애노테이션과는 전혀 다르다.

```java title:"직접 만들어보는 옛식"
@Component("/springmvc/old-controller")  
public class OldContrloller implements Controller {  
    @Override  
    public ModelAndView handleRequest(HttpServletRequest request, HttpServletResponse response) throws Exception {  
        System.out.println("OldController.handleRequest");  
        return null;    }  
}
```
- @Component로 빈 등록과 url을 등록해준다.
- <span style="background:#fff88f">빈의 이름으로 URL을 매핑</span>할 것이다.


### 우리가 구현한 Controller를 어떻게 찾을까?

1. <span style="background:#fff88f">핸들러 매핑</span>
	- 핸들러 매핑에서 우리가 등록한 OldController를 찾을 수 있어야 한다.
	- ex) 우리가 등록한 이름으로 핸들러를 찾을 수 있는 그러한 핸들러 매핑이 필요함
2. <span style="background:#fff88f">핸들러 어댑터</span>
	- 핸들러 매칭을 통해 찾은 핸들러를 실행할 수 있는 핸들러 어댑터가 필요함
	- ex) Controller인터페이스를 실행할 수 있는 핸들러 어댑터를 <u>찾고</u> 실행해야함

스프링은 이미 필요한 핸들러 매핑과 핸들러 어댑터를 대부분 구현해두었다. 개발자가 직접 핸들러 매핑과 핸들러 어댑터를 만드는 일은 거의 없다.

## HttpRequestHandler의 경우
___

![[Pasted image 20250225223246.png]]
- return이 void -> 거의 순수한 servlet이랑 다를 바가 없음

![[Pasted image 20250225224109.png]]
- 먼저 핸들러매핑에 의해서 핸들러를 찾는다
	- 지금의 경우는 빈 이름으로 핸들러를 찾기 때문에, BeanNameUrlHandlerMapping이 실행이 됨
	- 우리가 @Component("/springmvc/request-handler")로 빈을 등록 했기 때문에 찾아짐 -> 찾은 핸들러인 MyHttpRequestHandler를 반환
- 이제 핸들러 어댑터를 찾아야함
	- HandlerAdapter의 surrport()가 순서대로 호출됨
	- HttpRequestHandlerAdpater가 HttpRequestHandler를 지원하기 때문에 잡힘(우리가 만든 MyHttpRequestHandler는 HttpRequestHandler를 구현했으므로)
- DispatcherServlet이 지원하는 핸들러 어댑터인 HttpRequestHandlerAdapter가 실행되면서 내가 만든 핸들러 정보도 넘어감
	- MyHttpRequestHandler가 실행이 됨

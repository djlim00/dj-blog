- 모든 데이터는 HTTP를 기반으로 주고 받는다.
	- HTML, TEXT
	- IMAGE, 음성, 영상, 파일
	- JSON, XML(API)
	- 거의 모든 형태의 데이터 전송
	- 심지어 서버간의 데이터 전송도 HTTP

___
# 1.  웹 서버

- HTTP 기반으로 동작
- <font color="#ffff00">정적 리소스 제공</font>, 기타 부가 기능
- 정적(파일) HTML, CSS<,JS, 이미지, 영상
	- ex) NGINX, APACHE

___
# 2. 웹 애플리케이션 서버(WAS)

- HTTP 기반으로 동작
- 웹서버의 기능 포함
- <font color="#ffff00">프로그램 코드를 실행해서 애플리케이션 로직 수행</font>
	- 동적 HTML, HTTP API(JSON)
	- 서블릿, JSP, 스프링 MVC
	- ex) 톰캣(Tomcat), Jetty, Undertow

___
# 3. 웹 서버, WAS 비교


- 웹 서버는 정정 리소스, WAS는 애플리케이션 로직
- 사실 둘의 용어도 경계도 <span style="background:#d4b106">모호</span>함
	- 웹 서버도 프로그램 실행 기능을 포함하기도 함
	- 웹 애플리케이션 서버도 웹 서버의 기능을 제공함
- <font color="#ffff00">자바에선  서블릿 컨테이너 기능을 제공하면 WAS라고 함</font>
	- 서블릿 없이 자바코드를 실행하는 서버 프레임워크도 있음
	- <u>WAS는 애플리케이션 코드를 실행하는데 더 특화</u>

___
# 4.  웹 시스템 구성 - WAS, DB만 있을 때,,

- WAS, DB 만으로 시스템 구성 가능
- WAS는 정적 리소스, 애플리이케이션 로직 모두 제공 가능

-> 가장 비싼 애플리케이션 로직이 정적 리소스 때문에 수행이 어려울 수 있음
-> WAS 장애 시 <u>오류 화면도 노출 불가능</u>

___
# 5. 웹 시스템 구성 - WEB, WAS, DV

- 정적 리소스 처리는 웹 서버
- 애플리케이션 로직 같은 동적인 처리는 WAS에 요청 위임
	- <font color="#ffff00">WAS는 중요한 애플리케이션 로직 처리만 전담</font>

![[Pasted image 20250104181124.png]]

- 효율적인 리소스 관리
	- 정적 리소스가 많은 호출? -> Web서버 증설
	- 애플리케이션 리소스가 많은 호출? -> WAS 증설
![[Pasted image 20250104181254.png]]

- 정적 리소스만 제공하는 웹 서버는 잘 안죽음 -> 로직 처리할게 별로 없음
- WAS는 잘 죽음 ->  복잡한 애플리케이션 로직이 동작하므로
- WAS, DB 장애 시<u> WEB 서버가 오류 화면 제공 가능</u>

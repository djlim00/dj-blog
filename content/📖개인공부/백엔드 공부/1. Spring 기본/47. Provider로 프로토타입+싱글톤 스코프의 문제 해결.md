
## 1. 싱글톤 빈이 프로토타입을 사용할 때 마다 스프링 컨테이너에 새로운 요청하기 - 안좋은 방법
![[Pasted image 20241229142504.png]]
- ClientBean을 생성할 때, ApplicationContext를 선언하고
- logic()을 호출할 때 마다, PrototypeBean을 생성함(getBean())
  -> <font color="#ffff00">DL(Depedency Lookup), 의존관계 조회</font>

- 그럼 위의 DL정도의 역할을 하는게 ApplicationContext에 없을까? -> <font color="#ffff00">Provider</font>
___
## 2. ObjectFactory, ObjectProvider

```java title:"ObjectProvider를 이용한 방식", hl:5,8
@Scope("singleton")  
static class ClientBean{  
  
    @Autowired  
    private ObjectProvider<PrototypeBean> prototypeBeanObjectProvider;  
  
    public int logic(){  
        PrototypeBean prototypeBean = prototypeBeanObjectProvider.getObject();  
        prototypeBean.addCount();  
        int count = prototypeBean.getCount();  
        return count;  
    }  
}
```
- getObject를 호출하면 스프링 컨테이너에서 찾아서 반환해준다.(ApplicationContext까지 사용하지 않아도 됨)
- Factory와 Provider의 차이
	- Provider가 Factory 인터페이스 상속을 받고 있음 -> 편의성 기능이 좀 추가됨
#### 장점
1. 스프링이 제공하는 기능이지만, 기능이 단순하므로(AnnotationConfigApplicationContext보다 단순하기 때문에) 유닛테스트나, mock코드를 만들기 훨씬 쉬워진다.
2. 딱 필요한 DL정도의 기능을 제공한다
#### 특징
1. ObjectFactory : 기능이 단순, 라이브러리 필요없음, <font color="#ffff00">스프링에 의존</font>
2. ObjectProvider : ObjectFactory상속, 스트림 처리등 편의 기능이 많음, 라이브러리 필요없음, <font color="#ffff00">스프링에 의존</font>


### 스프링에 의존하지 않는 방법은 없나?

___
## 3. JSR-33 Provider

- javax.inject.Provider 라는 JSR-330 자바 표준을 사용하는 방법

```java hl:8,5
@Scope("singleton")  
static class ClientBean{  
  
    @Autowired  
    private Provider<PrototypeBean> prototypeBeanObjectProvider;  
  
    public int logic(){  
        PrototypeBean prototypeBean = prototypeBeanObjectProvider.get();  
        prototypeBean.addCount();  
        int count = prototypeBean.getCount();  
        return count;  
    }  
}
```

- implementation 'jakarta.inject:jakarta.inject-api:2.0.1'를 build.gradle에 추가
#### 특징
1. get() 메서드 하나로 기능이 매우 단순
2. <font color="#ffff00">별도의 라이브러리 필요</font>
3. 자바 표준이므로 스프링이 아니라 다른 컨테이너에서도 사용가능
___

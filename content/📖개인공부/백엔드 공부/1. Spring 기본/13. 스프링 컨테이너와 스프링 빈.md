


``` Java
// 스프링 컨테이너 생성
ApplicationContext applicationContext = new AnnotationConfigApplicationContext(AppcoConfig.class);
```
- ApplicationContext는 <u>인터페이스</u>다.
- 즉 AnnotationConfigApplicationContext는 <u>구현체</u>임.
- 요즘에는 XML기반으로 설정하는 것은 딱히 쓰지 않음. 어노테이션이 주류.
- AppConfig를 사용했던 방식(@Confijuraion, @Bean)이 <u>어노테이션 기반의 자바 설정 클래스로 스프링 컨테이너를 만드는 방식</u>임.


### 0. 스프링 컨테이너 생성
![[Pasted image 20241123153954.png]]
- 스프링 컨테이너를 만들면, 빈(Bean) 저장소가 생김. -> 그래서 구성 정보(AppConfig)를 넘겨줘야함


### 1. 스프링 빈 등록
![[Pasted image 20241123154115.png]]
- @Bean을 확인해서 전부 호출한 다음에, 빈 이름(key)를 지정하고 return값(호출하면 반환하는 값)을 빈 객체로 저장을 한다.
- 빈 이름은 보통 메서드 이름  or  @Bean(name="빈이름")이런 식으로 지정 가능
- <font color="#ffff00">빈의 이름은 항상 다르게 만들어야 함.</font>


### 2. 스프링 빈 의존관계 설정 - 준비
![[Pasted image 20241123154503.png]]
- 일단 객체를 생성을 하고, 의존관계를 주입할 준비를 한다.


### 3. 스프링 빈 의존관계 설정 - 완료
![[Pasted image 20241123154605.png]]
- 실제로 동적인 인스턴스 의존관계를 연결을 해놓음.
- <span style="background:#d4b106">단순히 자바 코드를 호출하는 것이 아니다! 추후 싱글톤 컨테이너에서 설명할 것.</span>

> 스프링을 사용하지 않고, 순수 자바코드(생성자)를 이용하면 
> 의존관계가 준비(빈 생성)-완료(의존관계 주입) 단계로 나뉘어 지는 것이 아니라 생성자를 호출하면서 바로 의존관계 주입도 한번에 되버림.

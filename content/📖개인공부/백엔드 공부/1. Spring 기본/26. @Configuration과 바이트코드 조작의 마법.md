
- 스프링 컨테이너는 싱글톤 레지스트리다 -> 즉 어떠한 상황에서도 스프링 빈이 싱글톤이 되도록 보장해줘야 한다.
- 근데 스프링이 자바 코드까지 어떻게 할 수는 없다.. -> 그래서 스프링은<span style="background:#d4b106"> 클래스의 바이트코드를 조작</span>하는 라이브러리를 사용한다.



모든 비밀은 @Configuration이 적용된 AppConfig에 있다..

``` java title:"Appconfig의 클래스를 보자", hl:10
@Test  
void configuraionDeep(){  
    ApplicationContext ac  =  new AnnotationConfigApplicationContext(AppConfig.class);  
    AppConfig bean = ac.getBean(AppConfig.class);  
  
    System.out.println("bean = " + bean.getClass());;  
}

------------------------------------------------------
bean = class hello.example.core.AppConfig$$SpringCGLIB$$0
```

- 보면 CGLIB가 붙어있다.
	- 이것은 바이트 코드 조작 라이브러리 -> AppConfig를 상속받은 새로운 클래스를 만들어서 등록을 한다.
	- 그러니까 우리가 쓰는<font color="#ffff00"> AppConfig는 사실 적용되는 것이 아니라. 바이트 코드 조작을 해버린 후에 그걸 등록해서 사용한다.</font>

![[Pasted image 20241126173328.png]]
- 이런 식으로 <font color="#ffff00">이미 스프링 컨테이너에 있으면 그냥 </font><font color="#ffff00">찾아서 반환</font>한다.
- 없으면 진짜 로직을 발동해서 생성을 한다(new MemoryRepository)
- 우리가 AppConfig을 Bean으로 만들어서 조회했을 때 다른 값이 나오는 것은 사실은 바이트 코드가 조작된 새로운 AppConfig이기 때문이다.


### <font color="#ffff00">그렇기 때문에 싱글톤이 보장된다.</font>

## 만약에 @Configuration을 안 붙이면?


=> 일단 진짜 AppConfig가 진짜로 스프링 빈에 등록이 된다.

=> 그리고 심지어, MemberService에서 불러오는 MemberRepository는 스프링 빈이 아님! 그냥 진짜 순수 자바가 만든 객체가 됨.(관리가 안됨)


=> 그런데 이제 싱글톤이 보장되지 않아서, 여러번 call된다

즉 싱글톤이 깨진다. 많은 new가 생기고, 성능이 안좋아지고~

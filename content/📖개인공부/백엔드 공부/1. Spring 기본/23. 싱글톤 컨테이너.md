
- 스프링 컨테이너는 싱글톤 패턴을 적용하지 않아도, 객체 인스턴스를 싱글톤으로 관리한다.
	-<u> 관리하는 인스턴스는 한 종류당 하나만 관리하고 있으므로
</u>
- 더 이상 귀찮고 안좋은 싱글톤 패턴의 문제점을 해결해준다. => 마치 싱글톤을 안하듯이 코드를 짜지만 스프링이 알아서 싱글톤으로 관리해준다. => 유연성은 그대로 가지고 싱글톤 패턴도 가져갈 수 있음.

```java title:"스프링 코드에서는 같은 주소가 나온다 - 싱글톤", hl:5
@Test  
@DisplayName("스프링 컨테이너와 싱글톤")  
void springContainer() {  
  
    AnnotationConfigApplicationContext ac = new AnnotationConfigApplicationContext(AppConfig.class);  
  
    // 1. 조회 : 호출할 때 마다 객체를 생성  
    MemberService memberService1 = ac.getBean("memberService", MemberService.class);  
  
    // 2. 조회 : 호출할 때 마다 객체를 생성  
    MemberService memberService2 = ac.getBean("memberService", MemberService.class);  
  
    // 참조값이 다를 것을 확인  
    System.out.println("memberService1 = " + memberService1);  
    System.out.println("memberService2 = " + memberService2);  
  
    assertThat(memberService1).isSameAs(memberService2);  
}

==============결과=============
memberService1 = hello.example.core.member.MemberServiceImpl@4ed5eb72
memberService2 = hello.example.core.member.MemberServiceImpl@4ed5eb72
```
- 이미 만들어진 객체를 가지고 오기 때문에 효율적이다.
	- 22번 글에서는 appConfig에서 직접 가져오기 때문에, 생성자에 의해서 새로운 객체가 생김 -> 여기서는 ac를 이용해서 스프링 빈을 가져오기 때문에 같은 객체를 가져온다.

> 스프링의 기본빈 방식은 싱글톤, 
> -> 근데 요청할 떄 마다<u> 새로운 객체를 생성해서 반환하는 기능도 제공</u>을 한다.

- 서버를 띄울 떄, 애플리케이션이랑 DB를 미리 연결을 해놓는다
	- 더 빠르게 값을 가져온다던지, 나중에 연결하면 오래 걸리니까

- 애플리케이션 시작 시점에 미리 연결 / 애플리케이션이 종료되면 연결을 모두 안전하게 종료
	- <font color="#ffff00">객체의 초기화와 종료작업</font>



``` java, title:""
public class NetworkClient {  
  
    private String url;  
  
    public NetworkClient(){  
        System.out.println("생성자 호출, url = "+url);  
        connect();  
        call("초기화 연결 메세지");  
    }  
  
    public void setUrl(String url){  
        this.url = url;  
    }  
  
    // 서비스를 시작시 호출하는 메서드  
    public void connect(){  
        System.out.println("connect : "+url);  
    }  
  
    public void call(String message){  
        System.out.println("call : " + url + " message = " + message);  
    }  
  
    // 서비스 종료시 호출 되는 메서드  
    public void disconnect(){  
        System.out.println("close: "+url);  
    }  
}

public class BeanLifeCycleTest {  
  
    @Test  
    public void lifeCycleTest(){  
        // 그냥 ApplicationContext로 하게 되면 .close()가 안됨 -> 보통 할 일이 없음  
        // 그래서 기본 ApplicationContext에서는 지원하지 않아서  
        // ConfigurableApplicationContext or AnnotationConfigApplicationContext와 같은  
        // 상위의 인터페이스를 사용함  
        ConfigurableApplicationContext ac = new AnnotationConfigApplicationContext(LifeCycleConfig.class);  
        NetworkClient client = ac.getBean(NetworkClient.class);  
        ac.close();  
    }  
  
    @Configuration  
    static class LifeCycleConfig{  
  
        // 스프링 빈이 생성이 되고, url이 set된 상태로 호출이 된다.  
        @Bean  
        public NetworkClient networkClient(){  
            NetworkClient networkClient = new NetworkClient();  
            networkClient.setUrl("https://www.inflearn.com/courses/lecture?courseId=325969&unitId=55383&subtitleLanguage=ko");  
            return networkClient;  
        }  
    }  
}

connect : null
call : null message = 초기화 연결 메세지
```
- 생성자를 처음 만들면,  url이 없음 -> null
- 생성자에서 connect()를 호출, url 값이 없음 -> null
- call()에서도 url은 없음

#### 실패!

- 외부에서 값을 세팅하고, 초기화를 해야하는 경우가 많음
	-> 수정자 주입을 해야 url이 들어오게 된다.
	- <font color="#ffff00">스프링 빈은 객체를 생성하고 -> 의존관계 주입 완료 -> 이후에 필요한 데이터를 사용할 수 있는 준비 된다.</font>


> 스프링 빈의 라이프사이클
> 1. 객체 생성
> 2. 의존관계 주입


> **<span style="background:#fff88f">스프링 빈의 이벤트 라이프사이클</span>**
> 1. 스프링 컨테이너 생성
> 2. 스프링 빈 생성
> 3. 의존관계 주입
> 4. <u>초기화 콜백</u>
> 5. 사용
> 6. <u>소멸전 콜백</u>
> 7. 스프링 종료

- 초기화 콜백 : 빈이 생성되고, 빈의 의존관계가 주입이 완료된 후 호출
- 소멸전 콜백 : 빈이 소멸되기 직전에 호출

> <font color="#ffff00">"객체의 생성과 초기화를 분리하자"</font>
> - 객체의 초기화는 초기화에만 집중해야한다
> - 객체 생성은 딱 메모리 할당까지만(최소한의 필수 데이터만 세팅)

초기화는 무거운 작업을 하기 때문에, 생성자와 초기화를 분리하는 것이 좋다.
-> 무거운 작업(외부 연결...etc)는 초기화 단계에서 따로 하자
-> 간단한 값을 바꾸는 작업 같은 경우는 생성자에서 해도 괜찮음

장점 :  객체 생성 -> 최초의 행위가 올 때까지 미루어 놓을 수 있음


### 스프링의 생명주기 콜백 지원 방법 3가지

1. 인터페이스(InitializingBean, DisposableBean)
2. 설정 정보에 초기화 메서드, 종료 메서드 지원
3. @PostConstruct, @PostDestroy 에노테이션

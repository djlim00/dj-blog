
## 0. 제어의 역전 IoC(Inversion of Control)

- 지금까지는, 클라이언트 구현 객체가 스스로 필요한 구현 객체를 생성,연결,실행 했다.
	: 한마디로 구현 객체가 프로그램의 제어 프름을 스스로 조종함(개발자 입장에서는 자연스러운 흐름)

- 뭐 내가 필요하면 호출해서 쓰고, 또 필요하면 호출하고 이런 식.

- 그러나 <font color="#d83931">AppConfig가 등장 이후</font>에는, <u>구현 객체는 자신의 로직을 실행만 한다.</u> 프로그램의 제어 프름(필요할 떄 어떤 것을 호출하는지)은 AppConfig가 전부 가져간다. 
	: 예를 들어서 OrderServiceImpl은 <span style="background:#d4b106">필요한 인터페이스를 호출하지만 어떤 구현 객체가 올지는 전혀 모름</span>
	: 심지어 OrderServiceImple 조차도 AppConFig가 불러온다..


### 즉 프로그램의 제어 흐름을 직접 제어하는 것이 아니라 외부에서 관리하는 것을 IoC라고 한다.

==========================================================
- 프레임워크가 내가 작성한 코드를 제어하고, 실행하면 프레임워크 
- 내가 작성한 코드의 제어의 흐름을 내가 직점 담당하면 라이브러리
==========================================================

## 2. 의존관계 주입 DI(Dependecy Injection)

- 의존관계는 <span style="background:#d4b106">**"정적인 클래스 의존 관계와, 실행 시점에 결정되는 동적인 객체(인스턴스) 의존 관계"** </span>를 구분해서 생각해야 한다.


#### "정적 의존관계"
- 그냥 import를 보면 알 수 있음.
- 애플리케이션을 싱행하지 않아도 알 수 있음 + 클래스 다이어그램도
![[Pasted image 20241122225824.png]]
- 그러나 이런 다이어그램을 봐도 DiscountPolicy에 어떤 구현체가 올지는 알 수가 없다. <span style="background:#d4b106">즉 OrderServiceImpl에 어떤 객체가 주입될지는 알 수가 없다.(동적인 의존관계)</span>


#### "동적인 객체 인스턴스 의존관계"
- 애플리케이션 실행 시점에 실제 생성된 객체 인스턴스 참조가 연결된 의존 관계이다.

**<객체 다이어그램 - 정액 요금제인 경우>**
![[Pasted image 20241122230118.png]]

- 애플리케이션 실행 시점에 외부에서 실제 구현객체를 생성하고, 클라이언트에 전달해서 클라이언트와 서버의 실제 의존관계가 연결되는 것을 <span style="background:#d4b106">의존관계 주입</span>이라고 한다.
- <span style="background:#d4b106">의존관계 주입을 사용하면, 정적인 클래스 의존관계를 변경하지 않고, 동적인 객체 인스턴스 의존관계를 쉽게 변경 가능</span>

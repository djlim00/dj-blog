## 0. SOLID란?

- 클린코드로 유명한 로버트 마틴이 좋은 객체 지향 설계의 5가지 원칙을 정리
	- SRP
	- OCP
	- LSP
	- ISP
	- DIP

___
## 1. SRP - 단일 책임 원칙

- <span style="background:#d4b106">한 클래스는 하나의 책임만 가져야 한다.</span>
- 하나의 책임은 모호하다.
	- 책임 클 수도 있고, 작을 수도 있음
	- 문맥과 상황에 따라 다름
- 중요한 기준은 변경. 변경이 있을 떄 파급효과가 적으면 단일 책임 원칙을 잘 따른 것.

- 책임의 범위를 너무 작게해도 문제고 너무 크면 변경에 취약함

___
## 2. OCP - 개방/폐쇄 원칙

- 소프트웨어 요소는 <span style="background:#d4b106">확장에 열려 있으나</span> <span style="background:#d4b106">변경에는 닫혀 있어야</span> 한다.
- 확장을 하려면 기존의 코드를 변경하는거 아닌가? -> <span style="background:#d4b106">다형성을 잘 이용하라는 뜻</span>
- 인터페이스를 구현한 <u>새로운 클래스</u>를 새로 만들어서 새로운 기능을 구현

=> 이러한 문제를 해결하기 위해서 <span style="background:#d4b106">객체를 생성하고, 연관관계를 맺처주는 별도의 조립, 설정자</span>가 필요함! => <span style="background:#d4b106">Spring이 도와준다(DI) 의존성 주입</span>을 통해서 해결

___
## 3. LSP - 리스코프 치환 원칙

- <font color="#ffff00">프로그램의 객체는 프로그램의 정확성을 깨지 않으면서 하위 타입의 인스턴스로 바꿀 수 있어야 한다.</font>
- 다형성에서 하위 클래스는 인터페이스 규약을 다 지켜야 한다는 것, 다형성을 지원하기 위한 원칙, 인터페이스를 구현한 구현체를 믿고 사용하려면 이 원칙이 필요함.
- 단순하게 컴파일이 성공하는 것이 아님

ex) 자동차 인터페이스의 엑셀 기능 - 무조건 전진을 하도록 설계를 해야함. 무슨 일이 있어도 뒤로 나아가는 기능을 해서는 안됨.(후진 기능을 넣어도 컴파일은 되겠지만, 리스코프 치환 원칙에 위배됨.)


___
## 4. ISP - 인터페이스 분리 원칙

- <font color="#ffff00">특정 클라이언트를 위한 인터페이스 여러 개가 범용 인터페이스 하나보다 낫다.</font>

ex)
자동차 인터페이스 -> 운전 인터페이스, 정비 인터페이스
사용자 인터페이스 -> 운전자 클라이언트, 정비사 인터페이스

이런식을 인터페이스를 기능에 따라서 나누면 "정비 인터페이스" 자체가 변해도, "운전자 클라이언"에 영향이 없음

<font color="#ffff00">인터페이스가 명확해지고, 대체 가능성이 높아진다.</font>

___
## 5. DIP - 의존관계 역전 원칙

- <span style="background:#d4b106">프로그래머는 "추상화에 의존해야지, 구체화에 의존하면 안된다."</span>
- 클라이언트 클래스는 구현 클래스에 의존하면 안되고, 인터페이스에 의존해라
- 역할에 의존하라 -> 즉 배우가 아닌 배역에 의존해라. 차종이 뭔지가 아니라 자동차라는 역할에 의존해라


- 'basePackges' : 지정된 패키지부터 시작(하위로 봄)
- 'basePackgeClasses' : 지정된 클래스의 패키지를 탐색 시작(하위로 봄)
> 지정하지 않았을 때 :<font color="#ffff00"> @ComponentScan을 선언한 클래스부터 시작(우리의 프로젝트는 AutoAppConfig)아래로 다 보면 되기 때문에 Default로 지정하지 않으면 된다.</font>
> 
> <font color="#ffff00">그냥 설정정보 클래스의 위치를 프로젝트 최상단(root)에 두는 것이 좋다.</font>
> @SpringBootApplication를 이 프로젝트 시작루트 위치에 두는 것이 관례이다. (그리고 이 설정안에 바로 `@ComponentScan` 이 들어있다!) -> 사실 스프링 부트를 쓰면 @ComponentScan을 쓸 필요 자체가 없음.

``` java title:"탐색위치 지정 방법", hl:3-4,7,8,13
package hello.example.core;  
  
@Configuration  
@ComponentScan(  
  
        // 이렇게 베이스 패키지를 지정하면 거기서 부터 찾음, 아래로 하면 Order패키지 하위는 둥록이 안됨   
        // basePackages = "hello.example.core.member",  
        // basePackageClasses = AutoAppConfig.class,  
        // Default는 뭔가?(지정하지 않았을 떄) 
  
        // 처음에 만들었던 AppConfig에도 @Component가 되기 떄문에 이걸 뺴기 위해서  
        // 그리고 테스트 코드에도 만들어놓은 애들이 있어서 빼야함!  
        excludeFilters =  @ComponentScan.Filter(type = FilterType.ANNOTATION, classes = Configuration.class)  
)  
public class AutoAppConfig {  
}
```

### @ComponentScan의 대상

- **@Component** : 기본 컴포넌트 스캔 대상

- **@Controller** : 스프링 MVC 컨트롤러로 인식

- **@Service** : 스프링 비즈니스 로직 처리(별 다른 처리는 하지 않는다, 그저 서비스 로직이 있다고,비즈니스 계층이라고 인지하는데 도움이 된다.)

- **@Repository** : 스프링 데이터 접근 계층으로 인식하고, <u>데이터 계층의 예뢰를 스프링 예외로 변환</u>해준다. -> 이거 정말로 좋습니다..

- **@Configuration** : 스프링 설정 정보

> @Service안에 @Component가 있다고 상속관계가 아니다.

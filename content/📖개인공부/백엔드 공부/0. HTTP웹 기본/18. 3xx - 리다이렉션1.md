- 300 Multiple Choices (거의 안씀)
- 301 Moved Permanently
- 302 Found
- 303 See Other
- 304 Not Modified
- 307 Temporary Redirect
- 308 Permanent Redirect

- 웹 브라우저는 <font color="#ffff00">3xx 응답의 결과에 Location 헤더가 있으면, Location 위치로 자동 이동</font>한다(Redirect)

### <font color="#ff0000">일시적 리다이렉션 302, 307, 303을 자주 사용한다.</font>

# 2. 영구 리다이렉션 - 301, 308

- <font color="#ffff00">리소스의 URI가 영구적으로 이동</font>
- 원래의 URL을 사용X, 검색엔진 등에서도 변경 인지

>**- 301 Moved Permanently**
	- 리다이렉션 요청 메서드가 GET으로 변하고, <font color="#ffff00">본문이 제거될 수도 있음</font>
>**- 308 Permanent Redirect**
	- 301과 같은 기능
	- 리다이렉트 시 <font color="#ffff00">요청 메서드와 본문 유지</font>(처음 POST를 보내면 리다이렉트도 POST)

![[Pasted image 20250102185948.png]]
- 리다이렉트가 되면서,<font color="#ffff00"> 메서드가 GET으로 바뀌고, Body의 정보가 없는 상태로 다시 요청</font>한다.
	- 결론적으로 등록을 위한 POST 메서드가 제대로 동작하지 않음
	- 등록하려면 화면을 처음부터 시작을 해야함


![[Pasted image 20250102190134.png]]
- POST를 유지하고, 바디도 유지된다. ->  근데 솔직히 완전히 다른 URL로 바뀌는데 그렇게 되면 그냥 받는 값이 완전히 달라지기 때문에, GET으로 바뀌는 것이 웬만하면 맞음

# 3. 일시적인 리다이렉션 - 302, 307, 303

- 리소스의 URI가 일시적으로 변경
- 따라서 검색 엔진 등에서 URL을 변경하면 안됨

> **- 302 Found**
> 	- 리다이렉트 시 요청 메서드가 GET으로 변하고, 본문이 제거될 수 있음(MAY)
> **- 307 Temporary Redirect**
> 	- 302와 기능은 같음
> 	- 리다이렉트시 요청 메서드와 본문 유지(요청 메서드를 변경하면 안된다. MUST NOT)
> **- 303 See Other**
> 	- 302와 기능은 같음
> 	- 리다이렉트 시 요청 메서드가 GET으로 변경

- 302는 좀 모호하지만, 대부분 302와 307을 많이 사용함


## 3.1 PRG: Post/Redirect/Get

- Post로 주문 후에 웹 브라우저를 새로고침하면?
- 새로고침은 다시 요청
- <font color="#ffff00">중복 주문이 될 수 있다.(Post 요청이 한 번 더 가서 주문 요청이 중복으로 들어가기 때문에)</font>

### 3.1.1 PRG 사용 전

![[Pasted image 20250102191102.png]]
- 그냥 Post 요청을 하고 200 OK를 보내고 끝낸다.
- 여기서 새로고침을 하면 또 Post 요청을 보내고 200 OK를 보낸다 -> 중복


> - Post로 주문 후에 새로고침으로 인한 중복 주문 방지
> - Post로 주문 후에 주문 결과 화면을 Get으로 리다이렉트
> - 새로고침을 해도 결과 하면을 GET으로 도회
> - 중복 주문 대신에 결과 화면만 GET으로 다시 요청


### 3.1.2 PRG 사용 후

![[Pasted image 20250102191214.png]]
1. 클라이언트가 Post 요청을 보내서 주문 요청 -> 데이터 베이스에 1개 저장
2. 302 Found 상태코드와 함께, /order-result/19 화면으로 이동(GET메서드로)
3. 새로고침 -> GET/ order-result/19로 이동 -> GET으로 화면 조회만 일어남
4. GET 요청으로 인한 상태코드는 당연히 200 OK


#### 중복 주문을 막는 방법 POST(주문 완료) -> Redirect(결과화면으로) -> GET(결과화면을 받는 GET 메서드에서 새로고침

___

# 4. 그래서 302, 307, 303 중 뭘 써야하나?

- 302 Found는 -> 거의 항상 GET으로 변경
- 307 Temporary -> 메서드가 변하면 안됨
- 303 See Other -> 메서드가 GET으로 변경

- 307, 303을 권장하지만, 이미 많은 애플리케이션들이 302를 많이 사용해서 그냥 302로 기본값을 가지자

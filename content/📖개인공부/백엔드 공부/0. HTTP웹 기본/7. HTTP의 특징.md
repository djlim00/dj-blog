# 2. 무상태 프로토콜(Stateless protocol)

![[Pasted image 20241231212452.png]]
____
### 2.1 Stateful은 어떤 것일까?
![[Pasted image 20241231212548.png]]
- 중간에 점원이 바뀌게 되면 클라이언트의 상태를 모른다!
- 또한 질문에서 어떤 것을 말하는지 알 수 없다
![[Pasted image 20241231212857.png]]
- 그렇기 때문에 이렇게 <font color="#ffff00">상태를 유지</font>하고 있어야한다.
____
### 2.2 Stateless는 어떤 것일까?
![[Pasted image 20241231212713.png]]
- <font color="#ffff00">어떤 것을 얼마나 어떻게 사는지를 알려준다</font>. -> <u>중간에 직원이 바뀌어도 상관이 없다</u>
- 즉 <font color="#ffff00">클라이언트의 상태를 몰라도 응답이 가능</font>하다.
___
### 2.3 Stateful vs Stateless

![[Pasted image 20241231213135.png]]
- 클라이언트가 필요한 내용을 그때 그때 전부 알려준다.
	- <span style="background:#d4b106">Stateless 설계는 엄청난 확장성을 가진다</span>


#### 2.3.1 Stateful한 방식의 문제
![[Pasted image 20241231213434.png]]
1. 항상 같은 서버가 유지되어야한다(상태 저장)
2. 중간에 서버가 장애나 나면 다른 서버와 연결하여서 처음부터 다시 해야한다.


#### 2.3.2 Stateless한 방식의 장점
![[Pasted image 20241231213542.png]]
1. 어떤 서버랑 연결해도 상관이 없다
2. 기존의 서버가 장애가 생겨도<font color="#ffff00"> 다른 서버로 옮길 수 있다.</font>

___
# 3. Stateless 방식의 한계

![[Pasted image 20241231213642.png]]
- 로그인한 상태를 서버에 유지하는 방법 -> Stateless하게 할 수가 없다
	- 쿠키+세션 -> 세션이 끝나면 로그인이 풀린다
- 즉 Stateful한 방식은 <u>최소한</u>으로 사용해야한다.
- <font color="#ffff00">너무 많은 데이터를 보내는 문제가 있음</font>

### 1. Client Hello - 클라이언트
___
 : 클라이언트는 
 : TSL버전, <mark style="background: #FFF3A3A6;">사이버슈트</mark>와 클라이언트 랜덤값(무작위 문자열), <mark style="background: #FFF3A3A6;">임시 DH매개변수</mark>를
 : 서버에 보냄

### 2. Server Hello, EncryptedExtensions, Certificate, Certificate Verify - 서버
___
: 서버는 클라이언트로 받은
: 옵션을 확인함
: 서버와 클라이언트 모두에서 지원하는
: 가장 높은 버전의 TLS를 선정
: 사이버슈트 지원 여부를 확인하고(여러가지 암호화 알고리즘의 리스트)
: 사이버슈트 중에서 서버가 지원하는 것과
: <mark style="background: #FFF3A3A6;">공개키가 포함된 SSL 인증서</mark>, 서버 랜덤값, 임시 DH 매개변수를 
: 보냄
: 클라이언트와 서버 각각 교환한 DH매개변수를 사용하여
: **임시 암호 키(세션키)** 를 생성 => 이제 대칭 암호화 수행

🙉 TLS 1.3 이전에는 그걸 전부 하나씩 하는 바람에 엄청 RTT가 컷음!! 
근데 TLS1.3부터는 이걸 1-RTT로 할 수 있음!

### 3. Finished - 둘다
___
: 서버와 클라이언트 모두
: 이전에 교환한 모든 메시지에 대한 무결성 인증을 확인하는
: 메시지를 보내고
: 클라이언트와 서버와 세션키를 기반으로
: <mark style="background: #FFF3A3A6;">대칭 암호화된 통신이 시작</mark>(보안세션이 시작)

- <mark style="background: #FFF3A3A6;">공개값과 혼합값을 한번에 보내기 때문에 1-RTT만에 가능하다</mark>

```
### 🔑 **디피-헬만 키 교환 알고리즘 – 예제**

#### **배경 설정**

- 두 사람: **A(나)**, **B(상대방)**
- 공통 공개값(Prime, Base): **공개 소수 `p=23`**, **기본값 `g=5`**

---

### 🛠️ **1. 비밀값 선택 (개인 비밀키)**

- **A의 비밀값**: `a = 6` (비밀, 공개X)
- **B의 비밀값**: `b = 15` (비밀, 공개X)

---

### 🌐 **2. 공개값 생성 (혼합 값)**

각자 비밀값을 `g^비밀값 mod p`로 계산합니다.

- A → 공개값: `g^a mod p = 5^6 mod 23`
    
    - `5^6 = 15625`
    - `15625 mod 23 = 8`  
        **A의 공개값 = 8**
- B → 공개값: `g^b mod p = 5^15 mod 23`
    
    - `5^15 = 30517578125`
    - `30517578125 mod 23 = 19`  
        **B의 공개값 = 19**

**A와 B는 서로 이 공개값(8, 19)을 공유합니다.**

---

### 🔄 **3. 공유된 키 생성**

서로의 공개값을 가져와 비밀값과 다시 혼합합니다.

- A의 공유 키 계산: `(B의 공개값)^a mod p`
    
    - `19^6 mod 23`
    - `19^6 = 47045881`
    - `47045881 mod 23 = 2`  
        **A의 공유 키 = 2**
- B의 공유 키 계산: `(A의 공개값)^b mod p`
    
    - `8^15 mod 23`
    - `8^15 = 35184372088832`
    - `35184372088832 mod 23 = 2`  
        **B의 공유 키 = 2**

---

### 🧠 **결과 – 공유된 암호 키**

- A와 B는 **비밀 키 2**를 공유합니다.
- 이 키는 **네트워크에서 노출된 공개값(8, 19)**만으로는 알 수 없습니다.
```

2. 인증 확인 : 메시지를 복호화해서 해시를 서로 비교해 올바른 메시지인지 확인

	- 클라이언트가 접속한 서버가 클라이언트가 의도한 서버가 맞는지 확인

### 4. CA

- 인증서는 클라이언트가 접속한 서버가 클라이언트가 의도한 서버가 맞는지 보장한다.
- 이런 인증서를 발급하는 기업들을 CA(Certificate Authority)라고 한다.

## RSA의 취약점 - DH를 쓰는 이유 - 탈취 위험성!

	- <mark style="background: #FFF3A3A6;">DH는 탈취 당해도 공통의 암호키를 못 만듦 -> 혼합하는 과정이 있어서</mark>

# 정리
1. 클라이언트가 Clinet Hello를 하면서
	- DH매계변수
	- 클라이언트 랜덤값
	- TSL버전
	- 사이버슈트
2. 서버에서 TLS 버전 가장 높은걸로 맞추고, 사이버슈트 지원 가능한거 선정
3. 서버에서 DH 매계변수와 서버 랜던값을 만듦(DH타원곡선 알고리즘)
4. 이제 혼합 공통 혼합해서 공통의 암호키를 만듦
5. + 서버에서 인증서를 보냄
6. CA에서 인증서를 발급 받아서
	- 올바른 인증서인지 전자서명으로 확인(해시 알고리즘)
7. 사이버 슈트 - TSL, 암호화 알고리즘, 해싱 알고리즘

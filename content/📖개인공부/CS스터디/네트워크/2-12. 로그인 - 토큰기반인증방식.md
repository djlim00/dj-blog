# 🪙 토큰기반 인증방식
___
- 유저의 상태(State)를 토큰에 넣어놓고
- 서버는 Stateless하게 가자

#### 여러개의 서버가 있다고 보자
- Token을 관리하는 서버가 따로 있다
- 나머지 로직을 담당하는 서버에는 Token을 관리하지 않는다
	- 만약 다른 서버에 Token인증 기능이 있게 되면
	- <mark style="background: #FFF3A3A6;">해당 서버가 다운되면, 로그인까지 전부 마비가 된다..ㄷㄷ</mark>

#### <과정>
1. 인증로직 >> <mark style="background: #FFF3A3A6;">JWT토큰생성</mark>(Access토큰, Refresh토큰)
2. 사용자가 이후에 Access토큰을 HTTP Header - Authorization 또는 HTTP Header - Cookie에 담아 인증이 필요한 서버에 요청하여 원하는 컨텐츠를 가져옵니다.

## 😎 JWT란?
___
![[Pasted image 20250219231510.png]]
: JWT는 JSON Web Token을 의미
: 헤더, 페이로드, 서명으로 이루어짐
: <mark style="background: #FFF3A3A6;">JSON객체로 인코딩되며</mark>
: 메시지 인증, 암호화에 사용

#### 1. Header
: 토큰 유형
: 서명 알고리즘
: <font color="#ff0000">base64URI</font>로 인코딩
#### 2. Payload
: <mark style="background: #FFF3A3A6;">데이터</mark>
: 토큰 발급자
: 토큰 유효기간
: base64URI로 인코딩
#### 3. Signature
: (인코딩된 Header + payload) + 비밀키
: 를 기반으로 헤더에 명시된 
: 알고리즘으로 다시 생성된 <mark style="background: #FFF3A3A6;">서명값</mark>

![[Pasted image 20250219231723.png]]


## JWT 장점
____
1. 사용자 인증에 필요한 모든 정보는 토큰 자체에 포함하기 때문에 <mark style="background: #FFF3A3A6;">별도의 인증 저장소가 필요없음</mark>
2. 다른 유형의 토큰과 비교했을 때 <mark style="background: #FFF3A3A6;">경량화</mark>
	- SAML이란 토큰에 비해서 훨씬 경량화
	- ![[Pasted image 20250219231924.png|300]]
3. 디코딩하면 JSON이 나오기 때문에 <mark style="background: #FFF3A3A6;">JSON기반의 직렬화, 역직렬화</mark>가 가능하기 때문에 편함


## JWT 단점
____
1. 토큰이 비대해지면 서버 과부화
2. <mark style="background: #FFF3A3A6;">토큰이 탈취당하면, 디코딩했을 때 데이터를 볼 수 있음</mark> 🙉



# 토큰기반인증방식 실습
___

<mark style="background: #FFF3A3A6;">Access토큰은 수명이 짧게, Refresh토큰은 수명이 길게 한다</mark>
- refresh토큰은 access토큰이 만료되었을 때, 다시 access 토큰을 얻기 위해 사용되는 토큰!
	- 이를 통해 accss 토큰이 만료됬을 때마다, 인증에 관한 비용이 줄어듬
	- 로그인이 성공하면 access, refresh 둘다 받는다

#### 사용자가 로그인 요청을 보낼 때
![[Pasted image 20250220092310.png]]

- acess 코튼이 만료되거나, 사용자가 새로고침을 하면 refresh토큰을 기반으로 새로운 access token을 얻는다
![[Pasted image 20250220092358.png]]

access 토큰이 탈취당하게 되면, 큰일나기 때문에 만료기간을 짧게 한다
-> 이러면 자꾸 로그인을 해야하는 문제가 생김(UX저하)
-> acess토큰은 refresh를 보내는 것으로 받아낸다(마치 access토큰의 수명이 긴거 처럼 가능)


### 1. 로그인을 안하면 -> "허용되지 않은 접근"
### 2. 로그인 성공 -> Access, refresh 토큰 발급
![[Pasted image 20250220092912.png]]
### 3. header명 authorization에 access token을 넣어서 -> 허용된 유저임을 보임
![[Pasted image 20250220092926.png]]
### 4. Cookie에 refresh 토큰을 담아서 서버에 보냄
![[Pasted image 20250220093021.png]]
-> Cookie에 refresh 토큰을 보내서 Access 토큰을 받음

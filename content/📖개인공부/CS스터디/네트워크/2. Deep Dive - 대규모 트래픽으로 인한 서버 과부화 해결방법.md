# 해결방법4 : 서킷 브레이커
___
== 서킷 브레이커 패턴
#### <span style="background:rgba(205, 244, 105, 0.55)">서비스 장애를 감지하고 연쇄적으로 생기는 에러를 방지하는 기법</span>
: 서비스와 서비스 사이에 서킷 브레이커 계층을 두고
: 미리 설정해 놓은 <mark style="background: #FFF3A3A6;">timeout임계값에 도달</mark>하면 
: 서킷 브레이커가 그 이후의 추가호출에
: <mark style="background: #FFF3A3A6;">무조건 에러를 반환</mark>하게 합니다.

![[Pasted image 20250220144041.png]]
**"연쇄적인 오류전파를 끊내겠다..!"**

#### 사례1 : Thread 차단
![[Pasted image 20250220151021.png]]
- 서비스A 로직에 너무 많은 요청이 들어와서 A가 터지면
- 서비스B와 서비스C까지 서비스A의 응답을 기다리다가 하나씩 timeout이 되서 전부 터질 수도 있음!!
- 중간에 서킷 브레이커를 둬서 A가 에러가 났음을 알리고 그 다음으로 기다리는 로직들에 대한 요청을 차단한다.

![[Pasted image 20250220151047.png]]
- A가 터졌을 때 빠르게 에러를 반환해서 B와 C는 멈춤

![[Pasted image 20250220151232.png]]

## 서킷브레이커의 상태
___
: 100% 성공할 수도 있다 언제든 실패는 있을 수 있음 -> <span style="background:rgba(205, 244, 105, 0.55)">임계치를 정해놓음</span>
![[Pasted image 20250220151557.png]]
1. **closed[정상]** : 네트워크 요청의 실패율이 임계치보다 낮음
2. **open[에러]** : 임계치 이상의 실패율. 요청을 서비스로 전송하지 않고 바로 Error 반환. Fail Fast라고 한다
3. **half_open[확인중]** : open 상태에서 일정 timeout으로 설정된 시간이 지나면 장애가 해결되었는지 확인하기 위해서 hal_open상태로 전환. 여기서 요청을 다시 전송해서 응답을 확인. 장애가 풀렸다면 closed, 다시 실패하면 open으로~

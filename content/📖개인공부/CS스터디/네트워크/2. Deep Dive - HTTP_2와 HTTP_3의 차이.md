
# HTTP/2
___
- 2009년 구글이 1.1의 한계를 극복하기 위해 SPDY 프로토콜 개발
- 이런 SPDY를 기반으로 하는 HTTP/2.0!

## <font color="#ff0000">1. 바이너리 포맷 계층</font>
- 애플리케이션 계층과 전송 계층 사이에 바이너리 포맷 계층을 추가
- 1.0은 일반 텍스트 메시지를 전송하고 줄바꿈으로 데이터를 나누었다면, 
- <mark style="background: #FFF3A3A6;">2.0에서는 0과1로 이루어진 바이너리 데이터로 변경 -> 더 작은 메시지가 프레임으로 캡슐화되어서 전송됨</mark>
![[Pasted image 20250217143905.png]]
 - 과거와 같은 단순 텍스트가 아님!


## <font color="#ff0000">2. 멀티플렉싱</font>

- <mark style="background: #FFF3A3A6;">단일 TCP 연결의 여러 스트림에서 여러 HTTP 요청과 응답을 비동적으로 보낼 수 있음</mark>
	- 위의 HOL을 해결(다중으로 받으니까)
- 여러번 TCP연결을 할 필요가 없으니까 RTT가 줄어듦
- 리소스를 작은 프레임으로 나누고, 이를 스트림으로 프레임을 전달
	- 각각의 프레임은 스트림ID와 해당 청크의 크기를 알 수 있음
	- 작게 나누어져서 다운로드 되어도 올바르게 조립 가능
![[Pasted image 20250217144206.png]]
- <mark style="background: #FFF3A3A6;">병렬적으로 빠르게 데이터를 보낼/받을 수 있음!</mark>


"정리를 해보면 이전의 버전 1.0에서는 여러개의 데이터를 받을 때 먼저들어온 데이터의 크기가 커서 로드 시간이 길어지면, 그 뒤에 있는 작은 데이터들도 밀리게 되는 HOL현상이 발생하는 문제가 있음
근데 이런 문제를 해결하기 위해서 2.0에서는 새로운 방법을 가져옴 -> 바이너리 포맷 계층을 두어서
하나의 응답 및 요청을 하나의 텍스트 뭉치가 아니라, 0과 1로 이루어진 데이터로 바꿔줌
이를 통해서  멀티플렉싱도 가능하게 하는데, 이때 멀티 플렉싱이란 기존에 비동기적으로 해결하지 못했던 로드를 하나의 리소스를 작은 프레임으로 나누어지는게 이제 가능하기 때문에 이를 통해서 비동기적으로 요청과 응답을 통해서 리소스를 받아올 수 있음 -> 이때 쪼개진 데이터 즉 프레임들은 프레임ID와 크기를 알 수 있어서 다시 재조립이 가능함!@"


## <font color="#ff0000">3. 서버푸시</font>

- 서버가 리소스를 클라이언트에 푸시할 수 있음
- 요청된 html파일과 함께 다른 개체를 별도로 보낼 수 있음 -> css에 대한 요청이 별도로 없어도 같이 보낼 수 있음
![[Pasted image 20250217144901.png]]
=> 원래는 html한번, css한번 해서 두번의 연결
=> 클라이언트가 html만 요청해도, 서버가 html에서 css가 쓰이는지를 보고 그냥 별도로 첨부 가능


## <font color="#ff0000">4. 헤더압축</font>

- 1.1의 무거운 헤더를 허프만 인코딩 압축 방법 등으로 압축
- <mark style="background: #FFF3A3A6;">똑같은 서버에 2개의 이미지를 줄 때, 중복되는 헤더는 제외하고, 공통 필드로 헤더를 재구성하여, 중복되지 않은 헤더값은 허프만 인코딩 압축 방법으로 전송</mark>

[참고] 허프만 인코딩
문자열을 문자 단위로 쪼개 빈도수를 세어 빈도가 높은 정보는 적은 비트수를 사용해 표현하고, 빈도가 낮은 정보는 비트 수를 많이 사용하여 전체 데이터 표현에 필요한 비트양을 줄이는 알고리즘입니다.


## <font color="#ff0000">5. 우선순위</font>
- <mark style="background: #FFF3A3A6;">서버에서 원하는 순서대로 리소스 전달이 가능</mark>



# HTTP/3.0
___
여전히 TCP를 사용하기 때문에 초기연결에서의 RTT 시간 증가 라는 문제가 있음

### HTTP/3.0은..

- <font color="#ff0000">QUIC(Quick UDP Internet Connections)</font>이라는 계층에서 돌아가며
- TCP기반이 아니라 <mark style="background: #FFF3A3A6;">UDP기반</mark>으로 돌아가며
- 멀티플레싱 지원, <mark style="background: #FFF3A3A6;">초기연결설정시 지연시간 감소</mark>
![[Pasted image 20250217151736.png]]
- TCP는 3-RTT, UDP는 1-RTT
- 2.0은 세션을 만드는 핸드셰이크, 암호화통신을 위한 TLS핸드셰이크 두번
- 3.0은 TLS 암호화통신을 구축할 때, 한번에 핸드셰이크를 활용해서, 클라이언트와 서버간의 연결, 암호화 통신 모두 다 구축한다. -> 즉 1번만 연결하면 끝!
![[Pasted image 20250217154259.png]]
- <mark style="background: #FFF3A3A6;">패킷이 손실되었다면, 수신측에서 에러를 검출 수정</mark>
- 네트워크가 약해도 낮은 패킷 손실률을 자라하는 "순방향 오류 수정 메커니즘(FEC, Forward Error Correction)"이다.

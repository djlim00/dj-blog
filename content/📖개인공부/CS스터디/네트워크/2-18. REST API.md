# 1. Uniform-Interface
___
: API에서 자원들은 각각의 독립적인 인터페이스를 가지며 각각의 자원들이
1. **URL 자원식별**
2. **표현을 통한 자원조작**
3. **Self-descriptive message**
4. **HATEOAS 구조**

> **독립적인 인터페이스**
> : 서로 종속적이지 않은 인터페이스
> ex) 웹페이지를 변경했다고 웹 브라우저(크롬)를 업데이트하지 않음
> ex) HTTP명세나 HTML명세가 변경되어도, 웹페이지는 잘 작동해야함

### 1.1 URL 자원식별
: identification of resources -><mark style="background: #FFF3A3A6;"> 자원은 url로 식별된다</mark>


### 1.2 표현을 통한 자원조작
: manipulation of resources through representations
: url 과 GET, POST와 같은 <mark style="background: #FFF3A3A6;">HTTP 표준메서드를 통해</mark>
: 자원을 조회, 삭제 등의 작업을 설명할 수 있는 정보가 
: 담겨져 있어야 함


### 1.3 Self-decriptive message
: <mark style="background: #FFF3A3A6;">HTTP header에 타입을 명시하고</mark>
: <mark style="background: #FFF3A3A6;">각 메시지(자원)들은 MIME types에 맞춰 표현되야함</mark>
ex) <u>.json을 반환한다면 application/json으로 헤더인 Content-type에 명시해야함</u>
: MINE types는 문서, 파일 등의 특성과 형식을 나타내는 표준
: IETF의 RFC6938에 정의 및 표준화되어 있음.
: 'font/ttf', 'text/plain', 'text/csv' 등을 말한다.
=> <span style="background:rgba(205, 244, 105, 0.55)">즉 자체적으로 충분한 정보가 있어야함</span>
![[Pasted image 20250224102737.png]]
✔ **Self-descriptive Message 원칙을 따르는 올바른 RESTful API**
- `res.status(200)`: **HTTP 200 OK** 상태 코드 명확히 지정.
- `res.json({"a": 1})`: JSON 데이터를 반환하며, **자동으로 `Content-Type: application/json`이 설정됨.**
- 클라이언트는 응답을 받았을 때 **정상적인 JSON 데이터임을 바로 인식 가능**.
❌ **Self-descriptive Message 원칙 위반**
- **문제 2**: `Content-Type: application/json`을 명시했지만 실제 응답이 **JSON이 아닌 단순 문자열 `"큰돌이"`**.
    - JSON 응답을 기대한 클라이언트가 `JSON.parse()`를 수행하면 **파싱 오류 발생 가능**.
- **문제 3**: `res.setHeader()`를 호출 후, `res.send()`에서 문자열을 응답.
    - 하지만 `res.json()`을 사용했다면 자동으로 Content-Type이 설정되므로 불필요한 코드.


### 1.4 HATEOAS 구조
: <mark style="background: #FFF3A3A6;">하이퍼링크에 따라 다른 페이지</mark>를 보여줘야 하며
: <mark style="background: #FFF3A3A6;">데이터마다 어떤 URL에서 원했는지 명시</mark>해줘야함
: 보통은 <font color="#d83931">href, links, link, url 속성</font> 중 하나에 해당 데이터의 URL을 담아서 표기
: ( _links 속성에 담기도 합니다. 링크를 유추할 수 있는 변수명을 쓰면 됩니다.)


ex1) 동적으로 링크를 만들며 href 속성을 통해 설정
ex2) link 속성을 기반으로 설정
ex3) 실제 github에서 제공하는 REST API, url 속성을 기반으로 설정





# 2. Stateless
_____

HTTP 자체가 무상태 프로토콜 이기 때문에 HTTP를 사용하기만 해도 만족.
이는 REST API에서 제공해주는 서버는 세션(Session)을 해당 서버 쪽에서 유지하지 않는다는 의미.
=> 어떤 사람이 어떤 요청을 한 것에 대한 정보가 남아있지 않음(Stateless하니까)
=> Session(로그인의 경우), <mark style="background: #FFF3A3A6;">Rest API로 서버를 만들거면 서버에 세션관련 로직이 있지 않아야함</mark>




# 3. Cacheable
___
HTTP 자체가 원래 캐싱이 됨. 아무 로직을 만들지 않아도 자동 캐싱.
새로고침을 하면 304가 뜨면서 원래 있던 js와 css 이미지 등을 불러옴.
- GET 메서드의 경우에만 캐싱
- Cach-Control:max-age = 100 이런식으로 시간 설정 가능
- Last-Modified와 Etag를 이용해서 캐싱 데이터 유효여부를 확인

# 4. Client-Server 구조
___
: 클라이언트와 서버는 독립적인 구조
: HTTP를 사용하니까 가능한 것임
: 서버는 화면 API제공, 비즈니스 로직 처리
: 클라이언트는 HTTP로 로직만 잘 처리

# REST API의 URL 규칙
___

![[Pasted image 20250224104743.png]]
1. <font color="#d83931">즉 URL에 동사가 들어가지 않는다.</font>
2. 파일의 확장자를 표시하지 않는다.
3. 오직 명사
4. 계층적인 내용
5. <mark style="background: #FFF3A3A6;">언더바(_)가 아닌 그냥 바(-)를 쓴다</mark>
6. 응답 코드를 잘쓰자

```java title:"도서관 시스템"
app.get('/books')
// 모든 책을 조회합니다.
177app.post('/books/:booksid')
// 책을 생성합니다.
app.put('/books/:booksid')
// 책을 수정합니다.
app.get('/books/:booksid')
// 특정 책을 조회합니다.
app.put('/users/:userid/books/:booksid')
// 어떤 유저가 특정 책을 빌립니다.
app.patch('/users/:userid/books/:booksid')
// 어떤 유저가 특정 책을 빌립니다.
```

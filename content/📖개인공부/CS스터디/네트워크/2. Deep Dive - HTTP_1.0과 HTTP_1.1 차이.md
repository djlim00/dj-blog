# HTTP/1.0
___
: 수명이 짧은 연결
: HTTP요청은 자체 요청에서 완료됨
: 한번의 HTTP 요청마다 TCP 핸드셰이크가 발생
: 한 연결당 하나의 요청
: <mark style="background: #FFF3A3A6;">계속 TCP연결을 해야하니까 RTT(Round Trip Time)이 너무 늘어나는 문제점이 있음
</mark>
![[Pasted image 20250217132241.png]]
- HTML, CSS, JS가 필요하면
- 3-wayhandwshake를 3번 해야함 -> RTT가 너무 김!



### 그래서 이런 문제를 해결하기 위해 HTTP/1.1이 나옴



# HTTP/1.1
___

## <font color="#ff0000">1. Keep-alive default</font>
![[Pasted image 20250217132752.png]]
- <mark style="background: #FFF3A3A6;">한번의 연결로 계속해서 데이터를 받을 수 있음</mark>

#### Keep-alive header
<mark style="background: #FFF3A3A6;">: TCP 연결을 유지하는 시간을 의미하는 timeout과 최대 요청수 max를 정할 수 있음</mark>
![[Pasted image 20250217132913.png]]


## <font color="#ff0000">2. 호스트 헤더 - 여러개의 호스트</font>

- HTTP/1.0은 서버가 하나의 호스트만 가진다고 가정
	- 따라서 굳이 헤더에 호스트를 포함하지 않음 -> <mark style="background: #FFF3A3A6;">단 한개의 호스트만 가짐</mark>

- <mark style="background: #FFF3A3A6;">BUT 사실 서버는 여러개의 호스트를 가질 수 있음</mark>
	- <mark style="background: #FFF3A3A6;">이런 유연성을 위해 HTTP/1.1은 특정 호스트를 포함할 수 있게 변경</mark>
	- 항상 호스트를 포함해서 요청하도록 함
- 더 이상 하나의 서버가 하나의 호스만 가질 필요가 없음(하나의 IP에서 여러개의 사이트를 운영할 수 있음)

## <font color="#ff0000">3. 대역폭 최적화</font>

- HTTP/1.0은 어떠한 파일을 다운로드 받다가 끊기면 중간에서 다시 받는게 불가능
	- BUT 1.1은 가능!
- HTTP/1.1에서는 Range:byte=5000- 라는 헤더를 추가
	- 5KB까지 받았던 파일을 다시 받기

## 2. 코드 압축

- 띄어쓰기, 개행문자 같은 부분들을 압축시킴
- 파일이 크면 클 수록 줄일 수 있음!


## 3. 이미지 Base64 인코딩

- <mark style="background: #FFF3A3A6;">이미지 파일을 64진법으로 이루어진 문자열로 인코딩</mark>
- 이미지 서버에 대한 HTTP요청을 할 필요가 없이 만드는 것
- <mark style="background: #FFF3A3A6;">하지만 파일의 크기가 37%커짐</mark>
![[Pasted image 20250217134511.png|400]]
- 저렇게 64진법으로 이루어진 문자열로 바뀌는데 -> 진짜 엄청 김
- 131.75 -> 175.66KB로 파일 크기가 증가함

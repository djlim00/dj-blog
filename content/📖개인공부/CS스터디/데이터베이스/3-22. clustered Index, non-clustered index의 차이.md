# clustered index
___
- <span style="background:rgba(3, 135, 102, 0.2)">유일성과 최소성을 가지는 기본키가 클러스터 인덱스임</span>
- 테이블 당 한개, 보통 테이블의 기본키가 클러스터형 인덱스가 됨(수동으로 지정도 가능)

- 데이터페이지가 <mark style="background: #FFF3A3A6;">정렬</mark>되어 저장되며, 인덱스 페이지의 리프노드에 <mark style="background: #FFF3A3A6;">실제 데이터인 "데이터페이지"</mark>가 들어있음.
	- 정렬되어 있기 떄문에 탐색에 장점
- 데이터가 추가될 때 마다 다시 모든 테이블을 정렬해야 하기 때문에, <mark style="background: #FFF3A3A6;">삽입, 삭제, 수정이 느림</mark>
	- <u>B-트리 형태로 되어있기 때문에 더 오래 걸림</u>
- 테이블 레코드와 인덱스의 순서가 같게 조절됨.

# non-clustered index
___
- 보조 인덱스. <mark style="background: #FFF3A3A6;">한개가 아니라 여러개를 만들 수 있음.</mark>
- 클러스터형키가 복합키가 될 수도 있긴 하나 -> 보통 <mark style="background: #FFF3A3A6;">복합키</mark>면 보조인덱스(non-clustered)로 만듦.
- 클러스터형과 달리 인덱스 <span style="background:rgba(3, 135, 102, 0.2)">리프토드에 실제 데이터가 아니라, 데이터 페이지에 대한 포인터가 있음</span>
- <mark style="background: #FFF3A3A6;">정렬이되있지 않아 탐색은 느리나. 삽입, 삭제, 수정이 빠름</mark>
- 인덱스의 순서와 데이터의 순서가 일치하지 않음

# 클러스터, 보조 인덱스 생성방법
___
primary key : 클러스터형 인덱스
```sql
alter table add primary key (user_id);
```


보조인덱스
```sql
-- CREATE를 이용한 방법
CREATE INDEX idx_user_name_email ON users(name, email);
-- ALTER TABLE ADD INDEX를 이용한 방법
ALTER TABLE users ADD INDEX idx_user_name_email (name, email);
```

⚠️ 위의 예시는 MySQL의 경우이고, 다른 DB에 따라서는 PK로 선언을 해도 클러스터 인덱스 선언을 해줘야 하는 경우가 있음




# 클러스터 인덱스 사용 상황
____
1. PK를 기준으로 데이터를 자주 조회하는 경우
2. 범위 검색(RANGE SCAN)이 자주 발생하는 경우
3. ORER BY 정렬이 자주 사용되는 경우
4. JOIN에서 FK로 자주 조회되는 경우

데이터가 정렬되어 있기 떄문에, 특정 범위의 데이터를 빨리 찾을 수 있음
-> 리프 노드에 실제 데이터가 저장되므로, 인덱스만으로 데이터를 바로 조회 가능 -> 추가적인 테이블 조회가 필요없음
-> 테이블당 하나만 존재 가능

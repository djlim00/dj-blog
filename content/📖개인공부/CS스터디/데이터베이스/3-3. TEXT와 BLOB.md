# 문자열 : TEXT 
___
- 최대 65,535개 길이의 텍스트 데이터를 저장할 수 있음.
	- VARCHAR 
		1. 행 내부에 저장 : VARCHAR(100)을 선언하면 행 내부에 400바이트가 필요함
		2. 65535까지 max size limit(utf-8기준)
		3. 인덱스를 걸 수 있음
	- <mark style="background: #FFF3A3A6;">TEXT : 행 외부에 저장</mark>
		- 행의 데이터 크기를 생각할 필요가 없음
		- <mark style="background: #FFF3A3A6;">Disk에 저장되는 65,535에 대한 포인터만 저장(8바이트)</mark>
		- max size limit을 못건다. 무조건 최대 65535 길이의 데이터를 저장할 수 있음
		- <font color="#d83931">인덱스를 걸 때 크기제한을 해야 걸 수 있음</font>

##### TEXT에서 크기제한 없이 인덱스를 걸면 - ERROR!
![[Pasted image 20250303143629.png]]
🙉 **인덱스** : 테이블 내의 데이터를 빨리 찾을 수 있게 해줌


### 정리
- **VARCHAR** : 중간 정도의 문자열, 메모리에서 읽으므로 읽기 성능이 TEXT보다 빠름
- **TEXT** : 매우 큰 문자열, 디스크에서 읽기 때문에 읽기 성능이 느림. 검색과 수정이 빈번하지 않은 데이터를 저장할 때 쓰임


⚠️ MySQL은 TEXT로 저장해도 like 함수 사용 가능
```sql
SELECT * FROM your_table WHERE content LIKE '%example%';
```



# 이진데이터 : BLOB
____
- 이진 데이터(Binary Object)를 저장하는 데 사용되는 데이터 타입
- <mark style="background: #FFF3A3A6;">이미지, 오디오, 비디오 등을 저장</mark>하며 다음과 같이 파일을 직접 데이터에 삽입해서 씀
```sql
INSERT INTO `kundol`
(`img`)
VALUES
(LOAD_FILE('C:/Users/kundol/Desktop/dev/a.png'))
```

#### <span style="background:#d3f8b6">그러나 보통 BLOB 타입은 안쓰고, 이미지호스팅서비스(S3)를 이용해서 해당 서비스에 올리고, 해당 url을 VARCHAR 타입으로 저장하는게 일반적</span>
![[Pasted image 20250303144631.png|600]]

### 왜 BLOB을 많이 안쓸까?

1. **성능문제** : 데이터베이스에 큰 이진 파일(ex. 이미지)를 저장하면 DB 성능 저하, 관리에 어려움, 백업 및 복구 시간 증가
2. **보안문제** : DB에 이미지를 저장하면, 이 이미지에 대한 접근을 제어하고 관리하는 것이 복잡함. 보안 정책을 적용하고 유지하기 어려움.
	- S3에서는 버튼 클릭 하나씩 설정하면 그냥 단순함

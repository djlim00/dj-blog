# DI(의존성주입)과 DIP(의존관계 역전 원칙)
___
### DI(의존성주입)이란?

: 메인모듈이 '직접' 다른 하위 모듈에 대한 의존성을 주기 보다는
: 중간에 <span style="background:#fff88f">의존성 주입자(dependenct injector)</span>가 이 부분을 가로채
: 메인 모듈이 '간접'적으로 의존성을 주입하는 방식

### 여기서 의존한다는 무슨 의미인가?

"A가 B에 의존한다" = B가 변하면 A에 영향을 미치는 관계

```java title:"에시코드"

public class B{
	public void go(){
		System.out.println("Bgo");
	}
}

public class A{
	public void go(){
		new B.go();
	}
}
```

- 이런식으로 A가 B에 의존하게 코드가 짜져있다고 보자
- <span style="background:#fff88f">이러면 만약 B의 함수 이름을 go()에서 gorani()로 바꾸게 된다면, A클래스에서도 B.go()를 B.gorani()로 변경해야한다. -> 의존성</span>

# DI 적용 == DIP 적용

- 의존성 주입을 하게 되면, 의존관계역전이 일어나게 된다.
![[Pasted image 20250205154410.png]]
```java hl:2-4,18,7,

interface Developer {
	void develop();
}

class BackendDeveloper implements Developer {
	@Override
	public void develop() {
		writeJava();
	}

	public void writeJava() {
		System.out.println("자바가 좋아~ 새삥새삥");
	}
}

class FrontendDeveloper implements Developer {
	@Override
	public void develop() {}
		writeJavascript();
	}

	public void writeJavascript() {
		System.out.println("자바스크립트가 좋아~ 새삥새삥");
	}
}

public class Project {

	private final List<Developer> developers;

	public Project(List<Developer> developers) {
		this.developers = developers;
	}

	public void implement() {
		developers.forEach(Developer::develop);
	}

	public static void main(String args[]) {
		List<Developer> dev = new ArrayList<>();
		dev.add(new BackendDeveloper());
		dev.add(new FrontendDeveloper());
		Project a = new Project(dev);

		a.implement();
	}
}
```

1. <span style="background:#fff88f">상위 모듈은 하위 모듈에 의존해서는 안된다. 둘 다 추상화에 의존해야한다.</span>
2. <span style="background:#fff88f">추상화는 세부사항에 의존하면 안된다. 세부 사항은 추상화에 따라 달려야한다.</span>
![[Pasted image 20250205154410.png]]
: 만약 여기서 WriteJavaScript()함수가 go()로 바뀐다고 해도, <span style="background:#fff88f">Project에서 implement 함수는 바꿀 게 없다!</span>
: <font color="#ff0000">Developer에만 의존관계를 가진다.</font>
: 의존관계 역전이 일어난다(화살표의 방향이 바뀌었다)


## 의존관계역전원칙(DIP)

1. 상위 모듈은 하위 모듈에 의존해서는 안된다. 둘 다 추상화에 의존해야 한다.
	- 위의 Developer 추상화
2. 추상화는 세부사항에 의존해서는 안된다. 세부 사항은 추상화에 따라 달라져야 한다.

### DI 장점
> 1. 모듈을 쉽게 교체할 수 있는 구조
> 2. 단위 테스팅과 마이그레이션이 쉬워짐
> 3. 애플리케이션 의존성 방향이 더 일관되어 코드를 추론하기 쉬워짐

### DI 단점
> 1. 결국 모듈이 생기므로 복잡도가 증가
> 	 - Develop이라는 인터페이스가 추가되버림
> 2. <u>종속성 주입자체가 컴파일을 할 때가 아닌 런타임 때 일어나기 때문에 컴파일을 할 때 종속성 주입에 관한 에러를 잡기 어려워질 수 있음</u>


```css
**<개인적인 생각>**
- 그냥 일반적으로 내가 코드를 짜게되면 거의 모든 코드가 서로를 의존하게 짜게 된다.
	- A클래스의 함수 Ago()를 B클래스에서 사용하는 방식으로 그냥 짜면, 당연히 Ago()에 변경사항이 생기면 B의 코드도 바꿔야되게,, 그냥 다이렉트로 연결을 해버리니까 그런거다
- 그런데 이제 A클래스가 특정한 인터페이스를 구현하게 짜보자.
- interface Alphabet을 만들고 go()메서드를 만들어놓자.
- A클래스는 Alphabet 인터페이스를 구현하여서 go()를 만든다.
- 이제 B 클래스에서 A 클래스의 함수를 사용하려고 하면, B 클래스는 Alphabet이라는 인터페이스로 객체를 만든다. -> 근데 이제 구현객체를 A로 만든다.
- 그럼 이제 go()함수를 호출하면 A의 함수(이전의 Ago()메서드)가 호출되게 된다.
- 이렇게 되면, A 클래스에서 함수의 내용을 바꾸어도(이름을 바꾼다던가, 내용을 바꾼다던가)
- B에서는 그냥 Alphabet alphabet = new A(); 로 만들고, alphabet.go()를 하는 것은 바뀌지 않는다.
-> 즉 더이상 B는 A 클래스 자체에 의존성을 가지지 않고, B는 인터페이스 Alphabet만 의존하게 된다. 또한 A는 Alphabet을 구현하기 떄문에 Alphabet에 의존하게 된다.
-> 즉 의존관계의 화살표가 반대가 되어서, 의존관계의 역전이 일어난다.
```




# ♟️ 전략패턴
___
## 전략패턴이란?

: <span style="background:#fff88f">"전략"이라고 부르는 캡슐화된 알고리즘</span>을
: <u>컨텍스트 안에서 바꿔주면서</u>
: <span style="background:#fff88f">상호교체</span>가 가능하게 만드는 디자인 패턴

- 이때 로그인이라는 행위,상황, 상태 => **컨텍스트**
- 다양한 로그인 방식 => **전략**

=> <u>그러니까 전략 패턴은 컨텍스트에서 다양한 전략들이 교체가능하게 설계하는 것</u>

## 옵저버 패턴이란?

: **주체**가 어떤 <span style="background:#fff88f">객체의 상태 변화를 관찰</span>하다가
: 상태 변화가 있을 때마다
: 메서드를 등을 통해 
: <u>옵저버 목록에 있는 옵저버들에게</u>
: <span style="background:#fff88f">변화를 알려주는 디자인 패턴</span>
	ex) 트위터의 메인 로직, MVC패턴..etc

![[Pasted image 20250206094105.png|500]]
- <font color="#ff0000">꼭 주체와 객체가 나뉘는게 아니라, 주체 그 자체가 객체가 되어서, 변경사항이 옵저버들에게 알림이 가는 경우도 있음</font>

```java
import java.util.ArrayList;
import java.util.List;

interface Subject {
    public void register(Observer obj);
    public void unregister(Observer obj);
    public void notifyObservers();
    public Object getUpdate(Observer obj);
}

interface Observer {
    public void update(); 
}

class Topic implements Subject {
    private List<Observer> observers;
    private String message; 

    public Topic() {
        this.observers = new ArrayList<>();
        this.message = "";
    }

    @Override
    public void register(Observer obj) {
        if (!observers.contains(obj)) observers.add(obj); 
    }

    @Override
    public void unregister(Observer obj) {
        observers.remove(obj); 
    }

    @Override
    public void notifyObservers() {   
        this.observers.forEach(Observer::update); 
    }

    @Override
    public Object getUpdate(Observer obj) {
        return this.message;
    } 
    
    public void postMessage(String msg) {
        System.out.println("Message sended to Topic: " + msg);
        this.message = msg; 
        notifyObservers();
    }
}

class TopicSubscriber implements Observer {
    private String name;
    private Subject topic;

    public TopicSubscriber(String name, Subject topic) {
        this.name = name;
        this.topic = topic;
    }

    @Override
    public void update() {
        String msg = (String) topic.getUpdate(this); 
        System.out.println(name + ":: got message >> " + msg); 
    } 
}

public class HelloWorld { 
    public static void main(String[] args) {
        Topic topic = new Topic(); 
        Observer a = new TopicSubscriber("a", topic);
        Observer b = new TopicSubscriber("b", topic);
        Observer c = new TopicSubscriber("c", topic);
        topic.register(a);
        topic.register(b);
        topic.register(c); 
   
        topic.postMessage("amumu is op champion!!"); 
    }
}
/*
Message sended to Topic: amumu is op champion!!
a:: got message >> amumu is op champion!!
b:: got message >> amumu is op champion!!
c:: got message >> amumu is op champion!!
*/ 
```

- 같은 topic을 구독 중인 옵저버 a,b,c가 있다.
- 이때 해당 topic에서 메세지를 보내면, 각각의 옵저버는 해당 메세지를 전부 출력한다.
- Topic에는 옵저버들을 저장하는 List가 있다. 또한 postMessage()에서 해당 Topic의 message를 한다.

### 예제 코드
```js
function createReactiveObject(target, callback) { 
    const proxy = new Proxy(target, {
        set(obj, prop, value){
            if(value !== obj[prop]){
                const prev = obj[prop]
                obj[prop] = value 
                callback(`${prop}가 [${prev}] >> [${value}] 로 변경되었습니다`)
            }
            return true
        }
    })
    return proxy 
} 
const a = {
    "형규" : "솔로"
} 
const b = createReactiveObject(a, console.log)
b.형규 = "솔로"
b.형규 = "커플"
// 형규가 [솔로] >> [커플] 로 변경되었습니다
```

## 1. MVC패턴

: <span style="background:#fff88f">Model(모델)</span>

- 사용자가 화면에서 뭔가 액션을 함
- 컨트롤러가 그걸 인식해서 모델(데이터)의 변화를 줌
- 모델이 변화를 컨트롤러로 알림
- 컨트롤러가 뷰를 업데이트 함.
### 1.1 모델
> 1. 애플리케이션의 데이터인 데이터베이스, 상수, 변수
> 2. 뷰에서 데이터를 생성, 수정 -> 컨트롤러에서 모델이 생성 or 업데이트
> ex) 사용자가 네모박스에 글자를 적음
> Model { 
> 	int 네모박스 크기; 
> 	String 글자내용; 
> 	int[] 글자의 위치;
> 	String 글자 포멧;
> }
### 1.2 뷰
> 1. inputbox, checkbox, textarea 등 UI적인 요소
> 2. <span style="background:#fff88f">모델을 기반으로 사용자가 볼 수 있는 화면</span>
> 3. 모델의 정보를 가지고 있지 않음(오직 화면구성 정보만)
> 4. <span style="background:#fff88f">변경이 일어나면 컨트롤러에 전달</span>
### 1.3 컨트롤러
> 1. 하나 이상의 모델과 하나 이상의 뷰를 잇는 다리 역할
> 2. 이벤트 등 메인 로직 담당
> 3. 모델과 뷰의 생명주기 관리
> 4. 모델이나 뷰의 사항을 해석해서 각각의 구성 요소에 변경 내용을 알려줌

## MVC 패턴의 장점

1. 애플리케이션의 구성 요소를 세 가지 역할로 구분하여, 개발 프로세스에서 각각의 구성 요소에만 집중해서 개발 가능
2. 재사용성과 확장성이 용이함
## MVC 패턴의 단점

1. 애플리케이션이 복잡해지면, 모델 뷰 관계가 너무 복잡해짐

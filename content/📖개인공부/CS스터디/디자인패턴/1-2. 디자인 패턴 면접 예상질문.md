## 1. MVVM 모델에 대해 설명해주세요
___
: MVVM 모델은 MVC패턴에서 파상된 패턴으로 애플리케이션의 구성요소를 Model, View, ViewModel 세 가지 역할로 구분하는 디자인패턴입니다. 
ViewModel은 모델에서 필요한 데이터를 가져와서 가공한 다음 뷰에서 사용할 수 있게 하는 로직을 담당합니다. 뷰는 이런 뷰모델을 참조하여 UI를 구성합니다. 이때 MVVM모델은 데이터 바인딩을 통해서 뷰모델에 있는 데이터로 자동으로 UI를 업데이트 할 수 있습니다. 또한 커맨드를 통해서 여러 요소를 하나의 액션으로 처리할 수 있습니다. 이를 통해서 기존의 프레젠터와 뷰의 강한 의존성을 없엘 수 있습니다.
MVP패턴은 프레젠터와 뷰가 1:1관계이기 때문에 코드가 방대해질 수 있다는 문제점이 있지만, MVVM 패턴은 하나의 뷰모델이 같은 데이터를 필요로 하는 여러 뷰가 참조할 수 있습니다.
그러나 MVVM패턴은 모든 데이터가 비동기로 처리되어야하기 때문에 러닝커브가 상당하다는 단점이 있습니다.


## 2. MVC패턴을 설명하고 MVVM과의 차이를 말해주세요
___
MVC 패턴은 애플리케이션의 구성요소를 Model, View, Controller로 나누어서 개발하는 패턴입니다. Model은 애플리케이션의 데이터와 비즈니스 로직을 다룹니다. View는 화면 구성과 관련된 역할을 수행합니다. Controller는 모델과 뷰를 잇는 역할을 합니다. 
MVVM패턴은 MVC 패턴에서 파생된 패턴으로, 애플리케이션의 구성요소를 Model, View, ViewModel로 나눕니다. MVVM패턴에서는 데이터 바인딩을 이용해서 뷰가 뷰모델을 참조하여서 변경된 데이터를 자동으로 업데이트합니다. 이러한 여러개의 요소들의 업데이트를 커맨드를 통해서 하나의 액션으로 처리합니다. 이를 통해서 MVC패턴에서의 뷰와 컨트롤의 의존성이 강하다는 단점을 해소합니다.
그러나 MVVM패턴은 모든 데이터를 비동기적으로 처리해야 하므로 MVC 패턴에 비해 러닝커브가 상당하다는 단점이 있습니다.

## ~~MVVM 패턴과 Flux패턴의 차이점에 대해 설명해보세요~~

<aside> 💡

Flux패턴은 저번 스터디에서 다루지는 않았지만 React가 공식적으로는 FLUX패턴을 사용한다고 알려져 있기 때문에 추가했습니다. 저희 교재에도 Flux패턴 내용이 중요도 ⭐️⭐️⭐️ 으로 나와있으니 교안 읽어보셔도 좋을 것 같아요

</aside>

MVVM은 데이터 바인딩을 사용하여 View와 ViewModel 간에 양방향 데이터 흐름을 유지합니다. 반면 Flux는 단방향 데이터 흐름을 사용하여 Action -> Dispatcher -> Store -> View의 방향으로 데이터가 흐릅니다. MVVM에서는 ViewModel이 Model의 역할을 수행하고, View와 ViewModel 사이에서 데이터를 중개합니다. 반면 Flux에서는 Store가 애플리케이션의 데이터와 비즈니스 로직을 담당하고, Dispatcher를 통해 Action을 처리합니다. Flux는 단방향 데이터 흐름과 단일 책임 원칙을 따르므로 테스트하기 쉬운 코드를 작성하기 쉽습니다. MVVM에서는 ViewModel과 View 간의 양방향 데이터 바인딩 때문에 테스트하기가 어려울 수 있습니다. MVVM은 Angular, Vue.js 등의 프레임워크에서 사용되며, Flux는 React에서 주로 사용됩니다.

**→ MVVM과 Flux가 각각 어떤 프레임워크에서 사용되는 패턴인지 아는 것이 중요합니다.**

## 3. 절차지향 프로그래밍과 객체지향 프로그래밍의 차이를 설명해주세요
____
: 절차지향 프로그래밍은 프로그램을 물 흐르듯이 프로그램의 실행 순서대로 개발하는 것을 의미합니다. 그에 비해 객체지향 프로그래밍은 프로그램의 구성요소들을 각각의 역할을 가진 객체로 보고 이런 객체간의 상호작용으로 개발하는 것을 의미합니다. 절차지향 프로그래밍의 대표적인 언어로는 C언어, 객체지향 프로그래밍의 대표적인 언어로는 자바가 있습니다.
절차지향 프로그래밍은 코드가 하나의 파일에 있기 때문에 간단한 프로그램의 경우 유용할 수 있지만, 프로그램이 커질수록 코드량이 늘어나게 되면 가독성이 떨지는 문제점이 있는 반면 객체지향프로그래밍은 하나의 역할을 하고 있는 객체를 만들어서 이를 조합하여 프로그램을 만들기 때문에, 가독성이 좋으며 코드의 재사용이 굉장히 편하다는 차이가 있습니다. 
속도적인 측면에서는 컴퓨터의 처리순서와 비슷한 절차지향이 더 빠릅니다.


## 4. <font color="#ff0000">객체지향의 특징에 대해 설명해주세요</font>
___
: OOP는 프로그램을 독립된 객체들의 상호작용으로 봅니다.
객체지향 프로그래밍의 특징은 4가지가 있습니다.
1. <mark style="background: #FFF3A3A6;">추상화</mark>입니다. 대상의 공통된 특징과 행동을 모아서 클래스를 정의합니다. 프로그램의 목적에 맞는 정보를 뽑아내서 만듭니다.
2. <mark style="background: #FFF3A3A6;">캡슐화</mark>입니다. 외부에서 직접적인 데이터의 접근을 막고, 오직 메서드를 통해서만 데이터에 접근할 수 있도록 하는 것입니다. 
3. <mark style="background: #FFF3A3A6;">다형성</mark>은 동일한 요청에 서로다른 방식의 응답이 가능한 점 입니다. 예를 들어 자동차 객체를 구현한 클래스는 버스가 될 수도 있고, 택시가 될 수도 있습니다. 상속받은 함수를 재정의 하는 오버라이딩과, 함수의 매개변수를 다르게 받을 수 있는 오버로딩을 통해 구현합니다.
4. <mark style="background: #FFF3A3A6;">상속</mark>은 부모클래스의 속성과 기능을 이어받아 사용하는 것입니다. 범용적인 클래스를 만들어 상속받을 수 있게 하면, 다양한 클래스에서 공통된 기능을 사용할 수 있습니다.

# 1. DIP 위반 코드에서 어떤 방법으로 DI를 적용시켜볼 수 있을까?
___
DIP(Dependency Inversion Principle, 의존성 위반 원칙)란 SOLID원칙 중 하나로, 객체 간의 의존 관계를 맺을 때, 쉽게 변화하는 것보다는 추상적인 개념, 거의 변하지 않은 것에 의존하라는 원칙이다.

아래 결제 시스템에서 어떤 점에서 DIP를 위반했는지, 그리고 어떻게 DI를 적용하여 유연한 시스템으로 만들 수 있을까?

```java
class Payment {
    public void processPayment(double amount) {
        System.out.println(amount + "원을 결제합니다.");
    }
}

class OrderService {
    private Payment paymentProcessor = Payment(); // 강한 결합 (Bad Practice)

    public void checkout(double amount) {
        paymentProcessor.processPayment(amount);
    }
}
```

```jsx
// 강한 결합 (Bad Practice)
function processPayment(amount) {
  console.log(`${amount}원을 결제합니다.`);
}

function checkout(amount) {
  // 결제 방식을 변경하려면 checkout 함수 내부를 수정해야 함 (DIP 위반)
  processPayment(amount);
}

// 실행 코드
checkout(10000); // 10000원을 결제합니다.

```
![[Pasted image 20250210161804.png]]
: OrderService에서 Payment객체를 직접 호출하여서 사용하는 것은 의존성이 높은 상황
다른 결제 방식을 사용하려고 하면 코드가 바뀌어야 함.
1. 우선 Payment 인터페이스를 만들고 인터페이스에 processPayment 메서드를 만듦
2. 인터페이스의 생성자에서 어떤 payment를 선택할지 고를 수 있음(생성자 주입)
3. OrderService에서는 인터페이스의 생성자를 통해서 객체를 생성하고, 해당 객체에서 proceePayment함수를 호출한다.

그렇게 되면 OrderService는 인터페이스를 의존하게 되고, Payment도 인터페이스에 의존하게 된다. 이를 통해 결제방식을 선택함에 있어서 유연한 변경이 가능하게 된다.



# 2. 옵저버 패턴을 통한 알림 시스템을 프록시를 통해서 어떻게 개선할 수 있을까?
___
아래 시스템은 주식 가격 변동을 실시간으로 구독할 수 있도록 한 옵저버를 구현한 내용이다, 하지만 주식 가격의 잦은 변동으로 인한 불필요한 알림이 발생한다는 점이 흠이다. 해당 시스템을 프록시 패턴을 적용해서 개선할 수 있는 방법에는 무엇이 있을까?

```java
// 🔹 옵저버 (Observer) 인터페이스
Observer {
    update(stockSymbol, price)  // 주식 가격 변경 시 호출됨
}

// 🔹 주식 시장 (옵저버 패턴 적용)
StockMarket {
    observers = []  // 구독자 리스트

    addObserver(observer) {
        observers.append(observer)
    }

    updateStockPrice(symbol, newPrice) {
        notifyObservers(symbol, newPrice)
    }

    notifyObservers(symbol, newPrice) {
        for observer in observers:
            observer.update(symbol, newPrice)
    }
}

// 🔹 투자자 (옵저버)
StockInvestor(name) implements Observer {
    update(stockSymbol, price) {
        print(name + "님, " + stockSymbol + " 주식이 " + price + "원으로 변경되었습니다!")
    }
}

// 🔹 실행 예제
market = StockMarket()
investor1 = StockInvestor("Alice")
investor2 = StockInvestor("Bob")

market.addObserver(investor1)
market.addObserver(investor2)

market.updateStockPrice("AAPL", 1500)  
// Alice님, AAPL 주식이 1500원으로 변경되었습니다!
// Bob님, AAPL 주식이 1500원으로 변경되었습니다!
```

문제 : 너무 잦은 알람이 문제 -> 알람의 횟수를 줄여야함
해결책 : 프록시 패턴을 도입해서, 주식의 가격이 일정 수준 이상으로 변동이 되었을 때만 알림이 가도록 알림이 가기 전에 notifyObservers(symbol, newPrice)를 가로채서 호출되지 않도록 한다.(혹은 일정시간)



# 3. 인증 토큰 관리 시스템에서 싱글톤 패턴이 활용될 수 있는 사례에는 무엇이 있을까?
___
웹 기반인증 및 보안 시스템에서 제일 많이 이용되는 시스템은 인증 토큰 시스템인데, 이 토큰 기반 인증 과정에서 싱글톤 패턴이 사용될 만한 곳은 무엇이 있을까?

(Hint: 싱글톤 패턴의 활용은 단일 인스턴스의 활용, 즉 동일한 정보가 여러 요청에서 활용되어야 할 경우에 유용하다.)
![[Pasted image 20250210161916.png]]
- 싱글톤 패턴은 하나의 인스턴스만을 가지고 있으므로, 세션값을 싱글톤으로 캐싱하고 있으면 여러요청에 대해서 단 하나만 저장하고 있는 세션값을 통해 검증하여서 로그인 상태를 유지할 수 있다.



### 4. 디자인 패턴을 적용해서 앞서 말한 인증 방식을 다양화할 수 있는가? 사용한 패턴 및 해당 패턴을 통해서 얻는 장점은?
___
최근에는 과거의 비밀번호 + id를 조합한 인증 시스템 뿐만 아니라, 다양한 인증 수단을 동원하여 인증 시스템이 생겨나고 있다. 이에 따라 인증 방식 다양화를 할 수 있도록 설계 및 구현하는 게 중요해지는데, 이를 위해서 어떤 디자인 패턴을 통한 설계가 좋을까? 패턴 적용 방식 및 장점에 대해서 설명한다면?
![[Pasted image 20250210161943.png|300]]
- 전략패턴을 이용한 설계가 좋다.
- 전략패턴은 같은 기능을 하는 다양한 구현을 교체하기 편하게 설계하는 방식이다. 즉 다양한 구현 방식을 하나의 인터페이스를 구현하도록 하여서, 다른 전력(알고리즘, 방식)을 선택하게 되면 일부분만 수정하면 된다.
- 이를 통해서 유지보수와 유연성이 증가한다.

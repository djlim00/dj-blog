- 가상 주소는 페이지 테이블에 의해서 실제 주소와 매핑된다.


## 페이지 테이블
-> 가상 메모리를 가상주소와 실제 주속가 매핑되어 있는 페이지 테이블로 관리.
-> 속도 향상을 위해서 TLB를 사용(Translation Lookaside Buffer, 캐싱계층)
(TLB에서 바로 실제 메모리로 갈 수 있는지 확인)
(없으면 페이지 테이블 가서 찾기) -> 있으면 바로 물리주소
(없으면 RAM으로 감) -> 있으면  TLB갱신
![[Pasted image 20250317183524.png]]
-> <mark style="background: #FFF3A3A6;">한마디로 가상 주소는 가변적이라서 같은 가상주소가 같은 물리 주소를 의미하지 않는 다는 뜻이다.</mark>

### TLB
: 가상 메모리 주소를 물리적인 주소로 변환하는 속도를 높이기 위해 사요 ㅇ하는 캐시
: <mark style="background: #FFF3A3A6;">최근에 일어난 가상 메모리 주소와 물리 주소의 매칭 테이블 저장</mark>
: 그냥 일종의 주소 변환 캐시

## 페이지 폴트의 과정
![[Pasted image 20250317184034.png]]
1. 어떤 명령어가 유효한 가상주소에 접근했으나 해당 페이지가 없음 -> 트랩 -> 운영체제에 알림
2. 운영체제는 실제 디스크로부터 사용하지 않는 프레임을 찾음
3. 해당 프레임을 실제 메모리에 가져와서 **페이지 교체 알고리즘을 기반으로 특정페이지와 교체(스와핑 발생!)**
4. 페이지 테이블을 갱신시킨 후 해당 명령어를 다시 시작.


## 😡 스레싱
____
- <mark style="background: #FFF3A3A6;">메모리의 페이 폴트율이 높은 상황</mark>
- 메모리에 너무 많은 프로세스가 동시에 올라가게 되면 스와핑이 잦아짐
- 페이지 폴트가 발생하면 CPU이용률이 낮아짐(그럼 운영체제는 CPU가 일을 안한다고 인식)
- 운영체제는 CPU의 가용성을 높이기 위해 더 많은 프로세스를 메모리에 올리게 된다.
	- 그럼 또 메모리에 너무 많은 프로세스가 올라가고... -> 악순환
![[Pasted image 20250317184335.png]]
- 그래프에서 CPU 가용율이 급격하게 떨어진다.

## 스레싱 해결방안
1. 느린 HDD에서 빠른 SSD로 바꾼다.
2. 운영체제에서 해결하는 방법은 **작업세트**와 **PFF**가 있음.
#### 작업세트
: 프로세스의 과거 사용이력을 기반으로 많이 사용하는 페이지 집합을 만들어 한꺼번에 미리 메모리에 로드하는 것을 말한다.

#### PFF
: 페이지 폴트 빈도를 조절하는 방법으로 
: 상한선과 하한선을 만들고
: 상한선에 도달하면 프레임을 늘리고, 하한선에 도달하면 프레임을 줄인다.

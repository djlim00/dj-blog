## 1. 동준이와 은서의 마법 주문 🪄 ✨

동준이와 은서는 마법 학교에서 강력한 주문을 연구하고 있어요. 둘은 같은 마법 재료를 사용해야 하지만, 동시에 접근하면 주문이 꼬여버려요! 🤯

동준은 재료를 쓸 때마다 잠금을 걸어 사용하고, 은서는 빠르게 쓰고 넘어가요. 🏃‍♀️ 실험을 반복할수록 동준의 주문은 안정적으로 동작하지만, 은서의 주문은 매번 달라져요.

**문제**: 마법 학교에서 문제가 발생하는 이유는 무엇인가요? 그리고 이를 해결하기 위해서 어떻게 하면 좋을까요?
: 은서의 경우 동준과 다르게 lock을 사용하지 않아서 race condition이 발생하기 때문입니다. 락을 사용하지 않으면 공유자원을 두개 이상의 여러 프로세스가 동시에 하나의 공유자원에 대해서 읽고 쓰기를 하기는 상황을 말합니다. race condition이 발생하면  프로세스의 실행순서가 꼬이게 되서 의도한 대로 동작하지 않는 경우가 생깁니다. 이는 데이터의 정합성과 무결성을 해칩니다.
race condition을 방지하기 위해서는 은서는 동준이의 방식처럼 공유자원을 사용하는 경우 하나의 프로세스만이 사용할 수 있도록 lock을 사용해야합니다. 이러한 락의 종류로는 뮤텍스와 세마포어가 있습니다.

## 2. 윤혁이와 네명의 연금술사 😋🍽️

윤혁과 네명의 연금술사가 한 테이블에 앉아 실험을 진행 중이에요. 테이블 위에는 다섯 개의 마법 도구가 있지만, 모든 연금술사는 두 개의 도구가 있어야 실험을 할 수 있어요. 그런데, 만약 모든 연금술사가 동시에 하나의 도구만 집고 다른 도구를 기다린다면, 아무도 실험을 하지 못하는 상황이 벌어져요! 😱

**문제**: 위 문제는 무엇인가요? 윤혁이는 문제를 어떻게 해결할 수 있을까요?
	: 위 문제는 데드락을 말합니다. 두개 이상의 프로세스가 서로가 점유하고 있는 자원을 해제 않고 새로운 자원을 획득하려는 상황에 이러지도 저러지도 못하는 상황을 말합니다. 데드락 문제를 해결하기 위해서는 데드락이 발생하기 위한 조건을 하나를 깨야합니다. 데드락이 발생하는 경우 싸이클을 발견하고 해소시켜주거나, 데드락이 발생한 위험이 없는 경우에만 자원을 할당해주거나, 혹은 데드락이 발생하면 프로그램을 중지시키고 다시 시작하는 방법이 있습니다.

## 3. 교녕이와 은비의 마법 서고 🔮📚

교녕과 은비는 새로운 마법 서고를 관리하고 있어요. 서고에는 세 개의 책장만 있고, 각 책장은 한 번에 한 권의 마법서를 보관할 수 있어요. 학자들이 자주 찾는 마법서가 다음 순서대로 주어집니다. 📖

```
1, 2, 3, 4, 1, 2, 5, 1, 2, 3, 4, 5

```

교녕이는 가장 먼저 들어온 책을 먼저 버리는 정책을 사용하고, 은비는 가장 오랫동안 사용되지 않은 책을 버리는 정책을 사용해요.

**문제**: 교녕이와 은비가 사용하는 스케줄링 정책에서 페이지 폴트 횟수를 각각 계산하고, 더 좋은 다른 방식이 있다면 제안해주세요! 🧐
- **교녕**
	- 9회
- 은비
	- 1 -> 폴트
	- 1,2 -> 폴트
	- 1,2,3 -> 폴트
	- 2,3,4 -> 폴트
	- 3,4,1 -> 폴트
	- 4,1,2 -> 폴트
	- 1,2,5 -> 폴트
	- 1,2,5
	- 1,2,5
	- 2,5,3 -> 폴트
	- 5,3,4 -> 폴트
	- 3,4,5 
=> **6번** + 3번 = 9번

## 5. 동찬이의 고대 연금술서 관리 🏰🔖

고대 연금술을 관리하는 동찬이는 서고에 있는 연금술서들을 잘 보이게 배치하려고 해요. 서고의 공간은 한정되어 있으며, 각 고서는 사용될 때마다 꺼내서 읽히고 다시 돌아가요.

**문제**: 동찬이가 고서들의 배치를 어떻게 해야 효율적으로 공간을 활용할 수 있을지, 메모리 관리에서 사용되는 배치 전략을 활용해 설명하세요! 📂

1. FF(Frist Fit)
	- 처음부터 탐색해서 가장 빨리 들어갈 수 있는 공간이 나오면 바로 들어감
	- 공간이 남아 돌아서 -> 외부단편화, 내부단편화 문제 발생
2. BF(Best Fit)
	- 가장 최적(책보단 크지만 가장 작은) 공간에 들어감
	- 많은 검색이 필요함,  외부단편화가 발생
3. WF(Worst Fit)
	- 가장 큰 공간에 넣기
	- 나중에 다시 쪼갤 수 있음

-> 공간을 효율적으로 사용하기 위해서는 Best Fit을 사용하는게 나을 듯(책을 잘게 쪼갤 수는 없으니..)


## 문제 1: 페이징과 세그멘테이션

페이징과 세그멘테이션은 운영체제의 메모리 관리 기법입니다. 두 기법의 주요 차이점을 설명하고, 각각이 어떤 종류의 단편화(fragmentation) 문제를 해결하거나 발생시키는지 서술하세요.
: 페이지는 메모리를 고정된 크기의 블럭으로 나눕니다. 이는 빈 공간이 너무 많이 생기는 외부 단편화를 방지할 수 있습니다. 그러나 페이지의 크기가 너무 크게 되면 페이지 내에서 내부 단편화가 발생할 수 있습니다.
세그멘테이션은 프로그램의 논리적 단위(스택, 힙, 코드)로 메모리를 나눕니다. 이때 나누어진 세그먼트의 크기는 가변적입니다. 세그멘테이션은 필요한 만큼만 할당할 수 있기 때문에 내부단편화 문제는 해결할 수 있지만, 가변크기의 블럭을 사용하기 떄문에 외부단편화 문제가 생길 수 있습니다.(중간에 빈 공간이 생김)

## 문제 2: 프로세스와 스레드

프로세스와 스레드의 차이점을 설명하세요. 특히 자원 공유 측면에서 어떤 차이가 있는지, 그리고 컨텍스트 스위칭(context switching)이 각각에서 어떻게 다르게 발생하는지 서술하세요.
: 프로세스는 실행중인 프로그램의 인스턴스입니다. 프로세스는 메모리와 CPU점유를 할당받습니다.또한 프로세스끼리는 서로 독립적입니다. 프로세스는 스택, 힙, 코드와 같은 완전한 메모리 공간을 할당받습니다. 
스레드는 프로세스 내부의 작업의 단위입니다. 프로세스끼리와 달리 스레드 끼리는 서로 독립적이지 않고 메모라 공간을 공유합니다.
스레드는 메모리 공간과 자원을 공유하기 때문에 속도가 빠르지만, 프로세스는 컨텍스트 스위칭 비용이 발생하기 때문에 느립니다.

## 문제 3: 세마포어

세마포어(Semaphore)가 무엇인지 설명하고, wait(P)와 signal(V) 연산이 어떻게 작동하는지 서술하세요.
: 세마포어는 공유자원에 대한 접근을 제한하기 위한 방법 중 하나입니다. 프로세스당 정수값을 부여하고 해당 정수를 wait() 함수를 이용해서 -1감소하고, signal()함수를 이용해서  1증가시킵니다. 이때 0이하의 값이 되면 더 이상의 프로세스 접근이 불가능하고, 0이상의 값이면 접근이 가능합니다. 프로세스가 공유자원에 접근하면 wait()함수를 통해서 진입하고, 더 이상 공유자원을 점유하지 않으면 signal()함수를 호출하면 대기중인 다른 프로세스(스레드)를 깨워서 자원을 점유할 수 있도록 합니다.
